Chapter 2

Architecture Overview

The big picture — every service, API, and database in the system and how they connect.

2.1 The Big Picture

At its core, BSearch has two jobs:

  1. Get documents ready to search — take every file in Berenson's SharePoint and OneDrive, break it into searchable pieces, and store those pieces in an index.
  2. Answer questions — take a user's question, find the most relevant pieces, and use AI to synthesize an answer with citations back to the original files.

These two jobs are handled by two separate pipelines (a pipeline is just a sequence of steps that data flows through, one after another, like an assembly line). The two pipelines are connected by a shared search index — a specialized database optimized for finding things fast.

IB Analogy

Think of it like the sell-side process. The ingestion pipeline is the preparation phase — gathering materials, organizing the data room, tagging documents by deal and type. The query pipeline is the buyer Q&A phase — someone asks a question, the team finds the relevant documents, and drafts a precise answer with source references. The search index is the data room itself — organized, indexed, and searchable.

BSearch is built from about a dozen external services and APIs, each doing one job. Here is every piece of the system, what it does, what it costs, and why it was chosen over alternatives.

Vector embedding: A list of numbers (in this case, 1024 numbers) that represents the meaning of a piece of text. Two pieces of text with similar meanings will have similar numbers, even if they use completely different words. This is how BSearch can find documents about “enterprise value” when you search for “valuation.”

BM25: A traditional keyword-matching algorithm — the same type of search that powers Google’s basic text matching. It finds documents that contain the exact words you typed. BSearch uses BM25 alongside vector search so that exact matches (like file names or deal names) are not missed.

Cost Note

The total monthly cost for BSearch is roughly $300–600, dominated by the Azure AI Search index tier. At 18 users and 50–200 queries per day, the AI API costs (Claude, Voyage, Cohere) are modest — the per-query costs are fractions of a cent. The most expensive single item is the search index, which is a fixed monthly fee regardless of usage.

2.3 The Two Pipelines

The Ingestion Pipeline: Documents → Index

The ingestion pipeline is a background process that continuously keeps the search index up to date with the firm's documents. It runs in two modes:

  • Bulk import: The initial run that processes all ~550K documents. This runs once (and again whenever a full re-index is needed).
  • Incremental updates: Triggered by webhooks (real-time notifications from Microsoft Graph) whenever a file is created, modified, or deleted. These keep the index current without re-processing everything.

The pipeline follows these steps for each document:

flowchart TB
    A["Detect new/changed file\n(Graph API webhook or delta query)"] --> B["Download file\nfrom SharePoint/OneDrive"]
    B --> C["Parse content\n(format-aware: Word, Excel,\nPowerPoint, PDF)"]
    C --> D["Extract metadata\n(doc type, deal name,\ncompany, industry)"]
    D --> E["Chunk the content\n(split into searchable pieces\nby format type)"]
    E --> F["Generate embeddings\n(Voyage AI converts each\nchunk to a vector)"]
    F --> G["Write to search index\n(Azure AI Search)"]
    D --> H["Update knowledge graph\n(Neo4j: entities &\nrelationships)"]

Figure 2.2 — The ingestion pipeline: from SharePoint file to searchable index entry.

Chunking: Splitting a document into smaller pieces (called “chunks”) that are the right size for search and AI processing. A 50-page CIM might become 80–120 chunks. Different document formats are chunked differently — narrative text is split by paragraphs, tables are kept intact, and slides are one-per-chunk.

The Query Pipeline: Question → Answer

When a user types a question into BSearch, the query pipeline processes it through five stages to produce an answer:

flowchart TB
    Q["User asks a question"] --> C1["1. Classify the query\n(file lookup? data extraction?\nmulti-doc synthesis?)"]
    C1 --> C2["2. Search the index\n(hybrid: vector + keyword\nwith permission filters)"]
    C2 --> C3["3. Rerank results\n(Cohere scores each chunk\nagainst the original question)"]
    C3 --> C4["4. Enrich with entity context\n(Neo4j provides deal/company\nrelationship data)"]
    C4 --> C5["5. AI synthesis\n(Claude reads top chunks and\nstreams an answer with citations)"]
    C5 --> A["Answer with source links\nstreamed to the user"]

Figure 2.3 — The query pipeline: from question to streamed answer.

The entire query pipeline targets under 3 seconds to the first token of the AI response. Several stages run in parallel to hit this target — for example, query classification and embedding generation happen simultaneously.

IB Analogy

The query pipeline works like a well-run analyst research request. Classification is the associate deciding whether it’s a “find me a file” request or an “answer a question” request. Search is pulling relevant materials from the data room. Reranking is the associate reviewing the stack and pulling the five most relevant pages to the top. Entity context is checking the deal tracker for background. Synthesis is drafting a concise response with proper citations.

2.5 Repository Structure

All of BSearch’s code lives in a single repository (a “monorepo”). This keeps everything in one place rather than splitting the backend, frontend, and infrastructure across separate repositories.

Monorepo: A single code repository that contains multiple projects or components. The alternative is having separate repositories for the backend, frontend, infrastructure templates, etc. A monorepo is simpler to manage for a solo builder because everything is in one place and changes that span multiple components can be made in a single commit.

bsearch/
  src/          ← Python backend (FastAPI search API + ingestion pipeline)
  web/          ← React frontend (the web app users see)
  infra/        ← Bicep templates (Azure infrastructure-as-code)
  tests/        ← Automated tests
  docs/         ← This technical guide
Directory What Lives Here Language
src/ The FastAPI search API, the ingestion pipeline (Azure Durable Functions), parsers, chunking logic, embedding calls, Claude synthesis, and all backend business logic Python
web/ The React web application — the search interface, streaming answer display, source cards, conversation history TypeScript / React
infra/ Bicep templates that define every Azure resource (search index, app service, functions, Cosmos DB, etc.) as code. Running these templates creates or updates the infrastructure Bicep (Azure IaC)
tests/ Automated tests for the backend — unit tests for parsers and chunking, integration tests for the search pipeline Python (pytest)
docs/ This technical guide and design documentation HTML / Markdown

Bicep: Azure’s language for defining cloud infrastructure as code. Instead of clicking through the Azure portal to create a database or a web server, you write a template file that describes what you want. Running the template creates or updates the resources automatically and reproducibly — like a recipe rather than cooking from memory.

2.6 Why Each External API Was Chosen

BSearch uses several third-party AI services rather than building everything in-house. Each choice was driven by the same three criteria: quality (does it produce the best results?), operational simplicity (can one person manage it?), and cost at our scale (reasonable for 18 users?).

Claude Sonnet 4.6 for AI synthesis

The synthesis model is the most visible part of the system — its output is literally what users read. Claude Sonnet was chosen for its reasoning quality on synthesis tasks: it follows citation instructions reliably, handles multi-document reasoning well, and produces clear, concise answers. Streaming support means the first words appear quickly even before the full answer is generated.

Alternatives considered: GPT-4o (comparable quality but higher cost and no streaming via Azure OpenAI at the time), Claude Haiku (too terse for synthesis, better suited to classification).
Claude Haiku 4.5 for classification and metadata extraction

Some tasks do not need the most powerful AI model. Classifying a query as “file lookup” vs. “data extraction” or extracting a deal name from a document path are simpler tasks where a fast, cheap model performs well. Haiku runs these tasks at a fraction of a cent per call, and its responses arrive in under 200ms.

Alternatives considered: Rules-only approach (too brittle for edge cases), Sonnet for everything (unnecessary cost and latency for simple tasks).
Voyage AI (voyage-context-3) for embeddings

Embeddings are the foundation of semantic search — if the embeddings are poor, the system retrieves the wrong documents. Voyage AI’s contextual embedding model outperforms OpenAI’s text-embedding-3-large by roughly 14% on retrieval benchmarks. Its 32K token context window means it can consider surrounding context when generating each chunk’s embedding, leading to better representations of document meaning.

Alternatives considered: OpenAI text-embedding-3-large (solid baseline but lower retrieval accuracy), Azure OpenAI embeddings (same model, adds Azure dependency without benefit).
Cohere Rerank for result re-scoring

Initial search returns the top 20 candidates, but not all of them are equally relevant. The reranker reads each candidate chunk alongside the original question and assigns a relevance score. This “cross-encoder” approach is more accurate than the initial search because it sees the query and the document together rather than comparing pre-computed vectors. After reranking, only the top 5 most relevant chunks are passed to Claude for synthesis.

Alternatives considered: No reranking (lower answer quality — Claude would get more noise), Voyage reranker (newer, less battle-tested at the time of selection).
Neo4j Aura for the knowledge graph

Many real questions at an IB firm involve relationships: “What deals has Company X been involved in?” “Who are the contacts on deals above $50M EBITDA?” A traditional search index is not good at traversing relationships — it finds documents, not connections. Neo4j stores entities (deals, companies, people) and the relationships between them as a graph, making these traversal queries fast and natural.

Alternatives considered: Storing relationships in Cosmos DB (possible but requires manual join logic), no knowledge graph (would limit entity-aware search capabilities).
Key Takeaway

BSearch’s architecture is two pipelines connected by a search index. The ingestion pipeline continuously processes documents from SharePoint into searchable, AI-ready chunks. The query pipeline classifies questions, searches the index, reranks results, enriches them with entity context from a knowledge graph, and streams AI-synthesized answers. About a dozen external services each handle one job — the total cost is $300–600/month for 18 users, with the search index being the largest fixed expense.
