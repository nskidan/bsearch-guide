Chapter 2

How the System Fits Together

The full architecture — every service, API, and database in the system, what each one costs, and how they connect.

2.1 The Big Picture

BSearch does two things:

  1. Get documents ready to search — take every file in Berenson’s SharePoint and OneDrive, break it into searchable pieces, and store those pieces in an index.
  2. Answer questions — take a user’s question, find the most relevant pieces, and use AI to synthesize an answer with citations back to the original files.

These two jobs are handled by two separate pipelines. A pipeline is a sequence of steps that data flows through, one after another, like an assembly line. The two pipelines are connected by a shared search index — a specialized database optimized for finding things fast.

IB Analogy

Think of it like the sell-side process. The ingestion pipeline is the preparation phase — gathering materials, organizing the data room, tagging documents by deal and type. The query pipeline is the buyer Q&A phase — someone asks a question, the team finds the relevant documents, and drafts a precise answer with source references. The search index is the data room itself — organized, indexed, and searchable.

Entity store: A dedicated PostgreSQL database that stores structured information about deals, companies, people, and the relationships between them. While the search index stores document chunks for retrieval, the entity store tracks higher-level facts: Company A was the target in Deal B, which was advised by Person C. Relationship tables with JSONB provenance metadata make cross-entity queries (“who are the contacts on deals above $50M EBITDA?”) fast and reliable.

IB Analogy

The entity store is like the firm’s deal tracker combined with its contact database — but connected. Instead of flipping between a deal log and a CRM to figure out which people were on which deals for which companies, the entity store keeps all those connections in one place and can query across them instantly.

2.3 External APIs

BSearch uses several third-party AI services rather than building everything in-house. Each was chosen for three criteria: quality (does it produce the best results?), operational simplicity (can one person manage it?), and cost at our scale (reasonable for 18 users?).

Voyage AI (voyage-context-3) — Embeddings

Embeddings are the foundation of semantic search — if the embeddings are poor, the system retrieves the wrong documents. Voyage AI’s voyage-context-3 model creates contextual embeddings: when it converts a chunk into a vector, it considers the surrounding document context, not just the chunk in isolation. This produces better representations of what the chunk means.

In benchmarks, voyage-context-3 outperforms OpenAI’s text-embedding-3-large by roughly 14% on retrieval accuracy. Its 32K token context window means it can read long surrounding passages when generating each chunk’s embedding.

Alternatives considered: OpenAI text-embedding-3-large (solid baseline but lower retrieval accuracy), Azure OpenAI embeddings (same model, adds Azure dependency without benefit).
Cohere Rerank — Result re-scoring

Initial search returns the top 20 candidates, but not all are equally relevant. The reranker reads each candidate chunk alongside the original question and assigns a relevance score. This “cross-encoder” approach is more accurate than initial search because it sees the query and the document together rather than comparing pre-computed vectors. After reranking, only the top 5 most relevant chunks are passed to Claude for synthesis.

Alternatives considered: No reranking (lower answer quality — Claude gets more noise), Voyage reranker (newer, less battle-tested at time of selection).
Claude Sonnet 4.6 (Anthropic) — AI synthesis

The synthesis model is the most visible part of the system — its output is literally what users read. Claude Sonnet was chosen for its reasoning quality on synthesis tasks: it follows citation instructions reliably, handles multi-document reasoning well, and produces clear, concise answers. Streaming support means the first words appear quickly even before the full answer is generated.

Alternatives considered: GPT-4o (comparable quality but higher cost), Claude Haiku (too terse for synthesis, better suited to classification).
Claude Haiku 4.5 (Anthropic) — Classification and extraction

Some tasks do not need the most powerful AI model. Classifying a query type, extracting a deal name from a document path, or generating a document summary are simpler tasks where a fast, cheap model performs well. Haiku runs these tasks at a fraction of a cent per call with responses in under 200ms.

Alternatives considered: Rules-only approach (too brittle for edge cases), Sonnet for everything (unnecessary cost and latency for simple tasks).
Microsoft Graph API — M365 file access

The official (and only) API for reading SharePoint and OneDrive content. Already included in Berenson’s M365 subscription at no additional cost. Provides webhooks for real-time change notifications and delta queries for catching anything webhooks miss.

No alternatives — Graph API is the only supported way to access M365 content programmatically.

Vector embedding: A list of numbers (in BSearch’s case, 1024 numbers) that represents the meaning of a piece of text. Two pieces of text with similar meanings will have similar numbers, even if they use completely different words. This is how BSearch can find documents about “enterprise value” when you search for “valuation.”

Cross-encoder: A model that reads the search query and a candidate document together to score relevance. This is more accurate than comparing pre-computed vectors (which were generated independently) because the model can see how the query and document relate to each other directly. The trade-off is speed — it can only score a handful of candidates, not thousands — which is why it runs after initial search narrows the field to 20.

2.4 Embedding Provider Abstraction

The AI embedding space moves fast — better models emerge regularly, pricing changes, and providers occasionally deprecate models. BSearch is designed so the embedding provider can be swapped without changing the ingestion or query pipeline logic.

All embedding calls (both during ingestion and at query time) go through an abstract EmbeddingProvider interface. A factory function reads a configuration variable and returns the correct provider implementation. In v1, this returns the Voyage AI provider. If a better model emerges or Voyage changes pricing, switching providers is a configuration change plus a background re-index — not a code rewrite.

Each chunk in the search index carries provenance metadata — three fields recording which model generated its embedding, the vector dimensions, and a version tag. This means the system always knows which chunks were embedded by which model, which is essential during a provider transition when old and new embeddings coexist briefly.

IB Analogy

Think of this like how a deal team uses market data providers. The model runs on revenue multiples regardless of whether they come from CapIQ or Bloomberg. If the firm switches providers, the analysts update the data source and refresh the model — they do not rebuild the model from scratch. The embedding abstraction works the same way: swap the data source, re-populate, keep the same pipeline.

2.6 Monthly Cost Breakdown

BSearch’s ongoing monthly cost is roughly $343–471 at current usage (18 users, 50–200 queries/day). The largest fixed expense is the search index. AI API costs are modest at this scale — the per-query costs are fractions of a cent.

One-Time Costs

The initial bulk ingestion of ~550K documents has a separate one-time cost of roughly $11,000–12,000, dominated by Azure Document Intelligence for PDF parsing (~$9,200 after deduplication savings), Claude Haiku for metadata extraction (~$1,200–1,800), and Voyage AI for bulk embeddings (~$500–1,000). After the initial load, ongoing incremental costs are negligible since only changed files are reprocessed.

2.7 Repository Structure

All of BSearch’s code lives in a single repository (a “monorepo”). This keeps everything in one place rather than splitting the backend, frontend, and infrastructure across separate repositories.

Monorepo: A single code repository that contains multiple projects or components. The alternative is having separate repositories for the backend, frontend, infrastructure templates, etc. A monorepo is simpler to manage for a solo builder because everything is in one place and changes that span multiple components can be made in a single commit.

bsearch/
  src/
    shared/       ← Common code used by both the API and ingestion pipeline
    ingestion/    ← Azure Durable Functions (document processing pipeline)
    api/          ← FastAPI search API (query pipeline)
  web/            ← React frontend (the web app users see)
  infra/          ← Bicep templates (Azure infrastructure-as-code)
  tests/          ← Automated tests
  docs/           ← This technical guide
Directory What Lives Here Language
src/shared/ Configuration, Pydantic data models, the Microsoft Graph API client, the health-check base class, and provider abstractions (embedding, reranker, synthesis) — code used by both pipelines Python
src/ingestion/ The ingestion pipeline — format-aware parsers (Word, Excel, PowerPoint, PDF), chunking logic, entity extraction, embedding calls, and the Durable Functions orchestrator Python
src/api/ The FastAPI search API — query classification, hybrid search, reranking, Claude synthesis, streaming responses, and conversation history Python
web/ The React web application — search interface, streaming answer display, source cards, conversation history TypeScript / React
infra/ Bicep templates that define every Azure resource (search index, app service, functions, Cosmos DB, Key Vault, Application Insights, etc.) as code. Running these templates creates or updates the infrastructure Bicep (Azure IaC)
tests/ Automated tests — unit tests for parsers and chunking, integration tests for the search pipeline, sample fixture files Python (pytest)

Bicep: Azure’s language for defining cloud infrastructure as code. Instead of clicking through the Azure portal to create a database or a web server, you write a template file that describes what you want. Running the template creates or updates the resources automatically and reproducibly — like a recipe rather than cooking from memory.

The key architectural benefit of this layout is the src/shared/ directory. Both the ingestion pipeline and the query API import from it, which means changes to shared models, configuration, or provider abstractions (embedding, reranker, synthesis) are automatically consistent across both pipelines. There is one definition of what a “chunk” looks like, one configuration schema, one Graph API client.

Key Takeaway

BSearch’s architecture is two pipelines connected by a search index, supported by an entity store. The ingestion pipeline (Durable Functions) continuously processes documents from SharePoint into searchable, AI-ready chunks. The query pipeline (FastAPI on App Service B2) classifies questions, searches the index, reranks results, enriches them with entity context from PostgreSQL, and streams AI-synthesized answers via Claude. Five external AI services each handle one job — embeddings (Voyage), reranking (Cohere), synthesis (Claude Sonnet), classification (Claude Haiku), and file access (Microsoft Graph). The embedding provider is abstracted so it can be swapped without pipeline changes. Total ongoing cost is $343–471/month for 18 users, with the search index as the largest fixed expense.
