Chapter 9

Phase 2 — Document Parsers & Chunking

Build the document processing engine — the part of BSearch that reads every file format, breaks content into searchable chunks, and extracts the metadata that makes search work.

9.1 What You’re Building

This phase builds BSearch’s document processing engine — the code that takes a raw file from SharePoint, reads its contents, and produces the structured, searchable chunks that feed the search index. Every search result BSearch returns traces back to a chunk created by this pipeline. If the parsers miss content or the chunker splits in the wrong places, search quality degrades no matter how good the embedding model or the reranker is.

The work falls into five areas: format-aware parsers that extract content from Word, Excel, PowerPoint, and PDF files while preserving document structure; a chunking engine that splits content into 800–1000 token pieces at logical boundaries; a hierarchical table splitter for large financial tables; summary generation at both the document and deal level; and metadata and entity extraction that classifies documents, identifies companies and people, and populates the entity store.

This is the most technically dense phase. It touches four file format libraries, an Azure AI service, an LLM (Haiku), a relational database (PostgreSQL), and a set of regex-based NER heuristics. But the output is straightforward: for every file BSearch processes, this code produces a list of chunks ready to be embedded and indexed.

9.2 Before You Start

Prerequisites
  • Phase 1 complete. Monorepo structure in place, Pydantic config working, Bicep templates deployed. You need the src/ingestion/ directory and the shared config module.
  • Sample documents for each format. Get at least one Word document, one Excel model, one PowerPoint deck, and one PDF from a real deal folder. These should contain tables, headings, and multi-section content — not empty templates. Put them in tests/fixtures/.
  • Azure Document Intelligence resource provisioned. This is Azure’s PDF parsing service (formerly called Form Recognizer). It uses AI to understand document layout — headers, paragraphs, tables, reading order — rather than just extracting raw text. Provision an S0 tier instance in the Azure portal and store the endpoint and key in your .env file. The free tier works for development but has a 2-page-per-minute rate limit.
  • PostgreSQL instance running. A local or managed instance is sufficient for development. You need the connection string in your .env. The entity store holds deal entities, company relationships, and financial metrics extracted during ingestion.
  • Anthropic API key configured. Haiku is used for document summaries, entity extraction, and doc_type classification fallback. Add ANTHROPIC_API_KEY to your .env.
  • Python dependencies installed. This phase requires python-docx, openpyxl, python-pptx, azure-ai-formrecognizer, asyncpg, and anthropic. These should already be in your pyproject.toml from Phase 1.

9.3 The Spec — Format-Aware Parsers

Every parser follows the same contract: take in raw file bytes and metadata, return a list of ContentBlock objects that preserve the document’s structure. The abstract base class lives in src/ingestion/parsers/base.py.

ContentBlock — A dataclass representing a unit of parsed content. Each block has a content string, a block_type (paragraph, table, slide, formula), an optional section_heading from the nearest parent heading, and metadata like slide number or sheet name. Parsers produce ContentBlocks; the chunking engine consumes them.

Parser Interface

# src/ingestion/parsers/base.py
class BaseParser(ABC):
    @abstractmethod
    def parse(self, file_bytes: bytes, metadata: FileMetadata) -> list[ContentBlock]:
        """Extract structured content from a file."""
        ...

    def chunk(self, blocks: list[ContentBlock]) -> list[Chunk]:
        """Split content blocks into index-ready chunks.
        Default implementation handles narrative chunking;
        subclasses override for format-specific behavior."""
        ...

Four Parser Implementations

Parser Library What It Extracts Key Details
Word python-docx Paragraphs, tables, headers, styles Tracks heading hierarchy (H1 → H2 → H3) so each paragraph knows its section context. Tables extracted with row/column structure preserved. Styles indicate bold, italic, etc.
Excel openpyxl Sheets, cells, formulas, named ranges Processes sheet-by-sheet. Detects header rows automatically. Captures formula text on key model sheets (not just resolved values) — formulas are indexed as a separate chunk per sheet. Resolves formulas to values where possible.
PowerPoint python-pptx Per-slide content, speaker notes, text frames, tables Each slide becomes its own ContentBlock with a [Slide N: Title] prefix. Speaker notes are appended to slide content. Tables within slides are preserved as structured data.
PDF Azure Document Intelligence Tables, paragraphs, reading order Handles scanned documents, complex CIM layouts, watermarked teasers, and embedded tables. Understands document layout (columns, headers, reading order) rather than just extracting raw text line by line.
Why Azure Document Intelligence for PDFs?

Standard PDF text extraction (e.g., PyPDF2) treats a PDF as a flat stream of characters. It does not understand that a two-column layout should be read left-column-then-right-column, that a table has rows and columns, or that a header belongs to the following section. Azure Document Intelligence uses AI to reconstruct the document’s logical structure — critical for CIMs and financial presentations where layout carries meaning. The trade-off is cost: ~$0.13 per page during bulk ingestion.

PDF Pre-Deduplication

Before parsing any PDF, the pipeline checks whether a Word, Excel, or PowerPoint file with the same base name exists in the same SharePoint folder. If a match is found, the PDF is skipped entirely — no download, no parse, no embedding.

Why This Matters

Berenson’s SharePoint folders frequently contain PDF exports alongside their Office source files (e.g., Project Atlas - Teaser - 2024.01.15 v.003.pptx alongside Project Atlas - Teaser - 2024.01.15.pdf). Office files have richer structure — styles, heading hierarchy, cell formulas — so they produce better chunks. Roughly 75% of the ~225,000 PDFs are companion copies. Skipping them saves approximately $22,000 in Azure Document Intelligence costs during bulk ingestion.

The matching logic normalizes filenames: strip the extension, strip trailing version suffixes (the Office file has v.003, the PDF does not), and compare. The check runs against the Graph API folder listing, so it happens before any file download. On API error, the pipeline fails open and parses the PDF normally.

9.4 The Spec — Chunking Engine

The chunking engine takes ContentBlocks from the parsers and produces index-ready chunks. The goal: chunks that are large enough to contain meaningful context but small enough to fit efficiently in Claude’s prompt alongside other retrieval results.

Content Type Chunk Size Strategy
Narrative (Word, PDF) 800–1000 tokens, ~100-token overlap Split at paragraph or section boundaries. Never split mid-paragraph. Prepend document title + section heading to each chunk.
Tables (Excel, tables in Word/PPT) Intact as one chunk (≤1500 tokens) Prepend text description: [Sheet: Revenue Model] Table with columns: Year, Revenue, EBITDA — 15 rows. If >1500 tokens, use hierarchical row-group splitting (see below).
Slides One chunk per slide Each slide is its own chunk: [Slide 3: Management Team] {content}
Excel formulas Separate chunk per key sheet [Sheet: LBO Model — Formulas] {formula text}

Section Context Preservation

Every narrative chunk gets its section heading prepended. If a Word document has an H1 “Financial Analysis” with an H2 “Revenue Breakdown,” every chunk from that section starts with that context. This is critical for retrieval: without it, a chunk containing “grew 12% year-over-year” has no context for what grew 12%.

Hierarchical Table Splitting

Tables over 1500 tokens need to be split, but you cannot split a financial table at an arbitrary row. An income statement split between “Revenue” and “COGS” loses its structure. The table splitter detects logical boundaries in the data.

Row-group detection — The algorithm that finds logical split points in large tables. Investment banking tables encode hierarchy through structural signals: indent levels, bold headers, “Total” rows, and blank separators. The splitter reads these signals to find category boundaries without needing to understand what the numbers mean.

The splitting works in four steps:

  1. Detect column headers. Identify the first 1–2 rows as headers using a heuristic: rows where all non-empty cells are bold, or all cells are strings with no numeric values. These headers are stored and prepended to every split chunk.
  2. Detect category boundaries. Scan data rows top-to-bottom. A row is a boundary if any of these signals fire:
Signal Detection Example
Indent decrease cell.alignment.indent == 0 after rows with indent > 0 “Operating Expenses” at indent 0 after sub-items at indent 2
Bold first cell First non-empty cell is bold and at indent 0 Revenue by Segment (bold, unindented)
Total/Subtotal row First cell matches ^(Total|Subtotal|Net|Grand Total)\b “Total Operating Expenses”
Blank separator All cells in row are empty or whitespace Blank row between “Sources” and “Uses” sections
PDF/Word tables Bold first cell or blank preceding row (no indent data) Same logic, minus indent signals
  1. Build row groups. Segment rows at boundaries. Each row group is a dataclass with an optional category label, the rows, and a token count.
  2. Greedy chunk assembly. Merge adjacent groups until adding the next would exceed 1500 tokens. Each chunk gets column headers prepended, a category breadcrumb (e.g., [Operating Expenses > SG&A]), and a descriptive prefix with row range and column names.

How This Handles Real IB Tables

Table Type Hierarchy Signal Grouping
Income statement Bold category headers (Revenue, COGS, OpEx) + “Total” rows One group per category + its line items
Revenue by customer Bold customer names at indent 0, products indented One group per customer + sub-rows
Sources & Uses “Sources” / “Uses” bold headers or blank separator Two groups (one per section)
Quarterly bridge Typically flat with no hierarchy Fixed-size fallback (~25–30 rows)
Cap table Investor names at indent 0, share classes indented One group per investor
Edge Cases

Deeply nested tables (3+ indent levels): group at top-level (indent 0) boundaries only. Sub-hierarchy stays intact within the chunk. Merged cells: resolve openpyxl.MergedCell objects to the parent cell’s value. Every row is bold (common in summary tables): no boundaries detected, so the flat-table fallback kicks in at ~25–30 rows per chunk.

9.5 The Spec — Summary Generation

Document Summaries

For any document that produces four or more chunks, Haiku generates a ~200-token summary. This summary serves as a semantic bridge between distant sections of the same document — when a CIM has a company description on page 5 and financial metrics on page 30, the summary chunk mentions both, so a query about the company’s financials can surface the right document even if neither individual chunk alone is a strong match.

Parameter Value
Trigger Document produces ≥4 chunks
Input First ~2000 tokens of document content
Output ~200-token summary
Model Claude Haiku 4.5
Indexed as chunk_type: "doc_summary", chunk_index: -1
Timeout 5 seconds
On failure Log warning, skip summary. Document still indexed normally.

The summary captures the document type, deal name, key topics covered, entities and metrics mentioned, and a brief section map. The prompt is loaded from src/shared/prompts/doc_summary.txt — not hardcoded in the Python file.

Deal-Level Summaries

For deals with three or more documents and a non-null canonical name, Haiku generates a ~400-token deal-level summary. These are what make cross-deal queries possible — without them, answering “where do we stand on active deals?” would require Claude to reason across hundreds of raw chunks from dozens of deals in a single call.

Parameter Value
Trigger Deal has ≥3 documents AND non-null canonical_name
Input All doc_summary chunks for the deal + entity context from PostgreSQL
Output ~400-token deal overview
Model Claude Haiku 4.5
Indexed as chunk_type: "deal_summary", synthetic file_id
Timeout 8 seconds (more input context than doc summary)
On failure Log warning, skip. No retry.

Deal Summary Refresh Schedule

Deal summaries are refreshed through two mechanisms:

  • Event-triggered (webhook). After a file is ingested via webhook, the pipeline queues a deal summary update. A debouncing mechanism consolidates changes: the pipeline writes a {deal_id, last_file_change_ts} record to a Cosmos DB deal_summary_pending container. A timer trigger checks every minute for deals where the last file change was more than 5 minutes ago, and processes them. This prevents regenerating the summary 10 times when 10 files change in rapid succession.
  • Periodic (timer). Every 6 hours, a scheduled job checks all qualifying deals for stale summaries — any deal where a document’s last_modified is newer than the deal summary’s last_modified.

9.6 The Spec — Metadata & Entity Extraction

Document Type Classification

Every document gets a doc_type classification. This drives deal stage inference, entity extraction targeting, and search filtering. The classifier uses filename and path patterns first (fast, free), with a Haiku fallback for ambiguous cases.

doc_type Description
cimConfidential Information Memorandum
teaserConfidential teaser / one-pager
modelFinancial model (Excel)
engagement_letterEngagement letter / advisory agreement
ndaNon-disclosure / confidentiality agreement
management_presentationManagement presentation deck
diligence_trackerDue diligence tracker / VDR index
board_deckBoard of directors presentation
process_letterProcess letter to potential acquirers
ioiIndication of Interest
loiLetter of Intent
purchase_agreementPurchase / merger agreement
memoInternal memo / research note
otherAnything that doesn’t match the above

Deal Identity

The deal_id is a deterministic identifier derived from the deal’s subfolder path: a SHA-256 hash of the first-level subfolder name under one of the two deal root folders (Banking - General or Private Equity - General), truncated to 16 characters. All files anywhere under Banking - General/Project Atlas/** produce the same deal_id, regardless of nesting depth.

Deal root folders — The two top-level SharePoint folders that contain deal subfolders: Banking - General and Private Equity - General. These are defined as a code-level constant (DEAL_ROOT_FOLDERS) in the metadata module — not in config, because they are structural facts about the firm’s organization that change at the same frequency as the code itself.

Three-Tier Entity Extraction

Entity extraction runs at three tiers, each with a different cost/accuracy trade-off:

Tier Method Scope What It Extracts Where It Writes
Tier 1 Haiku (1 call/doc) Document-level Companies with roles and confidence, primary industry, deal name, doc_type PostgreSQL (deals, companies tables, relationships)
Tier 2 Haiku (selective) High-value chunks Financial metrics (revenue, EBITDA, EV, multiples), people with titles, additional company relationships PostgreSQL (financial metrics on deals, people tables, additional relationships)
Tier 3 Regex / heuristics Every chunk Company name mentions, person name mentions, dollar amounts, percentages, dates Azure AI Search chunk fields only (NOT PostgreSQL)

Tier 1 — Document-Level Haiku Extraction

One Haiku call per document, using the first ~1000 tokens. Extracts the document type, deal name, primary companies with roles (target, acquirer, advisor, lender, other) and confidence levels (high, medium, low), and industry classification. Only high and medium confidence entities are written to PostgreSQL; low confidence entities populate chunk-level fields only.

Tier 2 — High-Value Chunk Extraction

After chunking, certain chunks are flagged for additional Haiku extraction based on their content:

  • Financial tables — chunks with headers containing “Revenue,” “EBITDA,” “Enterprise Value,” or “Purchase Price”
  • People sections — chunks from management team sections, signature blocks, or engagement letter parties (detected via heading hierarchy from the parser)
  • IOI/LOI body chunks — chunks from documents with doc_type of ioi or loi
  • Process letter chunks — chunks from process_letter documents

Typically ~1–3 additional Haiku calls per document. Many documents (memos, board decks, general files) get zero additional calls.

Tier 3 — Local NER (No API Calls)

Every chunk gets lightweight named entity recognition via regex and heuristics. Company names are detected by matching capitalized multi-word phrases followed by entity suffixes (Inc, Corp, LLC, Ltd, Partners, Capital, Group, Holdings). Person names are detected in structured contexts (after Mr./Ms./Dr., in “Name, Title” patterns). A cached set of known entity names from PostgreSQL improves recall — the cache is refreshed at the start of each ingestion batch.

Why Three Tiers?

Tier 1 gives you deal-level intelligence cheaply (one Haiku call per document, ~$550 for all 550K documents). Tier 2 adds precision where it matters most — financial tables and structured sections — for an additional ~$200–400. Tier 3 adds per-chunk mentions for free. The tiers are additive: Tier 1 finds “Goldman Sachs is the advisor on this deal,” Tier 2 extracts the specific fee terms from the engagement letter, and Tier 3 tags every chunk that mentions Goldman Sachs for BM25 search.

9.7 The Spec — Deal Registry & Entity Store

Deal Alias Accumulation

Investment banking deals always have multiple names: a code name (“Project Atlas”) in early documents and the real company name (“Acme Corp”) in later ones. BSearch handles this through alias accumulation in PostgreSQL. The deals table stores a canonical_name, and a separate deal_aliases table stores all known name variants. Every time a file is ingested:

  1. Haiku extracts a per-file deal name from the filename and content.
  2. If the name is new, it is inserted via INSERT INTO deal_aliases ... ON CONFLICT DO NOTHING.
  3. If this file has the most recent last_modified timestamp, the canonical_name is updated. This naturally converges to the “real” company name as deals progress from code-name documents to formal ones.

When a deal’s canonical name changes, all existing chunks with that deal_id in Azure AI Search are batch-updated — a targeted field update, not a full re-ingestion.

Company Name Normalization

Company rows use normalized_name as the primary key: lowercase, with suffixes stripped (Inc., Corp., LLC, Ltd., & Co., Group, Holdings, International, Advisors, Partners, Capital). This prevents “Goldman Sachs,” “Goldman Sachs & Co.,” and “Goldman Sachs Group, Inc.” from becoming separate rows. New companies are inserted via INSERT ... ON CONFLICT DO UPDATE.

Deal Stage Inference

Deal stages are inferred deterministically from doc_type — no LLM required:

doc_type Inferred deal_stage
nda, engagement_letterorigination
teaser, cim, management_presentation, process_lettermarketing

The deal’s current_stage is always the highest stage reached. Stage history is tracked as rows in the deal_stage_history table linked to the deal by deal_id.

Deal Status Inference

Deal status is also deterministic, based on document history and activity recency:

9.9 What to Tell Claude Code

Prompt 1 — Document Parsers
Build document parsers for BSearch. Create an abstract BaseParser class in src/ingestion/parsers/base.py with a parse(file_bytes, metadata) -> list[ContentBlock] method, where ContentBlock is a dataclass with content, block_type (paragraph/table/slide/formula), section_heading, and format-specific metadata. Implement four parsers:

WordParser (python-docx): extract paragraphs with heading hierarchy tracking (H1/H2/H3), tables with row/column structure, styles.
ExcelParser (openpyxl): sheet-by-sheet extraction, automatic header row detection, formula text capture as separate content blocks, resolve formulas to values where possible.
PPTParser (python-pptx): per-slide content blocks with [Slide N: Title] prefix, speaker notes appended, tables within slides preserved.
PDFParser (Azure Document Intelligence): tables as structured data, paragraphs with reading order, section headers.

Also create src/ingestion/pdf_dedup.py with a has_matching_office_file() function that checks if a Word/Excel/PPT file with the same base name exists in the same folder before parsing a PDF. Normalize filenames by stripping extensions and trailing version suffixes.
Prompt 2 — Chunking Engine & Table Splitter
Build the chunking engine for BSearch. Add a chunk() method to BaseParser that takes list[ContentBlock] and returns list[Chunk]. Target 800–1000 tokens per chunk with ~100-token overlap. Split at paragraph boundaries — never mid-paragraph. Prepend the document title + nearest section heading to each chunk as context.

For tables: keep tables intact as one chunk up to 1500 tokens. For larger tables, create src/ingestion/table_splitter.py with hierarchical row-group splitting. Detect column headers (bold-all or all-string rows). Detect category boundaries via indent level changes (indent 0 after indent >0), bold first cells at indent 0, Total/Subtotal regex, and blank separator rows. Build row groups at boundaries, then greedy-merge groups up to 1500 tokens. Prepend column headers to every split chunk. Add category breadcrumb and descriptive prefix with row range and column names. Flat table fallback: if no boundaries detected, split at fixed intervals of 25 rows.

Handle edge cases: merged cells (resolve to parent value), deeply nested tables (group at indent 0 only), all-bold tables (flat fallback).
Prompt 3 — Summaries, Metadata & Entity Extraction
Add document and deal summary generation, metadata extraction, and entity extraction to BSearch.

Doc summaries (src/ingestion/doc_summary.py): for documents producing ≥4 chunks, call Haiku with the first 2000 tokens to generate a ~200-token summary. Index as chunk_type: "doc_summary", chunk_index: -1. Load prompt from src/shared/prompts/doc_summary.txt. 5s timeout, log and skip on failure.

Deal summaries (src/ingestion/deal_summary.py): for deals with ≥3 docs and non-null canonical_name, generate ~400-token deal overview using all doc_summary chunks + PostgreSQL entity context. Index as chunk_type: "deal_summary". Debounce webhook-triggered regeneration with a 5-minute consolidation window via Cosmos DB. Periodic refresh every 6 hours. 8s timeout.

Metadata (src/ingestion/metadata.py): classify doc_type via filename regex patterns with Haiku fallback. Compute deal_id as SHA-256 hash of first-level subfolder under deal root folders. Extract deal_name per-file via regex + Haiku.

Entity extraction: Tier 1 doc-level Haiku (companies with roles/confidence, industry). Tier 2 selective Haiku on financial table chunks, people sections, IOI/LOI chunks. Tier 3 regex NER in src/shared/ner.py (company suffixes, person patterns, cached known entities from PostgreSQL). Write entities to PostgreSQL via src/ingestion/entity_registry.py and src/ingestion/deal_registry.py using INSERT ... ON CONFLICT for alias accumulation. Build src/shared/pg_client.py as the PostgreSQL client wrapper (asyncpg) with Deal/Company/Person CRUD.

9.10 How to Verify

Verification Checklist
  1. Parse one sample of each format. Run each parser (Word, Excel, PPT, PDF) against a real deal document. Verify content is extracted completely — no missing paragraphs, no garbled tables, no lost speaker notes. Check that heading hierarchy is tracked for Word, formula text is captured for Excel, and slide numbers are correct for PowerPoint.
  2. PDF pre-dedup. Place a .pptx and its companion .pdf in the same test folder. Verify has_matching_office_file() returns True for the PDF. Rename the PDF to break the match and verify it returns False.
  3. Chunk a multi-section document. Parse and chunk a Word document with 5+ sections. Verify chunks are 800–1000 tokens. Verify each chunk has its section heading prepended. Verify no chunk splits mid-paragraph. Spot-check that the first and last sentences of each chunk form complete thoughts.
  4. Chunk a large financial table. Parse sample_hierarchical.xlsx (a table with indented rows, bold category headers, and Total rows exceeding 1500 tokens). Verify: chunks split at category boundaries, each chunk has column headers repeated at the top, descriptive prefix includes row range and category name. Then test sample_flat_table.xlsx — verify the flat fallback produces ~25-row chunks.
  5. Generate a document summary. Parse a 10+ chunk document (e.g., a multi-section CIM). Verify Haiku generates a summary. Verify the summary chunk has chunk_type: "doc_summary" and chunk_index: -1. Read the summary — it should mention key topics from different sections of the document. Verify a 3-chunk document does NOT generate a summary.
  6. Run entity extraction on a sample CIM. Verify Tier 1 extracts the target company name, advisor, and industry with high or medium confidence. Verify doc_type is classified correctly. Verify Tier 2 runs on any financial table chunks and extracts metrics (revenue, EBITDA, etc.). Verify Tier 3 NER tags company mentions in individual chunks.
  7. Check PostgreSQL after extraction. Query the deals table — verify the row has the correct deal_id, canonical_name, and that deal_aliases contains the expected variants. Verify companies rows exist with correct normalized_name values. Verify relationship rows (TARGET_OF, ADVISOR_ON, etc.) have source_file_ids populated. Verify deal stage is inferred correctly from the doc_type.
  8. Test deal alias accumulation. Ingest two files from the same deal folder — one with code name “Project Atlas” and one with real name “Acme Corp.” Verify the deal_aliases table contains both names for the deal. Verify canonical_name matches the most recently modified file’s extracted name.
Key Takeaway

Phase 2 builds the data quality foundation. Every search result traces back to a chunk created here. The format-aware parsers preserve document structure (tables as tables, slides as slides). The chunking engine splits at logical boundaries with section context preserved. Hierarchical table splitting handles the financial models that are central to the firm’s work. And the three-tier entity extraction populates the entity store that makes deal-aware, entity-aware search possible.
