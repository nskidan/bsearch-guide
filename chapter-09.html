Chapter 9

Infrastructure & Deployment

What runs where, what it costs, and how changes get deployed without downtime.

This chapter maps BSearch’s components to the actual Azure services that host them, explains what each service costs, and describes how code and infrastructure changes get deployed. If the previous chapters explained what BSearch does, this chapter explains where it physically runs and how it gets there.

9.1 Azure Services Map

BSearch is built on a set of managed Azure services. “Managed” means Microsoft operates the underlying servers, handles patching, and guarantees uptime — you just configure and use them. Here is every service and what it does:

Azure Services

Service Azure Tier Monthly Cost Purpose
Azure AI Search S1 or S2 ~$250–500 The search index — stores all document chunks with both keyword (BM25) and vector indexes for hybrid search.
Azure App Service B1 ~$13 Hosts the FastAPI search API. Always-on, meaning it is running 24/7 and responds instantly to requests (no cold start delay).
Azure Durable Functions Consumption Pay-per-use Runs the ingestion pipeline — the multi-step process that pulls documents from SharePoint, parses them, chunks them, and loads them into the search index.
Azure Static Web Apps Free $0 Hosts the React web application (the BSearch user interface). Also handles the Microsoft login flow automatically.
Cosmos DB Serverless ~$0 Stores application state: conversation histories, SharePoint sync tokens, user preferences. At 18 users, costs are negligible.
Neo4j Aura Free / Pro $0–65 Knowledge graph — stores relationships between entities (companies, people, deals, industries) for relationship-aware search.
Azure Document Intelligence S0 Pay-per-page Extracts text and tables from PDFs. Understands document layout (headers, columns, tables) rather than just raw text extraction.
Key Vault Standard Minimal Stores API keys, connection strings, and other secrets securely. Services fetch their credentials from here instead of having them hard-coded.
Application Insights Included $0 Monitoring and alerting — tracks request latency, error rates, and custom events. Sends alerts when something goes wrong.

External API Costs

In addition to Azure services, BSearch calls several external AI APIs. These are usage-based — you pay for what you use:

API Monthly Cost Usage
Claude Sonnet 4.6 ~$11 Query synthesis — generates natural-language answers from search results. Estimated at 50–200 queries/day.
Claude Haiku 4.5 ~$0.10–0.40/day Lightweight tasks: document classification, metadata extraction, chunk summaries during ingestion.
Voyage AI Pay-per-token Generates vector embeddings for document chunks and search queries.
Cohere Rerank Pay-per-query Re-scores search results for precision (the “senior associate review” from Chapter 3).
IB Analogy

Think of the Azure services as the firm’s overhead — office lease, Bloomberg terminals, IT support. They are fixed or near-fixed monthly costs that keep the lights on. The external APIs are more like deal-related expenses — they scale with activity. More searches mean slightly higher API costs, just like more active deals mean more data room fees and printing costs. The total monthly run-rate is modest: roughly $300–600 all-in for 18 users.

9.2 Bicep Templates — Infrastructure as Code

Infrastructure as Code (IaC) — Defining your cloud infrastructure in text files rather than clicking through a web portal. The text files are version-controlled (tracked in Git), reviewable, and repeatable. If you need to rebuild everything from scratch, you run one command instead of clicking through 50 screens.

Bicep — Microsoft’s declarative language for defining Azure resources. You describe what you want (a search service with these settings, a database with this configuration), and Azure figures out how to build it.

BSearch’s infrastructure is defined in Bicep template files stored in the infra/ directory:

  • infra/main.bicep — The orchestrator. References all the individual modules and wires them together.
  • infra/search.bicep — Azure AI Search service configuration.
  • infra/functions.bicep — Azure Durable Functions for the ingestion pipeline.
  • infra/appservice.bicep — Azure App Service for the FastAPI backend.
  • infra/cosmos.bicep — Cosmos DB for state storage.
  • infra/staticwebapp.bicep — Azure Static Web Apps for the React front-end.
  • infra/keyvault.bicep — Key Vault for secrets management.

Separate parameter files define differences between environments (e.g., dev vs. prod). The dev environment might use cheaper service tiers, while prod uses the tiers listed in Section 9.1.

IB Analogy

Bicep templates are like a detailed engagement letter that specifies exactly what infrastructure to set up. Instead of calling Azure support and saying “we need a search service, a database, a web server…” and hoping they configure everything correctly, you hand them a precise, written specification. If something goes wrong and you need to rebuild, you just re-execute the same engagement letter — every setting, every connection, every configuration is documented and repeatable.

9.3 Deployment Process

Deployment — The process of taking code from your development environment and putting it into production where users can access it. In BSearch, there are three things that get deployed independently: infrastructure, the backend API, and the front-end web app.

BSearch has three deployment tracks:

  1. Infrastructure — Run az deployment group create with the Bicep templates. This creates or updates all Azure services to match the template definitions. If a service already exists and is configured correctly, Azure skips it.
  2. Backend (FastAPI API) — The Python code is packaged and deployed to Azure App Service via zip deploy or Git deploy. Once deployed, the App Service restarts and begins serving the new code.
  3. Frontend (React web app) — The React application is built and deployed to Azure Static Web Apps. This happens automatically through GitHub integration — when code is merged to the main branch, Static Web Apps detects the change and deploys.

The ingestion pipeline (Azure Functions) follows the same deployment mechanism as the backend — Python code packaged and deployed to the Functions service.

9.4 Monitoring — Application Insights

Application Insights — Azure’s monitoring service. It collects data about every request the application handles — how long it took, whether it succeeded or failed, and any custom events the code reports. Think of it as a real-time dashboard for the application’s health.

p50 / p95 / p99 latency — Percentile-based latency measurements. p50 means “half of all requests finish faster than this.” p95 means “95% of requests finish faster than this” — it captures the slow outliers. p99 captures the worst 1%. These tell you not just the average performance but how consistent it is.

BSearch tracks latency at every stage of the search pipeline: query parsing, embedding generation, index search, reranking, and answer synthesis. This makes it possible to pinpoint which stage is slow when a query takes longer than expected.

Key Alerts

The system sends alerts when something degrades or fails:

  • Reranker degraded — fires when Cohere’s reranker fails 3 or more times in a window. BSearch falls back to its own scoring, but results may be less precise.
  • Embedding fallback rate >10% — fires when more than 10% of embedding requests are falling back to a secondary provider, indicating the primary (Voyage) may be experiencing issues.
  • Search errors — fires when the search index returns errors, which would mean queries are failing entirely.

Custom Events

Beyond standard request metrics, BSearch logs custom events for operational awareness:

  • reranker_degraded / reranker_recovered — tracks when the system enters and exits fallback mode for reranking.
  • neo4j_fallback — the knowledge graph was unavailable, and the system fell back to search-only results.
  • embedding_fallback — primary embedding provider was unavailable; a backup was used.

Dashboard

A monitoring dashboard provides at-a-glance visibility into:

  • Query volume — how many searches per hour/day.
  • Time-to-first-token (TTFT) by query type — how quickly users start seeing results, broken down by whether the query was a simple lookup, a synthesis question, etc.
  • Cache hit rates — how often the system serves results from cache vs. running a fresh search.
IB Analogy

Application Insights is like the daily P&L and risk report for a trading desk. You do not wait until quarter-end to discover a problem — you check the dashboard every morning. If latency spikes or errors increase, you see it immediately and can investigate before users start complaining. The alerts are like stop-loss triggers: automated notifications when a metric crosses a threshold.

9.5 Blue-Green Re-Indexing

Blue-Green Deployment — A deployment strategy where you run two identical environments (“blue” and “green”) side by side. You deploy changes to the inactive environment, verify everything works, then switch traffic to it. If something goes wrong, you switch back. Zero downtime, instant rollback.

There is a specific scenario where blue-green deployment is essential for BSearch: switching embedding providers (e.g., upgrading from one version of Voyage to a newer one, or switching to a different provider entirely).

The Problem

Every document chunk in the search index has a vector embedding — a list of 1,024 numbers generated by a specific AI model (currently Voyage voyage-context-3). When you search, your query is also embedded by the same model, and the system compares the two vectors using cosine similarity.

Here is the critical constraint: you cannot mix vectors from different embedding models in the same index. Cosine similarity between vectors from Model A and vectors from Model B is meaningless — the numbers represent completely different coordinate systems. It would be like comparing GPS coordinates to postal codes.

So if you want to switch embedding models, you have to re-embed all ~2.2 million chunks. That takes 1–2 hours. You cannot take the search index offline for 1–2 hours. Hence: blue-green re-indexing.

  1. Create a parallel “green” index. A brand-new search index is provisioned alongside the existing “blue” (live) index.
  2. Re-embed all chunks. Every chunk is processed through the new embedding model, and the resulting vectors are stored in the green index. This takes approximately 1–2 hours for 2.2 million chunks.
  3. During re-indexing, new files go to both indexes. Any documents that are added or updated while re-indexing is in progress are written to both the blue and green indexes, so neither falls behind.
  4. Validate the green index. Before switching:
    • Chunk counts must match between blue and green.
    • A sample of chunks is checked to confirm provenance (they trace back to the correct source documents).
    • A set of acceptance queries is run against the green index to verify result quality.
    • Deduplication thresholds are recalibrated — different embedding models produce different similarity score distributions, so the threshold for “these two chunks are duplicates” may need adjustment.
  5. Flip the active index. Once validation passes, BSearch switches to serving queries from the green index. This is instantaneous from the user’s perspective.
  6. Keep the old index for 48 hours. The blue index remains available as a rollback target. If the new model produces unexpectedly poor results in production, one configuration change switches back to blue. After 48 hours of stable operation, the old index is decommissioned.
IB Analogy

This is like migrating to a new data room provider mid-deal. You do not shut down the old data room, move all the files, and hope nothing breaks. Instead, you set up the new data room in parallel, copy everything over, verify the folder structure and permissions match, and only then send the new link to buyers. The old data room stays active for a few days in case you need to fall back. That is blue-green re-indexing.

When does this happen?

Blue-green re-indexing is not a daily occurrence. It happens when BSearch upgrades its embedding model — perhaps once or twice a year, when a meaningfully better model becomes available. The process is fully automated; once triggered, it runs to completion without manual intervention.

Key Takeaway

BSearch runs on a set of managed Azure services with a total monthly cost of roughly $300–600 for 18 users. All infrastructure is defined in code (Bicep templates), making it repeatable and auditable. Deployments are split into three independent tracks (infrastructure, backend, frontend), and embedding model upgrades use a blue-green strategy to avoid any search downtime. Application Insights provides real-time monitoring with automated alerts when components degrade.
