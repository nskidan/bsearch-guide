Chapter 7

The Data Model

Where every piece of data lives, what it means, and how the three data stores work together.

BSearch stores data in three places, each chosen for what it does best. Azure AI Search holds the searchable index. Cosmos DB stores operational data like conversation history and sync state. Neo4j holds the knowledge graph of deal relationships. This chapter walks through every field and collection so you understand exactly what the system is tracking and why.

IB Analogy

Think of it like how a deal team organizes information. The search index is your filing system — every document catalogued and retrievable. Cosmos DB is your operational tracker — which deals are active, what synced last, what failed. Neo4j is your relationship map — who is advising whom, which companies are targets, who submitted an IOI. Different tools for different jobs.

7.1 Azure AI Search Index Schema

Index Schema — The blueprint that defines every field in the search index: its name, its data type, and whether it is searchable, filterable, or sortable. Like a database table definition, but optimized for search rather than transactions.

Every chunk (Chapter 3, Section 3.6) stored in Azure AI Search carries the following fields. They are grouped by purpose:

Core identification

Field Type Purpose
chunk_id String (key) Unique identifier for this chunk. The primary key of the index — no two chunks share the same ID.
file_id String Identifies which source file this chunk came from. All chunks from the same file share the same file_id.
chunk_index Integer Position of this chunk within its source file (0, 1, 2, ...). Lets the system reconstruct document order.

Content

Field Type Purpose
content String (searchable) The actual text of the chunk. This is what BM25 keyword search operates on.
content_vector Vector (1024 dimensions) The embedding — 1,024 numbers representing the meaning of the content (Chapter 3, Section 3.3). This is what vector search operates on.

File metadata

Field Type Purpose
source_folder String (filterable) The SharePoint folder path. Used for filtering (“only search this deal folder”) and dedup grouping.
file_name String Original filename. Displayed in citations (Chapter 6, Section 6.5).
file_type String (filterable) File extension — pdf, docx, pptx, xlsx, etc. Enables “show only PowerPoint results.”
m365_url String The Microsoft 365 deep link. Powers the “Open in SharePoint” citation links.
last_modified DateTime (sortable) When the source file was last modified. Used for recency boosting (Section 7.2) and determining canonical duplicates.

Document classification

Field Type Purpose
chunk_type String (filterable) What kind of content this chunk contains: narrative text, table, slide, header, etc.
doc_type String (filterable) The document category: CIM, engagement letter, financial model, pitch book, board presentation, etc. Enables “show only CIMs.”
doc_section String (filterable) Which section of the document this chunk belongs to: executive summary, financial overview, risk factors, etc.

Deal context

Field Type Purpose
deal_id String (filterable) Links this chunk to a specific deal in the Neo4j knowledge graph. Enables cross-referencing.
deal_name String (searchable) The deal’s display name. Searchable so users can find chunks by deal name.
deal_aliases String collection Alternative names for the deal (project code names, ticker symbols). Ensures “Project Falcon” finds “Acme Corp” results.

Entity fields

Entity fields are split into two categories that serve very different purposes (see Section 7.5 for the full explanation):

Field Scope Purpose
companies Chunk Company names mentioned in this specific chunk. Varies per chunk. “Which chunks mention Goldman?”
people Chunk Person names mentioned in this specific chunk. Varies per chunk. “Which chunks mention John Smith?”
company_roles Document Company-to-role mappings for the whole document. Same for all chunks in a file. “Chunks from deals where Goldman was advisor.”
industries Deal Industry classifications for the deal. Same for all chunks in the deal. “Chunks from healthcare deals.”
deal_stage Document Current stage of the deal when this document was created. “Chunks from deals in LOI stage.”
deal_status Deal Current deal status (active, closed, terminated). Same for all chunks in the deal.

Deduplication fields

Field Type Purpose
duplicate_group_id String (filterable) Groups duplicate files together. All chunks from files in the same duplicate cluster share this ID.
is_canonical Boolean (filterable) Whether this chunk belongs to the canonical (most recent) version. Search filters on is_canonical eq true to hide duplicates (Chapter 6, Section 6.4).

Embedding provenance

Field Type Purpose
embedding_model String Which model produced the embedding (e.g., voyage-context-3). Critical for tracking when we upgrade models.
embedding_dimensions Integer Number of dimensions (1024). Ensures compatibility across index versions.
embedding_version String Version identifier. Lets us identify which chunks need re-embedding after a model upgrade.

7.2 Scoring Profile

Scoring Profile — A set of rules that adjust the relevance score of search results based on field values. In BSearch’s case, it boosts scores for recently modified documents.

Azure AI Search supports built-in scoring profiles that multiply the base relevance score based on field values. BSearch uses a recency profile on the last_modified field:

Recency Window Score Multiplier
Modified within 30 days 1.5×
Modified within 90 days 1.2×
Older than 90 days 1.0× (no boost)

The boost is a multiplier, not an override. A highly relevant document from last year will still outrank a barely relevant document from last week. But when two documents are equally relevant, the newer one wins.

IB Analogy

In IB, the latest version is almost always the right one. If there are three drafts of a purchase agreement, you want the one from yesterday, not the one from six months ago. The scoring profile automates this instinct: recent documents get a nudge up the rankings, reflecting how deal teams actually work.

7.3 Cosmos DB Collections

Cosmos DB — Microsoft’s managed NoSQL database. “NoSQL” means it stores flexible documents (like JSON) rather than rigid rows-and-columns like a traditional database. Ideal for data that does not fit neatly into a spreadsheet.

Partition Key — The field Cosmos DB uses to distribute data across servers. All items with the same partition key are stored together for fast access. Choosing the right partition key is critical for performance.

BSearch uses five Cosmos DB collections for operational data that does not belong in the search index:

Conversations

Partition key: user_id

Stores the full conversation history for each user — their queries, the system’s responses, which documents were cited. Partitioning by user_id means all of one user’s conversations are stored together, making retrieval fast. This enables features like conversation continuity (“earlier you asked about Project Falcon...”).

Delta sync tokens

Stores a sync token for each SharePoint drive. When BSearch checks for new or updated files, it sends the token to Microsoft Graph, which responds with only the changes since that token was issued. This is how BSearch does incremental sync — processing only new/changed files instead of re-scanning everything.

Webhook subscriptions

Tracks active Microsoft Graph webhook subscriptions. Each record includes:

  • subscription_id — the unique subscription identifier from Microsoft.
  • drive_id — which SharePoint drive this subscription monitors.
  • expiration — when the subscription expires (webhooks must be periodically renewed).
  • client_state — a secret value used to verify that incoming webhook notifications are genuine (not spoofed).

Retry queue

When a file fails to ingest (network timeout, parsing error, API rate limit), it goes into the retry queue rather than being silently dropped. The queue implements exponential backoff — the wait time between retries doubles each attempt (1 min, 2 min, 4 min, 8 min...) to avoid hammering a struggling service.

Duplicate clusters

Stores the deduplication groups from Chapter 6, Section 6.4. Each record contains:

  • group_id — the shared identifier for all duplicates of the same document.
  • normalized_name — the filename after stripping version/copy markers.
  • members — the list of files in this cluster, each with a canonical flag indicating which is the “winner.”

7.4 Neo4j Graph Schema

Graph Database — A database designed to store and query relationships between entities. Instead of rows in tables, it stores nodes (things) and relationships (connections between things). Perfect for answering questions like “which companies are connected to this deal, and in what roles?”

Neo4j stores the knowledge graph — the web of relationships between deals, companies, people, and deal stages. This is the data layer that powers queries like “show me all deals where Goldman was the buy-side advisor” or “which deals has John Smith worked on?”

Node types

Deal

Represents an M&A transaction or advisory engagement. Properties:

  • deal_id, deal_name, deal_aliases — identification and code names.
  • deal_status — active, closed, or terminated. Inferred deterministically from stage history and last activity (not manually set).
  • Financial metrics: revenue, ebitda, ev (enterprise value), ev_ebitda (valuation multiple), offer_price.
  • last_activity — timestamp of the most recent document or stage change.

Company

Represents any company involved in a deal — target, acquirer, advisor, lender. Properties include name, aliases, and industry.

Person

Represents an individual — executive, deal team member, board member. Properties include name, title, and email where available.

StageEntry

A timestamped record of a deal reaching a particular stage. Properties: stage_name, entered_at, source_file_id (which document provided the evidence).

Relationship types

Relationship From → To Meaning
TARGET_OF Company → Deal This company is the target (seller) in the deal.
POTENTIAL_ACQUIRER Company → Deal This company is a potential buyer. Carries an engagement_level property (see below).
ACQUIRER_OF Company → Deal This company is the confirmed acquirer (deal closed).
ADVISOR_ON Company → Deal This company (bank, law firm, etc.) is an advisor on the deal.
LENDER_ON Company → Deal This company is providing debt financing for the deal.
EXECUTIVE_AT Person → Company This person is an executive at the company.
APPEARED_IN Person → Deal This person appeared in documents related to the deal.
REACHED_STAGE Deal → StageEntry The deal reached this stage at the recorded timestamp.

POTENTIAL_ACQUIRER engagement levels

The POTENTIAL_ACQUIRER relationship carries an engagement_level property that tracks how far a potential buyer has progressed in the process:

  1. teaser_received — the buyer received the initial teaser document.
  2. cim_received — the buyer received the CIM (signed an NDA, accessed the data room).
  3. ioi_submitted — the buyer submitted an Indication of Interest.
  4. loi_submitted — the buyer submitted a Letter of Intent.

These levels map directly to the standard sell-side M&A process. They enable queries like “which potential acquirers submitted an IOI on Project Falcon?”

Deal status inference

Deal status (active, closed, terminated) is not manually set. It is deterministically inferred from two inputs:

  • Stage history — the chain of REACHED_STAGE relationships. A deal that reached “closing” is closed.
  • Last activity — the timestamp of the most recent document or stage change. A deal with no activity in an extended period may be inferred as terminated.

This avoids the staleness problem of manual status tracking, where someone forgets to update a field and the data goes stale.

IB Analogy

The graph schema mirrors how deal teams already think about relationships. You know that on Project Falcon, Acme is the target, PE Fund X and Strategic Buyer Y are in the process, Goldman is the sell-side advisor, and Bank of America is the staple financing provider. The graph just codifies these relationships so a machine can traverse them. When you ask “who else has PE Fund X looked at?” the graph follows the edges from PE Fund X to every deal they have a POTENTIAL_ACQUIRER relationship with.

7.5 Entity Field Distinction

This is one of the most important conceptual distinctions in the data model. The search index has entity fields at three different scopes, and confusing them leads to wrong queries. Here is the breakdown:

Chunk-Scoped Fields

companies, people

Source: NER (Named Entity Recognition) — an AI model that reads each chunk and extracts names it finds.

Varies per chunk. Chunk 3 of a CIM might mention Goldman Sachs; chunk 7 might not.

Use case: “Which chunks mention Goldman?” — search relevance. Find the specific passages that reference a company or person.

Document-Scoped Fields

company_roles, deal_stage

Source: Haiku extraction — an AI model that reads the full document and extracts structured information.

Same for all chunks in the file. Every chunk in a CIM shares the same company_roles.

Use case: “Chunks from deals where Goldman was advisor” — role-based filtering. Not about which chunks mention Goldman, but which documents are from deals where Goldman has a role.

Deal-scoped fields

industries, deal_status

Source: Neo4j knowledge graph. These values are set at the deal level and propagated to every chunk in every document associated with that deal.

Same for all chunks in the deal (across all documents). A deal classified as “healthcare” means every chunk from every file in that deal folder carries industries: ["healthcare"].

Use case: “Chunks from healthcare deals” or “chunks from active deals” — deal-level filtering.

Why This Matters

Searching companies for “Goldman Sachs” returns chunks that mention Goldman by name. Filtering company_roles for Goldman as advisor returns chunks from deals where Goldman has a role, even if that specific chunk never mentions Goldman. These are fundamentally different questions, and using the wrong field gives you wrong results.

IB Analogy

Think about the difference between “find pages that mention JPMorgan” and “find documents from deals where JPMorgan is the advisor.” The first is a text search — does this page say “JPMorgan”? The second is a relationship query — is JPMorgan listed as an advisor on this deal? A page about market conditions might mention JPMorgan as a comp data source without JPMorgan being involved in the deal at all. The entity field distinction prevents that confusion.

Key Takeaway

BSearch’s data model is purpose-built for IB workflows. The search index stores every field needed for fast, filterable retrieval. Cosmos DB handles the operational plumbing (sync state, retries, conversations). Neo4j captures the relationship structure of deals in the same way deal teams already think about them. Entity fields exist at three scopes — chunk, document, and deal — because different questions require different levels of granularity. Understanding which field to query for which question is the key to getting accurate results.
