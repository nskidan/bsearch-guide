Chapter 7

When Things Go Wrong

How BSearch handles failures, communicates degradation, and recovers automatically.

No system runs perfectly all the time. External APIs time out, networks hiccup, services restart. The measure of a well-built system is not whether failures happen — it is how the system responds when they do. BSearch is designed so that most failures are invisible, a few trigger a gentle warning, and only a handful result in an actual error screen. This chapter walks through every failure category, the monitoring that detects problems, and the recovery paths that restore normal operation.

IB Analogy

Think of BSearch’s resilience like a deal team’s backup plan. If the MD is stuck in another meeting, the VP runs the call. If the model is not ready, the team presents from the memo. The deal does not stop — it continues with a slightly leaner team. Similarly, BSearch always tries to give you an answer, even if some of its best tools are temporarily unavailable.

7.1 Three Categories of Failure

Everything that can go wrong in BSearch falls into one of three categories, defined by what the user experiences:

Graceful Degradation — A design principle where the system continues to function at reduced quality when a component fails, rather than failing entirely. Like a car that loses power steering — harder to drive, but you can still get where you are going.

Category 1: Silent, auto-handled

The user never knows anything happened. The system catches the failure, applies a fallback, and continues. These are by far the most common failures, and they are invisible by design — showing an error for something that fixes itself in seconds would just be noise.

What Fails What Happens Instead User Experience
Embedding timeout (Voyage AI >500ms) Falls back to BM25-only search (keyword matching without semantic understanding) Results still appear; slightly lower recall on queries phrased differently from document text
Neo4j timeout (at query time) Search proceeds without entity filters, alias expansion, or deal context Results still appear; entity-based features temporarily unavailable
Cached data serves stale Embedding cache returns a previously computed vector; delta sync catches up later No visible impact; data refreshes within minutes to hours
Query classifier timeout (Haiku >500ms) Defaults to “data extraction” intent Slightly less optimal retrieval strategy; next query works normally
Conversation log write failure Fire-and-forget with logging; one history entry lost No visible impact; next conversation saves normally
Webhook missed Delta sync catches up within 12–24 hours Brief window where a newly added file is not yet searchable

Category 2: Visible degradation with banners

The user sees a colored banner warning that quality may be reduced, but still gets results. These failures are significant enough that the user should calibrate their trust in the output.

What Fails What Happens Instead What the User Sees
Reranker failure (Cohere down) Skip reranking; reduce candidate pool from 20 to 12; force confidence to LOW (never HIGH); use Azure Search’s built-in ranking Orange banner: “Search quality may be reduced — results are ranked without AI reranking. Verify key details against source documents.”
LOW confidence results (rerank scores 0.15–0.50) Claude adds hedging language to its synthesis Yellow banner: “Results may not fully answer your query”
Orange vs. Yellow Banners

Orange means a system component failed and BSearch is running in a degraded mode — results are still useful but may not be as well-ranked. Yellow means the system is working normally but is not confident it found great results for your particular query. Both are informational — you can still trust and use the results. A query can trigger both banners simultaneously.

Category 3: Hard errors

The user sees an actual error message. These are rare and mean a core service is down with no workaround.

What Fails What Happens What the User Sees
Azure Search down (503) The search index — the core of the system — is unreachable. There is nothing to search. “Service temporarily unavailable” error page
Claude API failure Raw search results are returned as clickable source cards without AI synthesis Error message with retry option; source documents still accessible
Authentication token expired Azure AD session needs renewal Brief redirect to the Microsoft login page
Embedding provider misconfigured App refuses to start entirely (fail-fast) App is down until configuration is fixed
Why Fail Fast on Configuration?

If the embedding provider or API keys are wrong, BSearch refuses to start rather than producing garbage results. A misconfigured system that appears to work is far more dangerous than one that clearly says “I cannot start — fix this first.”

7.2 Reranker Health Monitoring

Reranker — The Cohere AI model that re-scores search results by reading each document against the query (Chapter 5). It is the quality bottleneck — the difference between Claude synthesizing from the best five chunks versus mediocre ones.

Health Check — A lightweight background task that periodically pings an external service to verify it is responding. Like a doctor’s pulse check — fast, frequent, and catches problems before they affect the patient.

Silent reranker degradation is unacceptable in IB. A subtly wrong answer in a client deliverable is worse than a flagged one. So BSearch monitors the reranker proactively, not reactively.

How it works

A background task runs inside the BSearch application and pings the Cohere reranker every 60 seconds with a minimal test payload. The result is stored in memory — a simple status object tracking whether the service is healthy, when it was last checked, and how many consecutive failures have occurred.

When the health check fails:

  • Reranking is skipped — BSearch uses Azure Search’s built-in ranking instead.
  • The candidate pool shrinks — top-k drops from 20 to 12. Without the reranker to filter noise, a smaller pool produces cleaner results.
  • Confidence is capped at LOW — the system never reports HIGH confidence without the reranker, because it cannot be sure the best documents were selected. This forces Claude to add hedging language.
  • An orange banner appears — the user is told that search quality may be reduced and should verify key details against source documents.
  • Application Insights fires an alert after three or more consecutive failures, notifying the operations team.

Recovery

When the next 60-second health check succeeds, BSearch automatically resumes normal operation. The orange banner disappears. Confidence scoring returns to its full HIGH/LOW/NONE range. No manual intervention required.

IB Analogy

Think of it like knowing when one of your analysts is out sick. You do not cancel the deal — you adjust the workload, tell the client you are working on it, and resume full capacity when they are back. The health check is the office checking in every hour. The orange banner is the “we’re working with a leaner team today” note to the client. And when the analyst returns, everything goes back to normal automatically.

7.3 Embedding Resilience

Embedding — A list of numbers (a “vector”) that represents the meaning of a piece of text (Chapter 3). BSearch sends each query to the Voyage AI embedding service to convert the query text into a vector for semantic search.

BM25 — A traditional keyword-matching algorithm (Chapter 3). It finds documents that contain the exact words in your query. Effective for literal matches, but does not understand synonyms or paraphrases the way vector search does.

The embedding service is a parallel step in the query pipeline — it runs at the same time as query classification. When it is fast (100–300ms typical), no one notices. When it spikes beyond 500ms, BSearch needs a plan.

Three layers of protection

Layer 1: LRU cache (~1,000 entries). Before calling Voyage AI at all, BSearch checks an in-memory cache of recently computed embeddings. At 50–200 queries per day from 18 users, common patterns recur frequently (“latest [deal] deck,” “what’s [deal]’s EBITDA?”). Repeated queries skip the API call entirely — the embedding is served from memory in under a millisecond.

LRU Cache — Least Recently Used cache. A fixed-size store that keeps the most recently accessed items and automatically discards the oldest when full. Like a desk that only has room for 1,000 folders — when you need to add a new one, the one you have not touched in the longest time gets filed away.

Layer 2: 500ms hard timeout. If Voyage AI has not responded within 500 milliseconds, BSearch stops waiting and proceeds without the embedding. This prevents a slow external service from dragging down the entire query.

Layer 3: BM25-only fallback. When the embedding is unavailable (timeout or API error), BSearch falls back to keyword-only search. This loses semantic matching — queries phrased differently from the document text will have lower recall. However, the quality degradation is bounded: static synonym expansion and query expansion from the classifier have already enriched the BM25 search terms, and the Cohere reranker still does full cross-encoder scoring on the actual text (not the vectors), compensating for lost semantic matching.

File lookup queries skip embedding entirely

When BSearch detects a file lookup query (“find the Catalyst NDA”), it skips the Voyage AI call altogether. File lookups use keyword-heavy search weighted on the filename field. Vector similarity adds negligible value for “find me Report_v3.docx”-style queries, so skipping the embedding call saves 100–300ms with no quality loss.

Not surfaced to users

Unlike reranker failure, embedding fallback does not trigger a banner. It is transient (per-query, not systemic) and BM25 plus the reranker is still a reasonable retrieval path. BSearch logs the event for monitoring instead. An Application Insights alert fires if the fallback rate exceeds 10% over any one-hour window, signaling that the Voyage AI service may be degraded and needs attention.

7.4 Pessimistic Health Initialization

Pessimistic Initialization — A startup strategy where the system assumes all external services are down until proven otherwise. The opposite of optimistic (assuming everything works until it fails).

When BSearch starts up — after a deployment, a restart, or an infrastructure event — it does not assume that the Cohere reranker, Voyage AI embeddings, or Neo4j are available. All three are marked as unhealthy by default.

The first health check cycle runs immediately on startup, not after the first 60-second interval. Each service is only marked healthy after its first successful ping. Until that happens, BSearch operates in degraded mode for that service: BM25-only search (if Voyage AI has not yet been verified), no reranking (if Cohere has not responded), and no entity enrichment (if Neo4j has not confirmed availability).

Why this matters

Without pessimistic initialization, there is a brief window after startup where BSearch would try to use services that may not be ready yet. If the reranker happens to be down during a deployment, a user querying in that first minute could get silently degraded results with no warning. Pessimistic initialization closes that gap: the system starts in its safest mode and upgrades to full capability only after verifying each dependency.

IB Analogy

It is the principle of “trust but verify.” When you walk into the data room for the first time, you do not assume every document is organized and up to date — you check. You verify the CIM is the latest version, confirm the financial model ties out, and make sure the org chart reflects current management. Only after checking do you start working from those materials. BSearch applies the same discipline: do not assume the services are ready until you have confirmed it.

7.5 Recovery Paths

When a service goes down, how does BSearch get back to normal? Here are the specific recovery paths for each major failure scenario.

Neo4j goes down

Entity filters, alias expansion, and deal context become unavailable. But search still works — it just operates without graph intelligence. Queries degrade gracefully to text-only search: you can still find documents by content, but questions like “deals where Goldman was the advisor” or “what has PE Fund X looked at?” will not work until Neo4j recovers. When it comes back online, the health check detects recovery within 60 seconds, and full capability restores automatically. No manual steps needed.

Azure Search returns 503

This is a hard failure with no workaround. The search index is the search engine — without it, there is nothing to search. BSearch shows the “service temporarily unavailable” page. Behind the scenes, auto-retry with exponential backoff attempts to reconnect. Once Azure Search is reachable again, the next query works normally.

Exponential Backoff — A retry strategy where the wait time between attempts doubles each time (1 second, 2 seconds, 4 seconds, 8 seconds...). This avoids hammering a struggling service with rapid retries, giving it time to recover.

Reranker failure

Covered in detail in Section 7.2 above. The health check detects the outage within 60 seconds, BSearch switches to degraded mode with the orange banner, and automatically resumes when Cohere recovers.

Deal summary debouncing

Debouncing — Waiting for a burst of activity to settle before taking action. Like waiting for all the dishes to hit the sink before starting the dishwasher, rather than running it after every plate.

When many files in a deal update at once — like uploading 50 documents to a data room — BSearch does not regenerate the deal summary after each file. Instead, it waits five minutes after the last change before regenerating. This prevents unnecessary repeated work: without debouncing, a 50-file upload would trigger 50 consecutive summary regenerations, each one immediately obsoleted by the next.

The debounce state is stored in Cosmos DB (not in-memory), so it survives application restarts. A timer checks every minute for deals that have been quiet for at least five minutes and processes them. If the debounce timer is missed due to a process restart, the six-hour periodic refresh cycle catches up — the worst case is a deal summary that reflects the state from six hours ago, while individual document chunks remain current and searchable.

IB Analogy

When a seller uploads a batch of diligence materials to the data room, you do not update the summary memo after each individual file arrives. You wait until the upload is done, review everything together, and then produce one comprehensive update. Debouncing applies the same common sense to automated summaries.

7.6 Five Resilience Principles

Every resilience decision in BSearch follows these five principles. They explain not just what the system does, but why it does it that way.

1. Degrade gracefully, never silently fail

If a service is down, reduce quality but keep working. And always tell the user when quality is reduced. The orange banner for reranker failure embodies this: BSearch does not pretend everything is fine when it is running without its best ranking tool. Silent degradation is unacceptable in IB — a subtly wrong answer in a client deliverable is worse than a flagged one.

2. Pessimistic on startup, optimistic on recovery

Assume everything is broken until proven healthy (Section 7.4). Once healthy, assume it stays healthy — with continuous monitoring via 60-second health checks. This is asymmetric by design: the cost of using an unhealthy service (bad results) is higher than the cost of briefly operating in degraded mode (slightly slower or less refined results).

3. Cache what you can

Query embeddings, health status, AD group memberships — caching reduces dependency on external services. Every cache hit is a call you did not have to make, which means one fewer opportunity for a timeout or error to affect the user. The embedding LRU cache (Section 7.3) is a prime example: at 10–15% hit rates, it meaningfully reduces the number of queries exposed to Voyage AI latency spikes.

4. Fast fallbacks over slow retries

A quick fallback is better than making the user wait for a retry that might also fail. When the embedding service times out after 500ms, BSearch immediately falls back to BM25-only search rather than retrying and potentially doubling the latency. The user gets results in under three seconds. The quality is slightly lower, but the experience is fast and functional.

5. Monitor everything, alert on patterns

Individual failures are noise. Patterns of failures are signals. A single embedding timeout is normal. Three consecutive reranker failures trigger an alert. A fallback rate above 10% over an hour signals a systemic issue. BSearch is instrumented to track every failure, every fallback, every degraded query — but alerts only fire on sustained patterns, not individual events. This keeps the operations team focused on real problems, not transient blips.

Key Takeaway

BSearch is designed to bend, not break. The majority of failures are invisible and self-healing. The few that are visible are clearly communicated with banners. Even in the worst realistic scenario — embedding service down, reranker down, Neo4j unreachable — BSearch still returns results. They come from BM25 keyword matching with Azure Search’s default ranking and no entity enrichment. That is a meaningfully worse experience than the full pipeline, but it still works. The floor is never zero.
