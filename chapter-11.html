Chapter 11

Key Design Decisions

All 18 architecture decisions: what was chosen, why, and when to revisit.

Building BSearch required hundreds of small choices, but 18 of them fundamentally shaped the system. Each decision involved real trade-offs — speed vs. quality, cost vs. capability, simplicity vs. flexibility. This chapter documents every one of them: what was chosen, the reasoning behind it, what alternatives were considered, and under what conditions the decision should be revisited.

How to read this chapter

Each decision is presented as an expandable card. Click any card to see the full reasoning. Decisions are grouped by category — infrastructure, search quality, resilience, and AI models — so you can focus on the area that matters most to you. If you’re reviewing the system for the first time, start with Infrastructure to understand the foundation, then read Search Quality to see how relevance is tuned.

11.1 Infrastructure

These decisions define where BSearch runs, how code is organized, and where data lives. They set the cost floor, operational complexity, and deployment model for everything else.

#1 — Bulk Ingestion (550K Documents)

Decision: Use Azure Durable Functions with the orchestrator pattern — fan-out/fan-in with checkpointing — to process all 550K SharePoint documents. The same codepath handles both the initial bulk load and ongoing synchronization.

Reasoning: Processing 550K documents is a multi-hour job that will inevitably encounter transient failures — network timeouts, rate limits from Microsoft’s Graph API, temporary service outages. Durable Functions solve this by automatically checkpointing progress. If the process crashes at document 300,000, it resumes from that point rather than starting over. Fan-out/fan-in means the system can process many documents in parallel (fan-out), then collect and finalize all results (fan-in). Using the same code for both bulk and ongoing sync eliminates an entire class of bugs where the two paths handle documents differently.

Alternatives Considered: A simple loop-based script (fragile, no checkpointing, restart = start over), a dedicated batch processing framework like Apache Spark (massive overkill for our scale and team size), or separate codepaths for bulk vs. incremental (doubles maintenance burden and divergence risk).

Revisit When: Document count exceeds 5M, processing time exceeds 48 hours, or Azure Durable Functions pricing changes significantly.

#2 — State Persistence: Cosmos DB + Neo4j

Decision: Use two specialized databases. Cosmos DB (serverless tier) stores conversations, sync tokens, webhook registrations, and the retry queue. Neo4j Aura stores the knowledge graph — deals, companies, people, and their relationships. Cosmos DB costs effectively $0 at 18 users; Neo4j costs ~$65/month in production (free tier for development).

Reasoning: Each database excels at a different job. Cosmos DB is ideal for simple key-value lookups that need to be fast and reliable — “what was the last sync token for this SharePoint site?” or “what messages are in this conversation?” Neo4j is purpose-built for relationship queries — “what deals involved Company X?” or “who worked on deals in the healthcare sector?” Trying to force either workload into a single database would mean worse performance, more complex queries, or both. Cosmos DB’s serverless tier means we pay only for actual usage, which at 18 users is negligible.

Alternatives Considered: A single relational database like PostgreSQL (workable but clumsy for graph queries and overkill for simple key-value patterns), Cosmos DB for everything (poor at relationship traversal), or Neo4j for everything (expensive for simple key-value storage).

Revisit When: Neo4j Aura pricing increases substantially, user count grows beyond 100 (Cosmos DB costs become non-trivial), or a unified database emerges that handles both workloads well.

#3 — Frontend Stack: Vite + React SPA on Azure Static Web Apps

Decision: Build the search interface as a single-page application using React (with Vite as the build tool), deployed to Azure Static Web Apps. This gives us built-in Azure AD authentication, CDN-served static files, and zero server operations for the frontend.

Reasoning: A search interface is inherently interactive — users type queries, refine results, expand documents, and navigate conversations. A single-page app handles this smoothly without full page reloads. Azure Static Web Apps is the natural deployment target because it integrates directly with our existing Azure AD setup (everyone at Berenson already has an Azure AD account), serves files from a global CDN for fast load times, and requires no server to manage. Vite makes the development experience fast with near-instant hot reloading.

Alternatives Considered: Server-rendered pages with Flask/Django (simpler but slower interaction, requires a server), a full Next.js app (more complexity than needed for a single-purpose search tool), or a SharePoint embedded web part (locked into SharePoint’s UI constraints).

Revisit When: SEO becomes a requirement (unlikely for an internal tool), the app grows beyond search into a multi-page platform, or Azure Static Web Apps pricing changes.

#4 — FastAPI on Azure App Service (B1)

Decision: Deploy the Python API backend on Azure App Service at the B1 tier (~$13/month) with always-on enabled. This avoids cold starts, which would violate the time-to-first-token (TTFT) target.

Reasoning: The TTFT target requires the API to begin responding within a few seconds. Serverless options like Azure Functions have cold starts of 5–15 seconds when they haven’t been used recently — common during off-hours or weekends, but devastating for user experience when someone runs a Monday morning search. The B1 tier keeps one instance warm at all times. At $13/month, the cost of eliminating cold starts is trivial compared to the productivity cost of slow searches. FastAPI is the Python framework because it’s async-native (important for calling multiple external APIs in parallel) and has excellent type validation built in.

Alternatives Considered: Azure Functions with premium plan (eliminates cold starts but costs ~$50+/month for always-warm), Azure Container Apps (more operational complexity), or Flask (synchronous, harder to parallelize external API calls).

Revisit When: Traffic grows enough to need auto-scaling beyond a single B1 instance, or Azure introduces a serverless option with sub-second cold starts.

#8 — Monorepo with Single pyproject.toml

Decision: Keep all Python code in a single repository with one pyproject.toml. Shared code is accessed via standard Python imports. The web frontend lives in a separate web/ directory with its own Node.js project configuration.

Reasoning: With a single developer (you), the overhead of managing multiple repositories — separate version control, separate CI pipelines, cross-repo dependency management — adds friction with zero benefit. A monorepo means one place to search, one branch to manage, one CI pipeline, and the ability to make a change that touches both the ingestion pipeline and the API in a single commit. Shared utilities (like the embedding provider abstraction or configuration loading) are just Python modules imported directly, not published packages.

Alternatives Considered: Separate repos per component (ingestion, API, frontend) with published internal packages (standard for large teams but massive overhead for a solo developer), or a single repo with multiple pyproject.toml files (unnecessary complexity).

Revisit When: A second developer joins and needs to work on a component independently, or the codebase grows large enough that CI times become painful.

#18 — Three-Tier Configuration Organization

Decision: Split configuration into three tiers. Module-level constants hold algorithm values (thresholds, weights) directly in the code files that use them. External infrastructure settings (API keys, connection strings) come from environment variables, not config files. Settings with defaults (~25 keys) live in a central Settings class and can be overridden via environment variables.

Reasoning: Putting everything in one big config file leads to bloat — a 200-line config where most values never change and the ones that matter are buried. Module-level constants (like the confidence threshold of 0.50) belong next to the code that uses them, so anyone reading that code can immediately see the value without jumping to a separate file. External infrastructure settings belong in environment variables because they differ between development and production. The central Settings class handles the ~25 values that have sensible defaults but might need tuning.

Alternatives Considered: A single config file (bloated, hard to navigate), environment variables for everything (awkward for algorithm tuning), or a YAML/JSON config file (adds a parsing dependency and file management for minimal benefit).

Revisit When: The number of configurable values exceeds 50, or a non-developer needs to change algorithm parameters without touching code.

11.2 Search Quality

These decisions directly affect whether users find what they’re looking for. They cover everything from how text is converted into searchable form, to how results are ranked, to how multi-document answers are synthesized.

#5 — Embedding Model: Voyage voyage-context-3 at 1024 Dimensions

Decision: Use Voyage AI’s voyage-context-3 model to convert text chunks into numerical vectors (embeddings) at 1024 dimensions. This model supports contextual chunk embeddings and a 32K-token context window. It outperforms OpenAI’s text-embedding-3-large by +14.24% on our benchmarks. Fallback model: voyage-4-large.

Reasoning: The embedding model is arguably the most important choice in the entire system — it determines how well the search engine understands what a document is about. Voyage’s contextual embeddings mean each chunk is embedded with awareness of its surrounding document, not in isolation. This is critical for investment banking documents where a table of financial metrics is meaningless without knowing which company and deal it belongs to. The 14.24% improvement over OpenAI translates directly to better search results — more relevant documents surfaced, fewer irrelevant ones.

Alternatives Considered: OpenAI text-embedding-3-large (simpler integration since we already use OpenAI/Anthropic, but measurably worse quality), Cohere Embed v3 (strong but not context-aware at chunk level), or open-source models like E5 (requires self-hosting infrastructure).

Revisit When: OpenAI or Cohere release context-aware embedding models that close the quality gap, Voyage pricing increases significantly, or our benchmarks show quality regression on new document types.

#9 — Three-Tier Relevance Gating on Reranker Scores

Decision: After retrieving candidate results, run them through a Cohere reranker and apply a three-tier confidence gate: HIGH (≥0.50), LOW (0.15–0.50), and NONE (<0.15). Individual chunks below 0.10 are dropped entirely. Thresholds are configurable and will be calibrated after launch.

Reasoning: Not all search results are equally relevant. Without gating, the system would either show irrelevant results (frustrating) or require users to manually judge relevance (slow). The three-tier system gives the AI synthesizer clear guidance: HIGH-confidence results can be cited directly, LOW-confidence results should be mentioned with caveats, and NONE means “don’t even try to answer.” The per-chunk floor of 0.10 prevents truly unrelated text from polluting the context. Making thresholds configurable means we can tune them based on real user feedback rather than guessing.

Alternatives Considered: A simple binary threshold (too rigid — misses the nuance between “probably relevant” and “definitely relevant”), no gating (shows everything, leaves users to judge), or user-configurable strictness (adds UI complexity for a niche need).

Revisit When: Post-launch data shows the thresholds need significant adjustment, or user feedback indicates the tiers don’t match their perception of relevance.

#10 — Query Classification: Regex Fast Path + Haiku Fallback

Decision: Classify incoming queries using a two-step approach. First, try regex pattern matching for unambiguous file lookups (e.g., “find the CIM for Acme Corp”). If regex doesn’t match, send the query to Claude Haiku for classification (~150ms, run in parallel with embedding). If Haiku doesn’t respond within 500ms, default to data_extraction query type.

Reasoning: Many searches are simple file lookups that don’t need AI classification — a regex can handle “find the NDA for [company]” in microseconds. But natural language queries like “what were the key risk factors in the last healthcare deal?” need an AI model to understand intent. Running Haiku in parallel with embedding means classification adds zero latency to the critical path. The 500ms timeout with a sensible default ensures the system never stalls waiting for classification.

Alternatives Considered: AI classification for all queries (adds latency for simple lookups), regex only (can’t handle natural language), or a local ML model for classification (requires training data we don’t have yet).

Revisit When: The regex patterns grow complex enough to become a maintenance burden, or Haiku latency consistently exceeds 200ms.

#11 — Query Expansion: Static Synonyms + Haiku Expansion

Decision: Expand queries using two complementary methods. A static synonym map for investment banking terminology (e.g., “CIM” → “Confidential Information Memorandum”) is applied to keyword search at zero latency cost. Haiku generates broader expansions in the same classification call (also 0ms added). The Cohere reranker scores expanded results against the original query to filter noise.

Reasoning: IB professionals use jargon interchangeably — “CIM,” “book,” “offering memorandum,” and “info memo” all mean the same thing. Without synonym expansion, searching for “CIM” would miss documents titled “Confidential Information Memorandum.” Haiku expansion catches synonyms we haven’t manually mapped. Scoring against the original query (not the expanded one) prevents expansion from pulling in tangentially related results — if Haiku over-expands, the reranker catches it.

Alternatives Considered: Synonym expansion only (misses unexpected terminology), Haiku expansion only (adds latency if done as a separate step), or no expansion (users must guess the exact terms used in documents).

Revisit When: The synonym map exceeds 500 entries and becomes hard to maintain, or Haiku expansion consistently introduces noise that the reranker doesn’t catch.

#13 — Chunking Quality: Hierarchical Splitting + Doc Summaries

Decision: Split documents into search-ready chunks using structure-aware logic. Large tables (>1500 tokens) are split at row-group boundaries rather than arbitrary token limits. Documents with 4 or more chunks get a Haiku-generated summary that bridges information across sections.

Reasoning: Naive chunking — splitting every N tokens regardless of content — destroys the meaning of tables (a row might be split from its header) and loses cross-section context (the “Risk Factors” section references the “Business Overview” but they end up in separate chunks with no link). Structural splitting preserves table integrity by keeping headers with their data rows. Document summaries give the AI synthesizer a birds-eye view of long documents, so it can understand how chunks relate to each other even when it only sees a few of them.

Alternatives Considered: Fixed-size chunking (simple but destroys tables and context), page-level chunking (too coarse for targeted retrieval), or no chunking with full-document retrieval (exceeds context window limits for large documents).

Revisit When: Embedding models with 128K+ context windows become affordable (reducing the need for chunking), or user feedback shows specific document types being chunked poorly.

#17 — Multi-Document Synthesis: Tiered Chain-of-Thought + Deal Summaries

Decision: For queries that span multiple documents, use Claude Sonnet with structured chain-of-thought reasoning (~2000 tokens of “extended thinking”) and pre-computed deal summaries. Accept a relaxed 5-second TTFT target for these complex queries. Two-pass synthesis (draft then refine) is deferred to v2.

Reasoning: Multi-document queries like “compare the financial profiles of our last three healthcare deals” are the hardest thing BSearch does. The AI needs to read chunks from multiple documents, understand which deal each belongs to, extract comparable data points, and synthesize a coherent answer. Extended thinking lets the model reason through this step by step rather than generating a hasty answer. Pre-computed deal summaries reduce the cognitive load at query time — instead of piecing together deal context from raw chunks, the model can reference a summary. The 5-second TTFT is acceptable because users asking complex analytical questions expect a thoughtful response.

Alternatives Considered: Simple concatenation of per-document answers (loses cross-document comparison), two-pass synthesis in v1 (adds latency and complexity for a v1 launch), or limiting to single-document answers only (too restrictive for IB workflows).

Revisit When: User feedback shows synthesis quality is insufficient (trigger for adding two-pass), or faster models make two-pass feasible within latency targets.

11.3 Resilience

These decisions ensure BSearch keeps working — or degrades gracefully — when external services fail, data is imperfect, or providers need to be swapped.

#6 — Webhook Reliability: Primary Webhooks + Delta Sync Fallback

Decision: Use Microsoft Graph API webhooks as the primary notification mechanism (30–120 second latency) with a scheduled delta sync running every 12–24 hours as a fallback. Both feed into the same processing pipeline.

Reasoning: Webhooks give near-real-time awareness of document changes, but they’re not 100% reliable. Microsoft may fail to deliver a notification, a network partition could block receipt, or BSearch could be restarting when a webhook arrives. The delta sync acts as a safety net — it asks Microsoft “what changed since my last check?” and catches anything the webhooks missed. Running both through the same pipeline means a document is processed identically regardless of how it was discovered, eliminating a class of bugs where webhook-discovered vs. delta-discovered documents behave differently.

Alternatives Considered: Webhooks only (risks missing changes), polling only (12–24 hour latency for all changes, unacceptable for active deals), or separate pipelines per mechanism (doubles the code to test and maintain).

Revisit When: Microsoft Graph webhook reliability improves to the point where misses are near-zero, or a third notification mechanism becomes available.

#12 — Reranker Degradation: Health Check + Transparency

Decision: Proactively check Cohere reranker health every 60 seconds with a ping. When the reranker is down, force all results to LOW confidence and show users an orange banner explaining that result quality may be reduced. No secondary reranker in v1 — transparency over silent quality degradation.

Reasoning: The reranker is one of the most important components for result quality — it separates truly relevant results from merely keyword-matching ones. When it goes down, the system could silently fall back to unranked results, but users would see noticeably worse quality without understanding why. The orange banner sets expectations honestly. Forcing LOW confidence prevents the AI from treating unreranked results with the same authority as properly ranked ones. A secondary reranker was considered but adds complexity for a failure mode that should be rare — if Cohere is down for more than a few minutes, something unusual is happening.

Alternatives Considered: A secondary reranker from a different provider (complexity and cost for a rare scenario), silent degradation (users lose trust without knowing why), or blocking searches entirely when the reranker is down (too aggressive for a non-critical failure).

Revisit When: Cohere experiences repeated outages, or a high-quality secondary reranker becomes available at minimal additional cost.

#14 — Embedding Latency Resilience: Cache + BM25 Fallback

Decision: Cache recent query embeddings in an LRU cache (~1000 entries). If the embedding API doesn’t respond within 500ms, fall back to BM25-only search (keyword matching). File lookups skip embedding entirely. Fallback is not surfaced to users. Alert triggers if fallback rate exceeds 10%.

Reasoning: The embedding API is an external dependency that adds latency to every search. Caching means repeated or similar queries don’t need to call the API at all. The 500ms timeout prevents a slow embedding API from making the entire search feel sluggish — BM25 keyword search is less sophisticated but returns results instantly. File lookups (e.g., “find the Acme NDA”) are pure metadata queries that don’t benefit from semantic understanding, so they skip embedding entirely. Not surfacing the fallback avoids alarming users for what is usually a brief hiccup. The 10% alert threshold catches systematic problems before they affect most users.

Alternatives Considered: No caching (unnecessary API calls for repeated queries), a longer timeout (degrades user experience), or surfacing the fallback to users (creates anxiety for a transparent recovery that usually resolves in seconds).

Revisit When: BM25 fallback rate consistently exceeds 5% (indicates chronic embedding API issues), or cache hit rates are low enough to question the memory cost.

#15 — Near-Duplicate Deduplication

Decision: Detect near-duplicate documents at ingestion time using two signals: filename normalization (e.g., “Acme CIM v2.docx” and “Acme CIM v2 (1).docx”) and embedding cosine similarity ≥0.95, constrained to the same folder. The most recent version becomes canonical. Non-canonical documents are filtered out at query time using an is_canonical flag (0ms cost).

Reasoning: SharePoint is full of near-duplicates — download copies, version variants, and files moved between folders. Without dedup, a search for “Acme CIM” might return the same document five times, pushing genuinely different results off the first page. The same-folder constraint prevents false positives where two legitimately different documents happen to be similar (e.g., CIMs for different deals that follow the same template). Filtering at query time via a pre-computed flag means dedup adds zero latency to searches.

Alternatives Considered: No dedup (polluted results), hash-based exact dedup only (misses near-duplicates like “v2” vs. “v2 final”), or dedup at query time (adds latency to every search).

Revisit When: The 0.95 cosine threshold produces false positives (legitimate documents marked as duplicates) or false negatives (obvious duplicates not caught).

#16 — Embedding Provider Portability

Decision: Abstract the embedding provider behind a Python abstract base class (EmbeddingProvider). Store chunk provenance metadata (which model generated each embedding). Support blue-green re-indexing for zero-downtime provider switches.

Reasoning: The embedding model market is evolving rapidly — today’s best model may be surpassed in months. Without an abstraction layer, switching providers would require changing code in every file that calls the embedding API, plus re-indexing all 550K documents with downtime. The abstraction means switching providers is a configuration change plus a re-indexing job. Blue-green re-indexing runs the new embeddings alongside the old ones, switches search traffic to the new index once it’s ready, then retires the old one — zero downtime. Provenance metadata lets us verify that all chunks have been re-embedded and identify any stragglers.

Alternatives Considered: Direct API calls without abstraction (fast to build, painful to change), a full plugin system (overengineered), or re-indexing with downtime (unacceptable for a production system).

Revisit When: The embedding market stabilizes around a clear winner, or the abstraction layer becomes a bottleneck for adding provider-specific features.

11.4 AI Models

This decision governs which AI models power BSearch’s intelligence — the reasoning behind answers, the classification of queries, and the extraction of structured data from documents.

#7 — Claude Model Selection: Sonnet for Synthesis, Haiku for Classification

Decision: Use Claude Sonnet 4.6 for answer synthesis and multi-document reasoning (~$11/month at projected usage). Use Claude Haiku 4.5 for lightweight tasks: query classification, entity extraction, synonym expansion, and document summarization during ingestion.

Reasoning: Not all AI tasks require the same level of intelligence. Synthesis — reading multiple document chunks, understanding their relationships, and generating a coherent answer with citations — is genuinely hard and benefits from Sonnet’s stronger reasoning ability. But classification (“is this a file lookup or an analytical question?”) and extraction (“what company is mentioned in this chunk?”) are simpler tasks where Haiku performs nearly as well at a fraction of the cost and latency. Using Haiku for these tasks keeps the per-query cost low and, critically, keeps classification fast enough to run in parallel with embedding without adding to the critical path. At ~$11/month total for Sonnet, the cost is negligible compared to the value of high-quality answers.

Alternatives Considered: Sonnet for everything (unnecessarily expensive and slow for simple tasks), Haiku for everything (insufficient quality for multi-document synthesis), GPT-4o (comparable quality but higher cost and no Anthropic ecosystem benefits), or open-source models (requires GPU hosting infrastructure we don’t have).

Revisit When: A new Claude model offers Sonnet-level quality at Haiku-level pricing, GPT pricing becomes significantly more competitive, or query volume increases enough that Sonnet costs become material.

Key Takeaway

These 18 decisions share a common philosophy: choose the simplest option that meets the quality bar, keep costs proportional to a small user base, build in escape hatches for the future, and be transparent with users when things degrade. None of them are permanent — every decision includes a “Revisit When” trigger so you know exactly what conditions would warrant a different choice.
