Chapter 4

How Documents Get Into the System

Format-aware parsing, intelligent chunking, summaries, deduplication, and entity extraction — the full journey from SharePoint file to searchable index entry.

Before anyone can search for a document, BSearch has to process it. Every file in SharePoint — every CIM, teaser, model, NDA, and engagement letter — must be discovered, downloaded, parsed, classified, sliced into searchable pieces, converted into mathematical representations, and loaded into the search index. This chapter walks through that journey, focusing on the decisions that make ingestion work well rather than just work at all.

IB Analogy

Think of the ingestion pipeline like the onboarding process for a new deal. A pitch deck doesn’t just appear in your working folder ready to reference — someone has to receive it, file it in the right deal folder, tag it with deal metadata, extract key terms, and make sure it’s accessible to the right people. The ingestion pipeline automates all of that for every document across the entire firm.

4.1 The Pipeline at a Glance

A document enters at the top and exits at the bottom as a set of searchable, AI-ready index entries. Each step feeds its output into the next. If any single step fails for a document, that document gets queued for retry rather than blocking the rest of the pipeline.

flowchart TD
    A["1. Discovery\nDetect new/changed files"] --> B["2. Download\nFetch from Graph API"]
    B --> PDF{"PDF with\nOffice companion?"}
    PDF -- Yes --> SKIP["Skip PDF\n(Office file has better structure)"]
    PDF -- No --> C["3. Parsing\nFormat-aware text extraction"]
    C --> D["4. Metadata & Entities\nClassify, assign deal, extract entities"]
    D --> E["5. Chunking\nSplit into index-ready pieces"]
    E --> F["6. Summaries\nDoc + deal-level summaries"]
    F --> G["7. Embedding\nConvert chunks to vectors"]
    G --> H["8. Deduplication\nFilename + embedding similarity"]
    H --> I["9. Indexing\nPush to Neo4j + Azure AI Search"]

The rest of this chapter covers steps 3 through 9 in depth. Discovery (webhooks and delta sync) and download are covered in Chapter 7: The SharePoint Connection. Here we focus on what happens after BSearch has a file in hand.

4.2 Format-Aware Parsing

Parsing: Extracting structured content — text, headings, tables, formulas — from a file’s internal format. A Word document is not plain text; it’s a compressed archive of XML files defining paragraphs, styles, and embedded objects. Parsing cracks that container open and pulls out what matters.

Every file format stores information differently, which means every format needs its own parser. A one-size-fits-all approach would butcher the structure that makes documents useful.

Format Parser What Gets Extracted Why This Parser
Word (.docx) python-docx Paragraphs, headings (with hierarchy), tables, styles Reads Office XML directly — fast, free, reliable
Excel (.xlsx) openpyxl Sheet names, column headers, cell values, formula text (sheet by sheet), bold/indent formatting Preserves formatting signals needed for table splitting
PowerPoint (.pptx) python-pptx Slide title, body text, tables, speaker notes (slide by slide) Slide-by-slide extraction maps naturally to chunking
PDF Azure Document Intelligence Text, tables, layout structure (handles scanned docs, complex layouts, watermarks) Cloud AI service built for document understanding

The first three parsers are Python libraries that read the Office XML formats directly. They’re fast, free, and reliable for well-formed documents. PDF parsing is a different story entirely.

Why PDFs Are Hard

PDFs in investment banking are notoriously difficult. Scanned CIM pages, multi-column layouts, embedded financial tables, watermarked confidential pages — simple PDF-to-text libraries butcher these. Azure Document Intelligence is a cloud AI service specifically built for document understanding. It handles the complex layouts that simpler parsers cannot, but it comes at a per-page cost.

Why Not Use Azure Document Intelligence for Everything?

It’s overkill for Word and Excel files, which have perfectly structured XML internals. Using it for everything would be slower (cloud API round-trip vs. local parsing) and more expensive (per-page pricing). We use the right tool for each job.

PDF Pre-Dedup: Skipping Redundant PDFs

In Berenson’s SharePoint, many documents exist as both an Office file (the editable original) and a PDF export (the “printed” copy). When both live in the same folder, the PDF is redundant — and the Office file is always better because it preserves internal structure (headings, cell formatting, indent levels) that the PDF flattens away.

Before downloading or parsing any PDF, BSearch checks whether a matching Word, Excel, or PowerPoint file exists in the same folder. The matching logic normalizes filenames — Office files typically include a version suffix (“Project Atlas - Board Deck - 2025.03.15 v.003.pptx”) while PDFs omit it (“Project Atlas - Board Deck - 2025.03.15.pdf”). If a match is found, the PDF is skipped entirely — no download, no parsing, no indexing.

Why This Matters

Roughly 75% of Berenson’s ~225,000 PDFs are companion copies of Office files. Skipping them saves approximately $22,000 in Azure Document Intelligence costs during bulk ingestion — and produces better search results, since the Office-parsed versions retain more structure.

If the folder listing fails (a Graph API error), BSearch fails open — it ingests the PDF normally rather than risk missing content. A redundant PDF that slips through is caught later by the near-duplicate detection step.

4.3 Chunking: Breaking Documents into Searchable Pieces

Chunking: Breaking a document into smaller pieces (“chunks”) sized for the search index. A 100-page CIM can’t be searched as one unit — it needs to be split into pieces that are small enough to be relevant but large enough to be meaningful.

Different content types need different chunking strategies. A paragraph of narrative text behaves very differently from a financial table or a slide deck. Getting the chunk boundaries right is one of the most consequential decisions in the pipeline — bad boundaries mean the search system either retrieves too much noise or too little context.

IB Analogy

Chunking is like how you break up a CIM for a buyer’s internal review. You wouldn’t hand the CFO the entire 200-page document — you’d split it into the exec summary, the financial section, the industry overview, and each major exhibit. Each piece is self-contained enough to be useful on its own, but together they form the complete picture.

Narrative Content (Word, PDF)

Target size: 800–1,000 tokens per chunk, with roughly 100 tokens of overlap between adjacent chunks. Splits happen at natural boundaries — paragraph breaks, section headings — never mid-sentence.

Each chunk is prepended with its section heading so it retains context when searched in isolation. A chunk from a CIM’s “Industry Overview” section carries that heading, so the search system knows it’s about industry context even if the chunk text itself doesn’t mention the section name.

Slides (PowerPoint)

One chunk per slide, prefixed with the slide number and title: [Slide 3: Management Team]. Slides are already natural units of meaning — each one typically covers a single topic or data point.

Formulas (Excel)

Excel formula text (the formulas themselves, not their computed values) gets a separate chunk per key sheet. This lets BSearch answer questions like “how is EBITDA calculated in the Acme model?”

Tables: The Hard Problem

Tables are kept intact as one chunk if they fit within 1,500 tokens. Each table chunk gets a descriptive prefix: [Sheet: Revenue Model] Table with columns: Year, Revenue, EBITDA — 15 rows. This prefix gives the search system context about what the table contains even before reading the data.

But many IB tables — income statements, cap tables, sources-and-uses — are much larger than 1,500 tokens. These need to be split. And this is where it gets interesting.

4.4 Hierarchical Table Splitting

Splitting a large financial table is not like splitting a paragraph. You can’t just cut at row 25 and call it a day. If you split an income statement between “Depreciation” and “Total Operating Expenses,” neither chunk makes sense on its own. The top half has line items without their subtotal; the bottom half has a subtotal with no context for what it totals.

IB Analogy

Think of how you’d break up a long financial model for a management presentation. You wouldn’t split the income statement at an arbitrary row number. You’d split it into logical sections — Revenue, Cost of Goods Sold, Operating Expenses, EBITDA — each with its own subtotal. Each section tells a complete story on its own, and you can understand any one section without seeing the others.

BSearch does exactly this. For tables exceeding 1,500 tokens, the system uses structural signals already present in the spreadsheet to find intelligent split points. It doesn’t need to understand financial semantics — it reads formatting cues that Excel and Word already encode.

Step 1: Detect Column Headers

The first one or two rows of a table are identified as column headers. The heuristic: a row where all non-empty cells are bold, or where all cells are strings with no numeric values (think “FY2022A, FY2023A, FY2024E”). These headers are stored separately — they get prepended to every chunk so each piece of the table is self-contained.

Step 2: Detect Category Boundaries

Below the headers, the system scans data rows top-to-bottom looking for structural signals that mark where one logical section ends and another begins:

Signal How It’s Detected Example
Indent decrease A row at indent level 0 after rows at indent level 1 or 2 “Operating Expenses” at indent 0 after “Raw Materials” at indent 2
Bold first cell First non-empty cell is bold and unindented Revenue by Segment (bold header row)
Total/Subtotal row First cell text starts with “Total,” “Subtotal,” “Net,” or “Grand Total” “Total Operating Expenses”
Blank separator All cells in the row are empty or whitespace Blank row between “Sources” and “Uses” sections

For PDF and Word tables (where indent data isn’t available), the system relies on bold first-cell and blank-row signals only.

Step 3: Build Row Groups

Using the detected boundaries, rows are grouped into logical sections. Each group starts at a category boundary and runs until the next one. The result is a list of row groups that correspond to the table’s natural structure.

Step 4: Assemble Chunks

Adjacent row groups are merged until adding the next group would exceed the 1,500-token limit. Each chunk gets:

  • Column headers (always prepended — so every chunk is readable standalone)
  • Category breadcrumb for nested groups: [Operating Expenses > SG&A] when the group’s parent category is known from indent hierarchy
  • Descriptive prefix: [Sheet: Income Statement] Rows 15–42 of 95 (Operating Expenses) — Columns: FY2022A, FY2023A, FY2024E

What This Looks Like in Practice

Table Type Structure Detected How It Splits
Income statement Bold section headers + indented line items + “Total” rows Revenue section, COGS section, OpEx section, each with subtotals
Sources & Uses Blank row separator between “Sources” and “Uses” One chunk per section
Cap table Investor names at indent 0, share classes indented One group per investor
Flat data table No structure detected (no bold, no indent variation) Fixed groups of ~25–30 rows
Edge Cases

Deeply nested tables (3+ indent levels) are grouped at the top-level category only — sub-hierarchy stays intact within the chunk. Tables where every row is bold (common in summary tables) are treated as flat and use the fixed-size fallback. Merged cells are resolved to the parent cell’s value.

4.5 Document Summaries: Bridging Distant Sections

Long documents create many chunks, and sometimes the answer to a query spans information from distant sections. A company description on page 5 of a CIM references financial metrics on page 30. Without some way to connect them, these chunks are isolated — a query like “Acme’s revenue growth story” will surface either the narrative or the numbers, but not both.

Document summary: A ~200-token condensed representation of an entire document, generated by Claude Haiku at ingestion time. Indexed as a special chunk alongside the document’s regular chunks, it acts as a semantic bridge connecting distant sections.

For any document that produces 4 or more chunks, Haiku generates a summary capturing the document type, deal name, key topics covered, and key entities and metrics mentioned. This summary is stored as an additional chunk in the search index with a special marker (chunk_type: doc_summary).

IB Analogy

This is like the executive summary at the top of a CIM. It gives you the overview — what the company does, the key financials, why it’s for sale — even if you haven’t read every section. If someone asks you a question about the deal, the exec summary often has enough context to point you toward the right section, even if it doesn’t contain the detailed answer itself.

How It Helps Retrieval

The summary chunk acts as a semantic bridge. When a user asks “tell me about Acme’s revenue growth,” the summary mentions both “company overview” and “revenue projections” and “EBITDA growth,” so it matches the query via vector search. If the summary lands in the top results alongside specific data chunks, Claude gets a document-level map that helps it synthesize across the specific chunks it also received.

4.6 Deal Summaries: Cross-Document Intelligence

Document summaries bridge sections within a single document. But some questions require reasoning across multiple documents for the same deal — the CIM, the management presentation, the bid letters, the NDA. Deal summaries provide that cross-document bridge.

Deal summary: A ~400-token overview of an entire deal, generated by aggregating information from all documents in that deal’s folder. Covers deal status, key metrics, involved parties, and stage progression.

For deals with 3 or more documents, Haiku generates a deal-level summary. The input is not the raw documents themselves — it’s the individual document summaries plus entity data from the knowledge graph (deal status, target company, financials, advisors, acquirers, key contacts, stage progression). This means the deal summary is enriched with structured intelligence, not just free text.

IB Analogy

This is like the deal team’s internal status memo — the one-pager that summarizes everything known about a deal so far. Who’s the target, who’s the buyer, what stage are we at, what are the key metrics, who are the advisors. A new team member can read it and immediately understand the deal without digging through every document in the folder.

When Deal Summaries Get Refreshed

Deal summaries are not static. They stay current through three triggers:

  • After bulk ingestion: Generated for all deals meeting the threshold
  • After file changes: When a file is added or updated in a deal folder, the deal summary is queued for regeneration (debounced by 5 minutes to batch rapid multi-file updates)
  • Periodic refresh: Every 6 hours, any deal summary where a constituent document has been modified since the summary was last generated gets refreshed

How It Helps at Query Time

When a user asks “summarize our healthcare pipeline,” the deal summary chunks for each healthcare deal match the query via vector search. If three deal summaries land in the top results, Claude gets pre-digested deal cards instead of raw heterogeneous chunks — the cross-deal reasoning is largely already done. Claude can focus on synthesizing across deals rather than extracting basic facts from individual document chunks.

4.7 Near-Duplicate Detection

If you’ve ever looked inside a deal folder, you know the problem:

Deal Summary v3 FINAL.docx
Deal Summary v3 FINAL (2).docx
Deal Summary v3 FINAL_JK edits.docx
Deal Summary v3 FINAL FINAL.docx
Copy of Deal Summary v3 FINAL.docx

Without deduplication, a search for this deal’s summary would return five near-identical results, burying other relevant documents. BSearch catches these with a two-phase approach.

IB Analogy

This is like knowing that “Acme CIM v3 FINAL (2).docx” and “Acme CIM.docx” are the same document. You always want to look at the latest version. You don’t want five copies cluttering your search results when you just need the current one.

Phase 1: Filename Normalization

Strip away the noise that humans add to filenames:

  • Version markers: v3, v2.1, version 4
  • Finality markers: FINAL, FINAL FINAL, revised, draft, clean, markup, redline
  • Copy markers: (2), Copy of, - copy
  • Editor initials: _JK edits

After normalization, files are grouped by their cleaned name within the same folder. Files that normalize to the same name in the same folder are candidate duplicates.

The Same-Folder Constraint

Only files in the same folder are compared. This prevents false positives between intentionally different documents that happen to share a name — a buyer’s “Term Sheet.docx” and a seller’s “Term Sheet.docx” are not duplicates. Neither are a template in a Templates folder and a filled version in an Active Deals folder.

Phase 2: Embedding Similarity

Filename normalization alone isn’t enough. Sometimes files with similar names have genuinely different content — a v1 and v3 that diverged significantly. So BSearch compares the first-chunk embeddings of candidate duplicates using cosine similarity (the same mathematical measure of meaning-similarity described in Chapter 3).

Cosine similarity threshold (0.95): Two chunks scoring 0.95 or above are nearly identical in meaning. For context: two unrelated documents about the same company might score 0.70–0.80. Two versions of the same document with minor edits typically score 0.96–0.99. The 0.95 threshold sits right at the boundary, catching true duplicates while preserving meaningfully different versions.

If the cosine similarity is ≥ 0.95, the files are confirmed duplicates. No additional API calls are needed — these embeddings were already computed during the embedding step.

Canonical Selection

The most recently modified file becomes the canonical version (on tie: the largest file by byte count, since a more complete version is likely larger). All other copies remain in the index but are marked non-canonical and excluded from search results. They’re tracked so they can be promoted back to canonical if the current canonical version is deleted.

At query time, a simple filter (is_canonical eq true) excludes non-canonical chunks. This adds zero measurable latency to searches.

4.8 Metadata Extraction

Once BSearch has the document’s text, it needs to understand what kind of document this is and which deal it belongs to. Two key metadata fields are extracted for every document.

Document Type Classification

BSearch classifies each document into one of 15 types: CIM, teaser, financial model, NDA, engagement letter, management presentation, bid letter, process letter, board materials, and so on. The approach is two-tiered:

  1. Pattern matching first: Filename and folder path patterns handle the obvious cases. A file named “Project Atlas - CIM vFinal.pdf” in a “Marketing Materials” folder is clearly a CIM. This is fast and costs nothing.
  2. LLM fallback for ambiguous cases: When pattern matching isn’t confident — say, a file named “Q3 Update.docx” that could be a management presentation, a board deck, or a status memo — BSearch sends the first ~500 tokens to Claude Haiku for classification.
Claude Haiku: A small, fast, inexpensive language model optimized for quick classification and extraction tasks. BSearch uses it throughout the ingestion pipeline for tasks that need language understanding but don’t require the full reasoning power of larger models.

Deal Identification

deal_id: A unique identifier for each deal, computed as a hash of the subfolder path within the firm’s deal root folders. The key insight: deal folder paths are stable identifiers even when deal names change. A deal might start as “Project Atlas” in early screening and become “Acme Corp Sale” once the engagement is signed, but the SharePoint folder path doesn’t change. Hashing that path gives a stable identifier that persists throughout the deal lifecycle.

4.9 Entity Extraction: Three Tiers of Intelligence

Not every document deserves the same level of entity extraction effort. A 200-page CIM has dense entity information worth deep analysis; a one-page NDA cover letter does not. BSearch uses a three-tier extraction strategy that balances thoroughness against cost and speed.

flowchart TD
    DOC["Document Parsed"] --> T1["Tier 1: Document-Level LLM\n(Every document)"]
    T1 --> T1OUT["Companies + roles, industry,\ndeal stage inference"]
    T1 --> CHECK{"High-value\nchunks?"}
    CHECK -- Yes --> T2["Tier 2: Chunk-Level LLM\n(Selected chunks only)"]
    CHECK -- No --> T3["Tier 3: Local NER + Regex\n(All remaining chunks)"]
    T2 --> T2OUT["Additional entities,\nrelationships, metrics"]
    T3 --> T3OUT["Known company names,\ndollar amounts, dates"]
    T2 --> MERGE["Merge & Deduplicate"]
    T3 --> MERGE
    T1OUT --> MERGE
    MERGE --> NEO4J["Write to Neo4j"]

    style T2 fill:#dbeafe,stroke:#4A7FB5,color:#3B2F2F
    style T3 fill:#d4edda,stroke:#2E7D5B,color:#3B2F2F

Tier 1: Document-Level LLM Extraction

Runs on every document. One Haiku call per document, using about 1,000 tokens of input. Extracts:

  • Companies with roles: target, acquirer, potential acquirer, financial advisor, legal advisor, lender — with confidence levels
  • Industry classification: what sector the deal falls in
  • Deal stage inference: derived from the document type (an NDA suggests early stage; a purchase agreement suggests closing)

This provides the baseline entity data for every document. The results feed into the Neo4j knowledge graph and enable queries like “show me all deals where Goldman was advisor.”

Tier 2: High-Value Chunk Extraction

Runs selectively. After chunking, Haiku runs additional extraction on specific chunk types only:

  • Table chunks with financial headers (“Revenue,” “EBITDA,” “Enterprise Value”) — extracts financial metrics with values, periods, and currency
  • Structured people sections (management team bios, signature blocks, engagement letter parties) — extracts person names, titles, and company affiliations
  • IOI/LOI body chunks — extracts offer terms and acquirer details
  • Process letter chunks — extracts potential acquirer lists

This adds roughly 1–3 additional Haiku calls per document. Many documents (memos, board decks) get zero additional calls because they lack high-value chunk types.

Tier 3: Local NER and Regex

NER (Named Entity Recognition): A technique for finding known types of things in text — company names, dollar amounts, dates, people — using pattern rules rather than an AI model. Fast and free, but only catches patterns it’s been told to look for.

Runs on every remaining chunk, with zero API calls. Simple pattern matching catches:

  • Company names already known from Tier 1 (matched against a cached list from Neo4j)
  • Dollar amounts, percentages, and dates in standard formats
  • Person names in structured contexts

Tier 3 doesn’t classify roles or relationships — it only detects mentions. But those mentions are valuable for search relevance (“which chunks mention Goldman?”).

Merge and Deduplicate

All three tiers feed into a merge step before writing to the knowledge graph. This ensures that “Goldman Sachs,” “Goldman Sachs & Co.,” and “GS” resolve to the same entity. Company names are normalized by stripping common suffixes (Inc., Corp., LLC, Ltd., & Co., Group, Holdings) and matching on the cleaned form.

What Goes to Neo4j (Structured Intelligence)

  • Companies with roles (Tier 1 + 2)
  • People with titles (Tier 2)
  • Financial metrics (Tier 2)
  • Deal stage and status
  • Industry classification

What Goes to Search Index (Per-Chunk)

  • Company mentions in this chunk (Tier 3)
  • Person mentions in this chunk (Tier 3)
  • Company roles from Tier 1 (copied to all chunks in the document)
  • Deal name, aliases, status, industry (from Neo4j, same for all chunks in the deal)

4.10 Indexing: Writing to Two Systems

The final step pushes all processed data into the two storage systems that power search. The write order matters:

  1. Neo4j first — Entity data (companies, people, deals, relationships) is written to the knowledge graph. Neo4j is the source of truth for entity information.
  2. Azure AI Search second — Chunks with all their metadata, entity fields, and embedding vectors are pushed to the search index. The chunks include denormalized (copied) entity data from Neo4j so that search queries don’t need to hit the graph at query time.
Denormalized: Storing a copy of data in multiple places for faster reads, even though it means the data could get out of sync. Like putting a deal summary in both the deal folder and the weekly pipeline report — it’s duplicated, but the report is faster to read than digging through the folder.

This write order ensures that if the pipeline crashes mid-indexing, the knowledge graph has the entity data and the search index can be rebuilt from it. The reverse order would risk orphaned search entries with no entity backing.

Key Takeaway

The ingestion pipeline is a series of deliberate trade-offs. Format-aware parsing preserves document structure at the cost of format-specific code. Hierarchical table splitting preserves financial logic at the cost of structural detection heuristics. Three-tier entity extraction balances thoroughness against API costs. Near-duplicate detection prevents clutter without risking false positives via the same-folder constraint. At every step, the goal is the same: transform raw SharePoint files into index entries that are small enough to be relevant, rich enough to be useful, and clean enough to not waste the searcher’s time.
