Chapter 4

The Ingestion Pipeline

How documents go from SharePoint files to searchable, AI-ready index entries.

Before anyone can search for a document, BSearch has to process it. Every file in SharePoint — every CIM, teaser, model, NDA, and engagement letter — must be discovered, downloaded, parsed, classified, sliced into searchable pieces, converted into mathematical representations, and loaded into the search index. This chapter walks through that nine-step journey in detail.

IB Analogy

Think of the ingestion pipeline like the onboarding process for a new deal. A pitch deck doesn’t just appear in your working folder ready to reference — someone has to receive it, file it in the right deal folder, tag it with deal metadata, extract key terms, and make sure it’s accessible to the right people. The ingestion pipeline automates all of that for every document across the entire firm.

4.1 The Pipeline at a Glance

Here is the full pipeline. A document enters at the top and exits at the bottom as a set of searchable, AI-ready index entries.

flowchart TD
    A["1. Discovery\nDetect new/changed files"] --> B["2. Download\nFetch file content from Graph API"]
    B --> C["3. Parsing\nExtract text, tables, structure"]
    C --> D["4. Metadata Extraction\nClassify doc type, assign deal, extract entities"]
    D --> E["5. Chunking\nSplit into index-ready pieces"]
    E --> F["6. Summary Generation\nCreate doc + deal summaries"]
    F --> G["7. Embedding\nConvert chunks to vectors"]
    G --> H["8. Deduplication\nIdentify and exclude duplicates"]
    H --> I["9. Indexing\nPush to Neo4j + Azure AI Search"]

Each step has a specific job, and each feeds its output into the next. If any single step fails for a document, that document gets queued for retry rather than blocking the rest of the pipeline. Let’s walk through each one.

4.2 Step 1 — Discovery

The pipeline starts with a simple question: what changed? BSearch needs to know when someone creates, edits, renames, moves, or deletes a file in SharePoint. It uses two complementary mechanisms to stay current.

Webhooks (Primary)

Webhook: A notification system where SharePoint proactively tells BSearch “something changed” rather than BSearch having to ask. Like getting a push notification on your phone vs. manually checking your email.

BSearch registers webhook subscriptions with the Microsoft Graph API for each monitored SharePoint site. When a file changes, Microsoft sends a notification to BSearch within 30–120 seconds. The notification itself is lightweight — it just says “something changed in this folder” — and BSearch then queries for the specific changes.

Webhook subscriptions have a maximum lifetime of about 2.95 days (a Microsoft limitation). BSearch stores subscription records in Cosmos DB and automatically renews them every 12 hours, well before they expire. If a renewal fails, the next scheduled renewal attempt picks it up.

Delta Sync (Fallback)

Delta sync: A periodic “catch-up” mechanism where BSearch asks the Graph API “what changed since the last time I asked?” using a delta token — a bookmark that marks the last known state.

Every 12–24 hours, regardless of webhooks, BSearch runs a delta sync. This catches anything webhooks might have missed — a network blip, a missed renewal, or an edge case where Microsoft didn’t fire a notification. It’s the safety net.

IB Analogy

Webhooks are like getting a Bloomberg alert when a deal you’re tracking has news. Delta sync is like your morning routine of scanning the Journal and checking your inbox — you probably already saw the alert, but you scan anyway to make sure nothing slipped through.

4.3 Step 2 — Download

Once BSearch knows a file changed, it downloads the actual file content from SharePoint via the Graph API. This is straightforward — the Graph API provides a download URL — but the real work is handling failures gracefully:

  • 404 errors (file was deleted between notification and download) — skip and mark as deleted
  • Auth errors (expired tokens) — refresh credentials and retry
  • Timeouts (large files, slow network) — retry with exponential backoff

The downloaded content is held in memory for the next steps. BSearch doesn’t store raw file copies — SharePoint remains the single source of truth for file content.

4.4 Step 3 — Parsing

A Word document is not plain text. It’s a compressed archive of XML files defining paragraphs, headings, tables, styles, fonts, and embedded objects. An Excel file is even more complex — multiple sheets, formulas, named ranges, merged cells. The parsing step extracts structured content from each format.

Format Parser What Gets Extracted
Word (.docx) python-docx Paragraphs, headings (with hierarchy), tables, styles
Excel (.xlsx) openpyxl Sheet names, column headers, cell values, formula text (sheet by sheet)
PowerPoint (.pptx) python-pptx Slide title, body text, tables, speaker notes (slide by slide)
PDF Azure Document Intelligence Text, tables, layout structure (handles scanned docs, complex layouts, watermarks)

The first three parsers are Python libraries that read the Office XML formats directly. They’re fast, free, and reliable for well-formed documents.

PDF parsing is a different story. PDFs in investment banking are notoriously difficult — scanned CIM pages, multi-column layouts, embedded financial tables, watermarked confidential pages. Simple PDF-to-text libraries butcher these. Azure Document Intelligence is a cloud AI service specifically built for document understanding. It handles the complex layouts that other parsers cannot.

Why Not Use Azure Document Intelligence for Everything?

It’s overkill for Word and Excel files, which have perfectly structured XML internals. Using it for everything would be slower (cloud API round-trip vs. local parsing) and more expensive (per-page pricing). We use the right tool for each job.

4.5 Step 4 — Metadata Extraction

Once we have the document’s text, we need to understand what kind of document this is, which deal it belongs to, and which companies and people are mentioned. Three fields are extracted for every document.

Document Type Classification

BSearch classifies each document into one of 15 types: CIM, teaser, financial model, NDA, engagement letter, management presentation, bid letter, process letter, board materials, and so on. The approach is two-tiered:

  1. Pattern matching first: Filename and folder path patterns handle the obvious cases. A file named Project Atlas - CIM vFinal.pdf in a Marketing Materials folder is clearly a CIM. This is fast and costs nothing.
  2. LLM fallback for ambiguous cases: When pattern matching isn’t confident — say, a file named Q3 Update.docx that could be a management presentation, a board deck, or a status memo — BSearch sends the first ~500 tokens to Claude Haiku for classification. Haiku is a small, fast, inexpensive language model optimized for quick classification tasks.

Deal Identification

deal_id: A unique identifier for each deal, computed as the SHA-256 hash of the subfolder path within one of the firm’s deal root folders (“Banking - General”, “Private Equity - General”).

The key insight: deal folder paths are stable identifiers even when deal names change. A deal might start as “Project Atlas” in early screening and become “Acme Corp Sale” once the engagement is signed, but the SharePoint folder path doesn’t change. Hashing that path gives us a stable deal_id that persists throughout the deal lifecycle.

Entity Extraction (Tier 1)

For each document, Claude Haiku extracts structured entity information:

  • Companies with roles: target, acquirer, potential acquirer, financial advisor, legal advisor, lender
  • Industry classification: what sector the deal falls in
  • Deal stage inference: derived from the document type (an NDA suggests early stage; a purchase agreement suggests closing)

This entity data feeds into the knowledge graph (covered later in this chapter) and enables queries like “show me all deals where Goldman was advisor.”

4.6 Step 5 — Chunking

Chunking: Breaking a document into smaller pieces (“chunks”) sized for the search index. A 100-page CIM can’t be searched as one unit — it needs to be split into pieces that are small enough to be relevant but large enough to be meaningful.

Different content types need different chunking strategies. A paragraph of narrative text behaves very differently from a financial table or a slide deck.

Narrative Content (Word, PDF)

Target size: 800–1,000 tokens per chunk. Splits happen at natural boundaries — paragraph breaks, section headings — not mid-sentence. Each chunk is prepended with its section heading so it retains context when searched in isolation. For example, a chunk from a CIM’s “Industry Overview” section carries that heading, so the search system knows it’s about industry context even if the chunk text itself doesn’t say so.

Tables (Excel)

Tables are kept intact as one chunk if they fit within 1,500 tokens. If a table is larger, BSearch uses the table’s own structure to find smart split points:

  • Hierarchical tables (the kind you see in CIM financials, with bold section headers, indented line items, and “Total” rows): split at the section-header level, keeping each group with its subtotals
  • Flat tables (uniform rows with no hierarchy): split into fixed groups of approximately 25 rows

Slides (PowerPoint)

One chunk per slide, prefixed with [Slide N: Title]. Slides are already natural units of meaning — each one typically covers a single topic or data point.

Formulas (Excel)

Excel formula text (the formulas themselves, not their computed values) gets a separate chunk per key sheet. This lets BSearch answer questions like “how is EBITDA calculated in the Acme model?”

IB Analogy

Chunking is like how you break up a CIM for a buyer’s internal review. You wouldn’t hand the CFO the entire 200-page document — you’d split it into the exec summary, the financial section, the industry overview, and each major exhibit. Each piece is self-contained enough to be useful on its own, but together they form the complete picture.

4.7 Step 6 — Summary Generation

Long documents create many chunks, and sometimes the answer to a query spans information from distant sections. Summaries act as semantic bridges — condensed representations that connect the dots.

Document Summaries

For documents with 4 or more chunks, Claude Haiku generates a ~200-token summary capturing key topics, financial metrics, and entities. This summary is stored as an additional chunk in the search index. When someone searches for “Acme Corp acquisition terms,” the summary chunk may surface even if no single text chunk contains all the relevant details.

Deal Summaries

For deals with 3 or more documents, Haiku generates a ~400-token deal overview covering deal status, key metrics, involved parties, and stage. These summaries are enriched with entity data from the Neo4j knowledge graph and refreshed every 6 hours to stay current as new documents are added to a deal folder.

4.8 Step 7 — Embedding

Embedding: Converting text into a list of numbers (a “vector”) that captures its meaning. Similar texts produce similar vectors, enabling the search system to find documents that are conceptually related to a query, not just keyword matches. (Covered in depth in Chapter 3.)

BSearch uses Voyage AI’s voyage-context-3 model, which produces a 1,024-dimensional vector for each chunk. The key feature of this model is that it’s contextual: each chunk is embedded with awareness of the full document text, not in isolation. This means a chunk that says “Revenue grew 15% year-over-year” gets a different vector depending on whether it’s from a healthcare CIM or an energy sector teaser — because the model sees the surrounding context.

Chunks are processed in batches for efficiency. Rather than making one API call per chunk, BSearch groups chunks and sends them together, significantly reducing latency and cost.

4.9 Step 8 — Deduplication

If you’ve ever looked inside a deal folder, you know the problem:

Deal Summary v3 FINAL.docx
Deal Summary v3 FINAL (2).docx
Deal Summary v3 FINAL_JK edits.docx
Deal Summary v3 FINAL FINAL.docx
Copy of Deal Summary v3 FINAL.docx

Without deduplication, a search for this deal’s summary would return five near-identical results, burying other relevant documents. BSearch catches these with a two-phase approach:

Phase 1: Filename Normalization

Strip version markers (v3, v2.1), finality markers (FINAL, FINAL FINAL), copy markers ((2), Copy of), and editor initials (_JK edits). Group files by their normalized name and source folder. Files that normalize to the same name in the same folder are candidate duplicates.

Phase 2: Embedding Similarity

Filename normalization alone isn’t enough — sometimes files with similar names have genuinely different content (e.g., a v1 and v3 that diverged significantly). So BSearch compares the first-chunk embeddings of candidate duplicates. If the cosine similarity is ≥ 0.95 (nearly identical), they’re confirmed duplicates.

The most recently modified file becomes the canonical version. All other copies are excluded from search results but tracked in the index so they can be restored if the canonical version is deleted.

Why 0.95?

A cosine similarity of 0.95 is extremely high — it means the two chunks are nearly identical in meaning. Two unrelated documents about the same company might score 0.7–0.8. Two versions of the same document with minor edits typically score 0.96–0.99. The 0.95 threshold sits right at the boundary, catching true duplicates while preserving meaningfully different versions.

4.10 Step 9 — Indexing

The final step pushes all processed data into the storage systems that power search. The write order matters:

  1. Neo4j first — Entity data (companies, people, deals, relationships) is written to the knowledge graph. Neo4j is the source of truth for entity information.
  2. Azure AI Search second — Chunks with all their metadata, entity fields, and embedding vectors are pushed to the search index. The chunks include denormalized (copied) entity data from Neo4j so that search queries don’t need to hit the graph at query time.
Denormalized: Storing a copy of data in multiple places for faster reads, even though it means the data could get out of sync. Like putting a deal summary in both the deal folder and the weekly pipeline report — it’s duplicated, but the report is faster to read than digging through the folder.

This write order ensures that if the pipeline crashes mid-indexing, the knowledge graph has the entity data and the search index can be rebuilt from it. The reverse order would risk orphaned search entries with no entity backing.

4.11 The Knowledge Graph

Steps 1–9 describe the linear pipeline. But there’s a parallel data structure being built alongside it: the knowledge graph in Neo4j. While the search index stores document chunks, the knowledge graph stores the relationships between entities that those documents mention.

Knowledge graph: A database that stores things (nodes) and the connections between them (relationships). Unlike a traditional database table with rows and columns, a knowledge graph naturally represents networks of relationships — exactly what you need for “which companies has Goldman advised on deals where JP Morgan was the lender?”

What Gets Stored

Neo4j holds four types of nodes:

Node Type What It Represents Example
Deal An engagement or potential transaction Project Atlas / Acme Corp Sale
Company A firm involved in a deal Acme Corp, Goldman Sachs, First Republic
Person An individual mentioned in documents John Smith (CEO of Acme Corp)
StageEntry A timestamped record of a deal reaching a stage “Project Atlas reached LOI stage on 2024-03-15”

These nodes are connected by typed relationships that capture the role each entity plays:

  • TARGET_OF — Acme Corp is the target of the deal
  • POTENTIAL_ACQUIRER — BuyerCo is a potential acquirer
  • ACQUIRER_OF — BuyerCo acquired Acme Corp (deal closed)
  • ADVISOR_ON — Goldman Sachs is an advisor on the deal
  • LENDER_ON — First Republic is a lender
  • EXECUTIVE_AT — John Smith is an executive at Acme Corp
  • APPEARED_IN — links entities to specific documents
  • REACHED_STAGE — links a deal to its stage progression

Deal Alias Accumulation

Early in a deal lifecycle, documents may use a code name (“Project Atlas”). Later documents use the real company name (“Acme Corp”). Some documents use both. The knowledge graph handles this by accumulating aliases:

  • Every name variation encountered for a deal is stored as an alias
  • The canonical name is always the name from the most recently modified file
  • Search queries match against all aliases, so searching “Project Atlas” or “Acme Corp” finds the same deal
IB Analogy

This is exactly how you mentally track deals. Early on it’s “Project Atlas” in every email. Then the engagement letter says “Acme Corp.” Six months later, someone asks about “that Acme deal” and you know they mean Project Atlas. The knowledge graph does the same mapping automatically.

4.12 Entity Extraction: The Three Tiers

Not every document deserves the same level of entity extraction effort. A 200-page CIM has dense entity information worth deep analysis; a one-page NDA cover letter does not. BSearch uses a three-tier extraction strategy that balances thoroughness against cost and speed.

flowchart TD
    DOC["Document Parsed"] --> T1["Tier 1: Document-Level LLM\n(Every document)"]
    T1 --> T1OUT["Companies + roles, industry,\ndeal stage inference"]
    T1 --> CHECK{"High-value\nchunks?"}
    CHECK -- Yes --> T2["Tier 2: Chunk-Level LLM\n(Selected chunks only)"]
    CHECK -- No --> T3["Tier 3: Local NER + Regex\n(All remaining chunks)"]
    T2 --> T2OUT["Additional entities,\nrelationships, metrics"]
    T3 --> T3OUT["Known company names,\ndollar amounts, dates"]
    T2 --> MERGE["Merge & Deduplicate"]
    T3 --> MERGE
    T1OUT --> MERGE
    MERGE --> NEO4J["Write to Neo4j"]

    style T1 fill:#f5e6d3,stroke:#D4793A,color:#3B2F2F
    style T2 fill:#dbeafe,stroke:#4A7FB5,color:#3B2F2F
    style T3 fill:#d4edda,stroke:#2E7D5B,color:#3B2F2F
Tier Method Applied To What It Finds Cost
Tier 1 Claude Haiku (document-level) Every document Companies with roles, industry, deal stage Low (one LLM call per doc)
Tier 2 Claude Haiku (chunk-level) High-value chunks only (exec summaries, transaction overviews, key terms sections) Additional entities, relationships, specific metrics Medium (selective LLM calls)
Tier 3 Local NER + regex All remaining chunks Known company names, dollar amounts, dates Zero (no API calls)
NER (Named Entity Recognition): A technique for finding known types of things in text — company names, dollar amounts, dates, people — using pattern rules rather than an AI model. Fast and free, but only catches patterns it’s been told to look for.

Tier 1 runs on every document and provides the baseline. Tier 2 goes deeper on the chunks most likely to contain rich entity information — an executive summary typically names every party, while page 47 of a CIM’s market analysis may not add new entities. Tier 3 sweeps everything else with simple pattern matching, catching company names already known from Tier 1 and standard formats like dollar amounts.

All three tiers feed into a merge-and-deduplicate step before writing to Neo4j, ensuring that “Goldman Sachs,” “Goldman Sachs & Co.,” and “GS” resolve to the same entity.

Key Takeaway

The ingestion pipeline transforms raw SharePoint files into two complementary data structures: a search index (Azure AI Search) containing document chunks with embeddings for fast retrieval, and a knowledge graph (Neo4j) containing the entities and relationships those documents mention. Together, these let BSearch answer both “find me the Acme CIM” and “which deals has Goldman advised on?” — questions that require fundamentally different data representations.

The pipeline processes documents incrementally — only changed files are re-ingested, not the entire corpus. For Berenson’s ~550,000 documents, a full re-index takes hours, but the steady state is handling a few dozen file changes per hour in near-real-time via webhooks.
