Chapter 1

Executive Summary

What BSearch is, the problem it solves, and why it matters for Berenson.

1.1 What Is BSearch?

BSearch is an enterprise search and AI synthesis system built for Berenson & Company. It connects to the firm's Microsoft 365 tenant — roughly 550,000 documents across SharePoint and OneDrive — and gives every user the ability to ask questions in plain English and get real answers, not just a list of file links.

Enterprise search — A system that lets people search across all of a company's internal documents, email, and files from one place, much like Google but for internal content only.

When someone types a query like "What was the EV/EBITDA for the Acme deal?", BSearch does not just find the relevant CIM or model — it reads the documents, extracts the answer, and streams back a written response with source citations. You can also ask broader questions like "What healthcare deals have we worked on in the last two years?" and get a synthesized summary drawn from multiple files.

IB Analogy

Think of BSearch as a junior analyst who has read every document in every deal folder, remembers all of it, and can answer any question about any deal in under five seconds — without ever pulling up the wrong version.

1.2 The Problem BSearch Solves

Berenson has roughly 550,000 documents spread across SharePoint and OneDrive: Word documents, Excel models, PowerPoint decks, and PDFs. These files contain the collective knowledge of the firm — deal terms, financial metrics, client communications, precedent transactions, and model outputs. But today, finding the right information is painful:

  • Wrong versions waste real time. When someone grabs an outdated model or an old draft of a CIM, the rework can cost 10 or more hours. In a firm of 18 people, that is a significant hit.
  • No precedent search. There is no practical way to ask "What comparable deals have we done?" and get a useful answer. Institutional knowledge lives in people's heads or is buried in folder structures that only make sense if you already know what you are looking for.
  • Universal pain. This is not a problem limited to one team or one role. All 18 people at the firm — from analysts to managing directors — regularly need to find documents or extract information from them.
IB Analogy

Imagine you are staffed on a new healthcare deal and need to reference the fee structure from a comparable transaction Berenson advised on two years ago. Today, you would ask around, dig through folders, and hope you find the right version. With BSearch, you type your question, and the answer appears in seconds — with a link to the source document.

1.3 The Key Insight: AI Synthesis Is the Product

BSearch is not just a better file search. The core insight behind the system is that AI synthesis is the product. Finding the right document is necessary but not sufficient — what users actually want is the answer that lives inside those documents.

Traditional File Search

You search, get a list of files, open each one, scan for the answer, and hope you picked the right version.

BSearch

You ask a question, BSearch reads the relevant documents, and streams back a written answer with citations and source links.

This distinction matters because it shapes every design decision in the system. The ingestion pipeline does not just index file names and metadata — it parses document content, understands table structures, extracts deal-level entities, and creates the kind of rich, searchable representation that an AI model needs to produce accurate answers.

AI synthesis — The process of an AI model reading multiple pieces of retrieved content and composing a coherent, cited answer. In BSearch, this is handled by Anthropic's Claude model, which reads the top search results and writes the response that the user sees.

1.4 The User Experience, End to End

From the user's perspective, BSearch is simple: you open the web app, type a question, and get an answer. But behind the scenes, two major pipelines make this possible.

The Ingestion Pipeline (runs in the background)

Before anyone can search, BSearch needs to read and index every document in the firm's SharePoint and OneDrive. This happens automatically:

  1. BSearch connects to Microsoft 365 via the Graph API (Microsoft's official interface for accessing M365 data).
  2. Each document is parsed by a format-aware parser — Word, Excel, PowerPoint, and PDF each get specialized treatment to preserve structure (tables stay as tables, slides stay as slides).
  3. The content is split into searchable chunks, converted into mathematical representations called embeddings (numbers that capture meaning), and stored in Azure AI Search.
  4. Deal-level entities (company names, deal stages, financial metrics) are extracted and stored in a knowledge graph for richer query understanding.
  5. When files change, BSearch detects the update via webhooks (real-time notifications from Microsoft) and re-indexes only what changed.

The Query Pipeline (runs when you search)

  1. You type a question in the web app.
  2. BSearch classifies your query: Are you looking for a specific file? Extracting a data point? Or asking a question that spans multiple documents?
  3. It searches the index using a hybrid approach (both keyword matching and semantic meaning).
  4. Results are re-ranked for relevance by a specialized model.
  5. The top results are sent to Claude, which reads them and streams back a written answer with citations.
IB Analogy

The ingestion pipeline is like the first week on a new deal — you read every document in the data room and build a mental model of what is where. The query pipeline is like getting a question from an MD at 11pm: you already know the material, so you can pull the answer immediately and cite your sources.

1.5 Key Constraints and Design Parameters

Every system operates within constraints. These are the ones that shape BSearch's design:

Constraint Value Why It Matters
Total users ~18 Small user base means we can optimize for quality over massive scale. Costs stay low.
Total documents ~550,000 Large enough to require proper indexing and chunking — you cannot brute-force search this.
Latency: file lookup / data extraction <3 seconds to first token If the system feels slow, nobody will use it. Speed is an adoption requirement, not a nice-to-have.
Latency: multi-doc synthesis <5 seconds to first token Cross-document reasoning takes longer, but users get a visual indicator that the system is working.
Content sources (v1) SharePoint / OneDrive only v1 focuses on files. Email (Outlook), Teams bot, and document vision are deferred to later versions.
File types Word, Excel, PowerPoint, PDF Each format gets a specialized parser to preserve structure (especially tables and slides).
Permissions model 5 top-level folders, no sub-file permissions Simplifies access control — BSearch filters results by which folders a user can access.
LLM calls per query 1 (Claude only) Single-call architecture keeps latency predictable and costs manageable.

Time to first token (TTFT) — How long the user waits before the first word of the AI response appears on screen. BSearch streams answers as they are generated, so even though the full response takes longer, the user sees progress almost immediately.

1.6 v1 Scope: What's In and What's Not

BSearch is being built incrementally. The first version (v1) focuses on the highest-value, most-tractable features:

In Scope (v1)

  • SharePoint and OneDrive file search
  • Format-aware parsing (Word, Excel, PPT, PDF)
  • Hybrid search (keyword + semantic)
  • AI-synthesized answers via Claude
  • Deal and entity extraction (knowledge graph)
  • Near-real-time sync via webhooks
  • Web app with Azure AD authentication
  • Near-duplicate file detection

Deferred (v2+)

  • Outlook email search
  • Teams bot interface
  • Document vision (images, charts within docs)
  • Model summaries (auto-generated Excel model descriptions)
  • Two-pass extraction (Haiku then Sonnet)
  • Sibling chunk retrieval
  • Reconciliation jobs

1.7 Why This Matters

BSearch exists because Berenson's collective knowledge — hundreds of thousands of documents spanning years of deals — is only valuable if people can actually access it. Today, finding the right document (or the right version of the right document) is unreliable and slow. BSearch changes that by making every document searchable and every answer extractable, in seconds.

Key Takeaway

BSearch is not just a search engine. It is an AI system that reads your firm's documents and gives you answers. The goal is not to return a list of files — it is to return the information you were actually looking for, cited back to its source, in under five seconds.
