Chapter 6

How BSearch Understands Deals & Relationships

The PostgreSQL entity store — what it is, why BSearch needs it, and what it stores.

6.1 Why a Dedicated Entity Store

A search index is good at one thing: finding documents that contain certain words or match a certain meaning. Ask it "Find documents mentioning Acme Corp" and it will deliver. But ask it "Which deals had Acme Corp as an acquirer?" and it is stuck. The search index knows that the words "Acme Corp" appear in a document — it does not know that Acme Corp was the acquirer in one deal and the target in another. That distinction lives in the relationships between things, not in the text itself.

This is the gap the entity store fills. Structured questions — "What is the deal status?", "Who is the advisor?", "Which deals have EBITDA above $50M?" — need structured data with proper relationships. The entity store provides exactly that:

  • Tables — the things themselves (deals, companies, people, stage transitions), stored as rows with typed columns.
  • Relationship rows — the connections between those things (Company X is the target of Deal Y, Person Z advised on Deal Y), stored in junction tables with foreign keys.

Entity store — A dedicated PostgreSQL database that stores structured entity data — deals, companies, people, and the relationships between them — separately from document chunks. Unlike the search index (which stores text for retrieval), the entity store is designed for structured queries: filtering by deal status, traversing relationships, and answering questions that span multiple entities.

BSearch uses a dedicated PostgreSQL database to maintain an entity store of deals, companies, people, and their interconnections. This database is the source of truth for entity intelligence — deal status, financial profiles, company roles, stage progression, and key contacts. It is what allows BSearch to answer questions that go beyond text matching and into the realm of structured reasoning.

IB Analogy

Think of the entity store as the mental model a senior banker carries around. A veteran MD does not just remember document titles — she knows that Acme Corp was the acquirer in the Widget deal, the target in the Gadget deal, and that John Smith from Evercore advised on both. She can answer "What other deals has Evercore been on the other side of?" without opening a single file. The entity store gives BSearch that same kind of connected understanding.

The entity store is not a replacement for the search index — the two work together. The search index finds relevant document chunks. The entity store provides structured context about the entities mentioned in those chunks, enables financial filtering, and powers cross-deal queries. At query time, BSearch reads from both.

6.2 Table Schema — the Deal-Centric Relational Model

The entity store is organized as a deal-centric star schema — deals sit at the center, and everything else connects to them through foreign keys. There are five core tables.

Deal-centric star schema — A relational data model where the deals table sits at the center and all other entities (companies, people, stage entries) connect to deals through foreign-key relationships in junction tables. This makes deal-oriented queries fast and natural — you can start at any deal and immediately join to its target, acquirers, advisors, stage history, and financials.

deals table

The hub entity. Every deal that BSearch has encountered gets a row in the deals table. A deal is identified by its folder path in SharePoint — all files within the same first-level subfolder of a deal root folder (such as Banking - General/Project Atlas/) belong to the same deal, regardless of how deeply nested they are.

ColumnTypeWhat It Stores
deal_idTEXT PKSHA-256 hash of first-level subfolder path — the unique deal identifier
folder_pathTEXTThe SharePoint folder that defines this deal
canonical_nameTEXTThe most recent name variant — converges to the real company name as deals progress
last_activityTIMESTAMPTZTimestamp of the most recently modified file in this deal
deal_statusTEXTactive, on_hold, dead, or closed — inferred from file activity patterns
current_stageTEXTHighest stage reached: origination → marketing → bidding → negotiation → diligence → closing
industriesJSONBArray of industry classifications extracted by Haiku
revenueNUMERICRevenue figure extracted from financial documents
ebitdaNUMERICEBITDA figure
evNUMERICEnterprise value
ev_ebitdaNUMERICEV/EBITDA multiple
offer_priceNUMERICOffer or purchase price

deal_aliases table

Every name variant ever used for a deal. Previously stored as an array property on the deal itself, aliases are now a separate table to support efficient upsert and deduplication. See Section 6.4 for how aliases accumulate.

ColumnTypeWhat It Stores
deal_idTEXT FKReferences the deals table
aliasTEXTOne name variant for the deal (code name, short name, full legal name, etc.)

UNIQUE constraint on (deal_id, alias). New aliases are added via INSERT ... ON CONFLICT DO NOTHING.

companies table

A company involved in one or more deals — as a target, acquirer, advisor, or lender. Companies are deduplicated using a normalized name: the company name is lowercased and common suffixes (Inc., Corp., LLC, Ltd., & Co., Group, Holdings) are stripped. This means "Goldman Sachs", "Goldman Sachs & Co.", and "Goldman Sachs Group, Inc." all resolve to the same row.

ColumnTypeWhat It Stores
normalized_nameTEXT PKLowercase name with suffixes stripped (Inc., Corp., LLC, etc.) — the deduplication key
canonical_nameTEXTDisplay name as extracted
aliasesJSONBArray of known name variants
industriesJSONBArray of industry classifications
first_seenTIMESTAMPTZWhen this company first appeared in a document
last_seenTIMESTAMPTZMost recent document mentioning this company

people table

A person mentioned across documents — executives, bankers, contacts. People rows are only created from structured sections of documents: signature blocks, management team sections, cover pages, and engagement letter parties. BSearch deliberately does not create people rows from narrative mentions (e.g., "as Mr. Smith noted in his remarks") because those produce too many false positives.

ColumnTypeWhat It Stores
normalized_nameTEXT PKLowercase name — the deduplication key
canonical_nameTEXTDisplay name as extracted
aliasesJSONBArray of known name variants
titlesJSONBArray of professional titles seen across documents
first_seenTIMESTAMPTZWhen this person first appeared in a document
last_seenTIMESTAMPTZMost recent document mentioning this person

stage_entries table

A timestamped record of a deal reaching a particular stage. Unlike the other tables, stage_entries rows are not standalone entities — they exist only in connection to a deals row. A single deal can have multiple stage entry rows, one for each stage transition, creating a timeline of the deal's progression.

6.3 Relationship Tables

Relationships between entities are stored in junction tables. Each relationship row has a type that describes the nature of the connection, and every relationship row carries a source_file_ids column — a JSONB array of the specific files that provided evidence for this connection. This provenance tracking is critical for cleanup when files are deleted or updated (see Section 6.8).

Relationship row — A row in a junction table that connects two entity rows. For example, a row in deal_companies with relationship_type = 'TARGET_OF' connects a company to a deal, indicating that the company is the target in that deal. The relationship type determines the semantic meaning of the connection.

deal_companies — links companies to deals with typed roles

engagementTEXTFor POTENTIAL_ACQUIRER: indicated_interest, submitted_ioi, submitted_loi, in_diligence

The relationship types in deal_companies map directly to how bankers think about a deal:

Relationship Type Junction Table What It Means Example
TARGET_OF deal_companies The company is the target (the company being sold) in this deal. Acme Corp is the target of the Project Atlas deal.
POTENTIAL_ACQUIRER deal_companies The company has expressed interest in acquiring the target. The engagement column tracks progression: indicated_interest, submitted_ioi, submitted_loi, in_diligence. Only upgrades, never downgrades. Blackstone received the teaser for Project Atlas and later submitted an IOI.
ACQUIRER_OF deal_companies The company won the deal and became the acquirer. When a deal closes, the winning company's POTENTIAL_ACQUIRER row is promoted to ACQUIRER_OF. Blackstone acquired Acme Corp in the Project Atlas deal.
ADVISOR_ON deal_companies The company is an advisor on this deal. Includes a side property indicating whether the advisory is on the sell-side or buy-side. Berenson & Company is the sell-side advisor on Project Atlas.
LENDER_ON deal_companies The company is providing financing for the deal. JPMorgan is the lead lender on the Project Atlas acquisition financing.
role deal_people The person appeared in documents related to this deal. The role column describes their involvement. John Smith (Managing Director at Evercore) appeared in the Project Atlas process letter.
title company_people The person holds (or held) a role at the company. Jane Doe is the CEO of Acme Corp.
IB Analogy

These relationship types map directly to how bankers think about a deal. Every live deal has a target, a list of potential acquirers at various stages of engagement (received the teaser? submitted an IOI?), advisors on both sides, and lenders. The entity store makes this structure explicit and queryable — so when someone asks "Which PE firms have been active acquirers across our deals?", BSearch can query the deal_companies table for POTENTIAL_ACQUIRER and ACQUIRER_OF rows across all deals and return a structured answer.

6.4 Deal Alias Accumulation

Deals almost never go by a single name. Early in a process, a deal is assigned a code name — "Project Atlas," "Project Catalyst," "Project Evergreen." As the deal progresses and confidentiality constraints relax, documents start using the real company name. By the time the deal closes, most people refer to it as "the Acme acquisition," and the code name is half-forgotten.

This creates a real search problem. A query for "Project Atlas" should find the deal even if the most recent 50 documents all call it "Acme Corp acquisition." Conversely, a query for "Acme Corp" should find the deal even if the earliest documents only use the code name.

BSearch solves this through alias accumulation. As BSearch ingests documents from a deal folder, it extracts the deal name from each document and inserts any new variant into the deal_aliases table. This happens automatically — no manual mapping, no master list, no one needs to tell the system that "Project Atlas" and "Acme Corp acquisition" are the same deal.

Here is how the process works:

  1. A document is ingested from a deal folder (e.g., Banking - General/Project Atlas/).
  2. The AI extracts a deal name from the document content (e.g., "Project Atlas" from an early teaser, or "Acme Corp" from a later CIM).
  3. BSearch checks the deal_aliases table. If the extracted name is not already present for this deal, it is inserted via INSERT INTO deal_aliases (deal_id, alias) VALUES ($1, $2) ON CONFLICT DO NOTHING.
  4. If this document is the most recently modified file in the folder, the canonical_name is updated via UPDATE deals SET canonical_name = $1 WHERE deal_id = $2 AND last_activity < $3. This means the canonical name naturally converges to the "real" company name as deals progress.
  5. All search index chunks for this deal are tagged with both the canonical_name (as deal_name) and a space-joined string of all aliases (as deal_aliases). A search for any known alias hits.

The same pattern applies to company aliases. Company rows accumulate all name variants through the same mechanism, with the normalized name (lowercase, suffix-stripped) serving as the deduplication key. A query for "GS" can expand to include "Goldman Sachs" and "Goldman Sachs & Co." because all three are aliases on the same company row.

When a deal's canonical name changes (because a newer file used a different name), the ingestion pipeline batch-updates the deal_name and deal_aliases fields on all existing chunks for that deal in the search index. This is a targeted update — only the metadata fields are patched, not the content — so it is fast and does not require re-parsing or re-embedding.

IB Analogy

Everyone on the deal team naturally knows all the code names. If you have been working on Project Atlas for three months, you know it is the Acme deal, and if someone mentions either name you know what they are talking about. But a new analyst who joins the team only hears "Acme" — they would not know to search for "Project Atlas" to find the early-stage documents. The alias accumulation system gives BSearch the same institutional memory that the deal team has, so even a brand-new user can find everything by any name.

6.5 Deal Stage Inference

Investment banking deals progress through a well-defined sequence of stages: origination, marketing, bidding, negotiation, diligence, and closing. At each stage, specific types of documents are produced. BSearch exploits this regularity — it infers what stage a deal has reached based on the types of documents found in the deal folder.

This inference is deterministic (no AI involved). Each document type maps to a specific stage:

Document Type Inferred Stage
NDA, Engagement Letter Origination
Teaser, CIM, Management Presentation, Process Letter Marketing
IOI (Indication of Interest) Bidding
LOI (Letter of Intent) Negotiation
Diligence Tracker Diligence
Purchase Agreement Closing
Model, Board Deck, Memo, Email, Other No stage entry created — these document types appear at multiple stages and do not indicate a specific transition.

When BSearch ingests a document whose type maps to a stage, it creates a stage_entries row linked to the deal via the deal_id foreign key. The row records the stage name, the timestamp (from the document's modification date), the document type that triggered it, and the source file ID.

A deal's current_stage is simply the highest stage reached, following the natural ordering: closing > diligence > negotiation > bidding > marketing > origination. Stages only move forward — if a deal has reached bidding, discovering more marketing-stage documents does not move it backward.

Because stage entries are separate rows (not just a column on the deals table), you can query stage history naturally. "When did Project Atlas move to bidding?" becomes a straightforward SQL query: find the deal, join to stage_entries, and return the row with stage = 'bidding' along with its entered timestamp.

IB Analogy

You can tell where a deal stands just by looking at what is in the data room. If you see only teasers and NDAs, the deal is in early stages. If IOIs are on file, the deal has moved to bidding. If there is a purchase agreement, someone signed. BSearch applies this same logic automatically — it reads the document types and infers the stage, just as an experienced banker would by scanning the folder structure.

6.6 Deal Status Inference

Beyond knowing what stage a deal has reached, BSearch also infers whether a deal is currently active or not. This is driven by recent document activity — specifically, how recently files in the deal folder were modified.

Deal status inference is deterministic and follows four simple rules:

Condition Status What It Means
A purchase agreement document exists in the folder closed The deal has been completed. The presence of a signed purchase agreement is definitive.
Last file activity was more than 9 months ago (no purchase agreement) dead The deal appears to have fallen through. No one has touched any file in the folder for over 9 months.
Last file activity was 3 to 9 months ago (no purchase agreement) on_hold The deal is stalled but not definitively dead. Activity has dropped off but has not been gone long enough to call it dead.
Last file activity was within the past 3 months active The deal is actively being worked on. Recent document modifications confirm ongoing activity.

Deal status is recomputed on every file ingestion for the affected deal. This means the status stays current — if a dormant deal suddenly sees new activity (someone opens the folder and starts updating models), it automatically transitions from on_hold or dead back to active.

Status inference enables powerful queries. "What deals are currently active?" filters on deal_status = 'active'. "What's our pipeline?" returns all active deals with their entity context. These queries run against PostgreSQL and return structured results — no document scanning required.

IB Analogy

Every banker has an instinct for whether a deal is live or dead. If no one has touched the data room in six months and there is no purchase agreement, the deal probably fell through. If the team is churning out updated models and term sheets, the deal is active. BSearch applies the same heuristic, using file modification timestamps as a proxy for activity. It is not perfect — a deal could be active through phone calls with no new documents — but it is right far more often than it is wrong, and it requires zero manual input.

6.7 Financial Metrics on Deals

The deals table in the entity store stores key financial metrics: revenue, EBITDA, enterprise value (EV), EV/EBITDA multiple, EV/Revenue multiple, offer price, purchase price, total debt, and equity value. These are not manually entered — they are extracted automatically from documents during ingestion.

Financial extraction happens in the Tier 2 extraction layer, which processes high-value chunks: table chunks with financial headers (columns labeled "Revenue," "EBITDA," "Enterprise Value"), IOI and LOI body sections, and model output tabs. The AI identifies the metric, its value, the reporting period, and a confidence level. Only high-confidence extractions are written to the deals row — medium-confidence values are logged but not stored, preventing noisy or speculative numbers from polluting the entity store.

Tier 2 extraction — A selective, targeted AI extraction step that runs only on high-value chunks (financial tables, IOI/LOI body sections, structured people sections). Unlike Tier 1, which runs on every document, Tier 2 fires only on chunks that are likely to contain deal-specific financial metrics or key personnel. A typical document triggers 0 to 3 Tier 2 calls.

Financial metrics are stored as flat numeric columns on the deals table. This is a deliberate design choice — it enables direct range queries like "find all deals with EBITDA over $50M," which are fast and simple with standard SQL WHERE clauses. Alongside each metric, the financials_meta JSONB column stores provenance: the reporting period, the confidence level, and the source file that provided the number.

How Updates Work

Deals evolve, and financial figures change. An early-stage teaser might quote preliminary revenue numbers, while a later-stage model has updated figures. BSearch handles this with a straightforward update rule:

  • Higher confidence replaces lower. A high-confidence extraction from a financial summary table overwrites a medium-confidence extraction from a narrative mention.
  • Same confidence: more recent source wins. If both extractions have the same confidence, the one from the more recently modified document takes precedence.

This means the deal row's financial metrics naturally converge on the most reliable, most current numbers available — without anyone manually updating a spreadsheet.

Financial Filtering at Query Time

Storing financial metrics on the deals table enables a powerful query pattern: "Show me deals with EBITDA over $50M." When BSearch receives a query like this, it parses the financial constraint, queries PostgreSQL for matching deal IDs, and then passes those IDs as a filter to the search index. The search index returns only document chunks from those deals, and the AI synthesizes the answer from the filtered results.

If the financial filter is too restrictive (fewer than 3 results), BSearch falls back: it retries the search without the financial filter and adds a note to the AI's context explaining that not all results may match the stated financial criteria. This graceful degradation prevents empty results while maintaining transparency.

IB Analogy

Think of this as a live deal sheet that updates itself. Every time a new model, term sheet, or CIM comes in with updated financial figures, the deal sheet reflects the latest numbers — no one has to remember to update it. And when a senior banker asks "Pull up all our deals with enterprise value over $200M," the answer is immediate because the numbers are already structured and queryable, not buried inside individual documents.

6.8 Entity Cleanup

Documents change. Files are deleted, updated, or replaced. When that happens, the entity store needs to reflect the change — otherwise, it would accumulate stale relationships and phantom rows that no longer have supporting evidence.

BSearch handles this through provenance-based cleanup. Every relationship row in the entity store carries a source_file_ids JSONB array — a list of every file that provided evidence for that relationship. When a file is deleted or updated, the cleanup process is straightforward:

  1. Remove the file's ID from source_file_ids on all relationship rows that reference it: UPDATE deal_companies SET source_file_ids = source_file_ids - $file_id::text WHERE source_file_ids ? $file_id (and likewise for deal_people and company_people).
  2. Delete relationship rows with no remaining evidence. If a row's source_file_ids is now empty: DELETE FROM deal_companies WHERE source_file_ids = '[]'::jsonb.
  3. Delete stage_entries rows that were sourced from the deleted file: DELETE FROM stage_entries WHERE source_file_id = $file_id.
  4. Recompute deal status for the affected deal, since the stage history may have changed.

For file updates (a document is modified and re-ingested), the process is: cleanup first, then normal ingestion with the new content. This is an atomic delete-then-rewrite that ensures the entity store always reflects the current state of the documents.

Orphan Row Handling

When relationship rows are deleted, company and people rows may become orphans — rows with no remaining relationships. BSearch does not delete these immediately. A company row that lost its last relationship might gain new relationships from a future document ingestion. Instead, a weekly cleanup job removes orphan rows that have had no remaining relationships and a last_seen timestamp older than 90 days. This strikes a balance between keeping the entity store clean and avoiding premature deletion.

Dual-Write Consistency

BSearch stores entity data in two places: PostgreSQL (the source of truth for entities and relationships) and Azure AI Search (denormalized copies of entity fields on each document chunk, used for search-time filtering). Keeping these two systems consistent is important.

The write order is always PostgreSQL first, then Azure AI Search. If something goes wrong:

Failure Scenario What Happens
PostgreSQL write fails Chunks are written to the search index without entity fields (deal status, industries, etc.). The file is queued for retry. On retry, PostgreSQL is written and the search index chunks are patched with the correct entity fields. Search still works in the interim — it just lacks entity filtering for those chunks.
Azure AI Search write fails PostgreSQL has the entity data. The file is queued for retry. On retry, chunks are written to the search index with the correct entity fields from PostgreSQL. No data is lost.

This approach ensures that a failure in one system does not corrupt the other. The retry queue (stored in Cosmos DB with exponential backoff) ensures that temporary outages self-heal. PostgreSQL remains the authoritative source of entity truth, and the search index is always eventually consistent with it.

IB Analogy

Think of entity cleanup like maintaining a deal log. When a potential acquirer drops out and you remove their IOI from the data room, you also update the buyer list. If someone later deletes the entire deal folder, you remove the deal from the pipeline tracker. The entity store does this automatically — every deletion or update in the file system triggers a corresponding cleanup in the entity store, so the two always stay in sync.

6.9 Why This Matters

The entity store is what transforms BSearch from a document search tool into a system that genuinely understands the deal world. Without it, BSearch could find documents that mention a company name — but it could not tell you whether that company was the target, the acquirer, or the advisor. It could not filter deals by EBITDA or show you which deals are active. It could not connect a code name to a real company name across hundreds of documents.

The entity store is also what makes BSearch's intelligence cumulative. Every document that gets ingested adds to it — new aliases, new relationships, updated financials, new stage evidence. The system gets smarter over time without any manual maintenance. That is the competitive moat over generic enterprise search tools: IB-specific entity understanding that encodes the tribal knowledge currently living in bankers' heads.

Key Takeaway

The PostgreSQL entity store gives BSearch structured understanding of deals, companies, people, and their relationships — not just the ability to find documents that contain certain words. Aliases accumulate automatically so no name is ever lost. Stages and status are inferred from document types and activity. Financial metrics update themselves as new documents arrive. And provenance tracking ensures that when documents change, the entity store stays clean. The result is a system that builds institutional memory the way a senior banker does — but never forgets and never leaves the firm.
