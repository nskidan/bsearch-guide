Chapter 6

Relevance & Quality Controls

How BSearch decides whether results are good enough to show — and what it does when something goes wrong.

Every search system faces the same question: “How confident are we in these results?” In investment banking, you would never hand a client a valuation without sanity-checking the comps. BSearch applies the same discipline to every query. This chapter covers the guardrails that separate high-confidence answers from marginal ones, how the system monitors its own health, and what users see at each quality level.

6.1 Relevance Gating

Relevance Gating — A system of confidence tiers that determines how BSearch presents results based on how well they actually match your query. Think of it as a quality filter that prevents bad results from reaching you.

After the reranker (Chapter 3, Section 3.5) scores each candidate chunk against your query, BSearch does not blindly pass everything to Claude for synthesis. Instead, it checks each chunk’s rerank score — a number between 0 and 1 that represents how relevant Cohere’s reranker thinks the chunk is to your query.

The three confidence tiers

Tier Rerank Score What Happens
HIGH ≥0.50 Strong match. Results go to Claude for normal synthesis. No banner shown to the user.
LOW 0.15 – 0.50 Marginal match. Claude still synthesizes, but adds hedging language. A yellow banner warns the user.
NONE <0.15 No meaningful match. BSearch skips the Claude call entirely and shows a no-results message with suggestions.

In addition to the tier-level thresholds, there is a per-chunk floor of 0.10. Even within a HIGH-confidence result set, any individual chunk scoring below 0.10 gets dropped before synthesis. This prevents a single irrelevant chunk from polluting an otherwise strong answer.

IB Analogy

Think of relevance gating like the quality controls on a comp set. If your comps are strong matches (same industry, similar size, recent transactions), you present them with confidence. If they are borderline (adjacent sector, different geography), you still show them but caveat heavily: “these are directional only.” And if the screen returned nothing relevant (wrong SIC code, no recent deals), you tell the MD there are no good comps rather than forcing a bad set into the book.

Threshold Calibration

The thresholds (0.50, 0.15, 0.10) are not arbitrary guesses. During Phase 6 testing, we run 30+ representative queries across different deal types and document categories, analyze the score distributions, and adjust the thresholds before shipping. Scores cluster differently depending on the type of content being searched, so calibration against real data is essential.

6.2 Reranker Health Monitoring

Health Check — A lightweight background task that periodically pings an external service to verify it is still responding. If it fails repeatedly, the system switches to a fallback mode.

The Cohere reranker is a third-party AI service. Like any external dependency, it can go down. BSearch does not wait for a user query to fail before discovering a problem — it proactively monitors reranker health with a background task that pings Cohere every 60 seconds.

Failure detection and degraded mode

If three or more consecutive health checks fail, BSearch marks the reranker as degraded and fires an alert to Application Insights (our monitoring platform). Here is what changes:

  • Skip rerank step — results come directly from hybrid search without the reranker’s second pass.
  • Reduce candidate count — top-k drops from 20 to 12. Without the reranker to filter noise, fewer candidates means less risk of irrelevant results slipping through.
  • Cap confidence at LOW — even if results look strong, the system never assigns HIGH confidence without the reranker’s validation.
  • Orange banner — the user sees a distinct orange banner: “Search quality may be reduced — results are ranked without AI reranking. Verify key details against source documents.”
Orange vs. Yellow Banners

Orange (degraded reranker) and yellow (LOW confidence) are distinct signals that can appear simultaneously. Orange means the system itself is impaired — a health issue. Yellow means the results for this particular query are marginal. A user could see both if the reranker is down and the query returns borderline matches.

When the health checks start succeeding again, BSearch marks the reranker as healthy and fires a recovery event. Normal operation resumes automatically — no manual intervention required.

IB Analogy

This is like having a backup process when Bloomberg goes down. You can still price comps using yesterday’s closing data and public filings, but you caveat everything: “Bloomberg was unavailable — pricing is indicative only, verify before client delivery.” The work continues, but with extra caution flags.

6.3 Embedding Resilience

LRU Cache (Least Recently Used) — A memory store that keeps the most recently used items readily available. When it runs out of space, it discards the items that have not been accessed in the longest time.

The Voyage AI embedding service converts your query text into a 1,024-number vector (Chapter 3, Section 3.3). This is an external API call — and like any network call, it can be slow or fail. BSearch has two layers of protection:

Layer 1: LRU cache (~1,000 entries)

If you or a colleague searched for the same query recently, the embedding is already cached in memory. The cache lookup takes effectively 0ms, bypassing the Voyage API entirely. This helps with common queries (“latest CIMs,” “engagement letters”) and rapid-fire refinements of the same search.

Layer 2: 500ms hard timeout with BM25 fallback

If the Voyage API does not respond within 500 milliseconds, BSearch stops waiting and falls back to BM25-only search — keyword matching without the vector (semantic) component. This means the search loses its ability to match on meaning (e.g., “trading multiple” matching “EV/EBITDA”), but the synonym expansion and reranker steps still compensate for much of that loss.

Not Surfaced to Users

Unlike the reranker degradation, embedding fallbacks are not shown to users. They are transient (per-query, not system-wide) and the quality impact is usually minor. However, they are logged, and if the fallback rate exceeds 10% of queries over a one-hour window, an Application Insights alert fires so we can investigate.

IB Analogy

Think of it like preparing a comp set. Normally you use CapIQ for both screening (keyword filters) and reading equity research (conceptual analysis). If CapIQ is being slow, you can still run the screen based on hard filters (industry, size, geography) while CapIQ catches up. You lose the conceptual layer temporarily, but the core screen still works.

6.4 Near-Duplicate Deduplication

Every IB professional has experienced this: you search for a deal summary and the top five results are “Deal Summary v2,” “Deal Summary v3,” “Deal Summary v3 FINAL,” “Deal Summary v3 FINAL (2),” and “Deal Summary v3 FINAL - Copy.” Five slots consumed by the same document. BSearch’s deduplication system prevents this.

How it works

Deduplication runs in two phases during ingestion, before documents ever appear in search results:

  1. Phase 1 — Filename normalization. Strip version markers (“v2,” “v3”), finality markers (“FINAL,” “DRAFT”), and copy markers (“(2),” “Copy of”). If two files reduce to the same normalized name, they are flagged as potential duplicates.
  2. Phase 2 — Cosine similarity check. Compare the first-chunk embeddings of the two files. If the cosine similarity is ≥0.95 (nearly identical content), they are confirmed as duplicates.

A same-folder constraint prevents false matches. Two files named “Financial Summary” in different deal folders are not duplicates — they are summaries for different deals. The system only groups files within the same folder.

Among duplicates, the most recently modified file is marked as the canonical version. All chunks from canonical files carry an is_canonical flag set to true, and search results filter on this flag. The filter adds zero latency to queries because it is evaluated at the index level, not after retrieval.

IB Analogy

This is exactly like the document management discipline a good deal team follows: when you save “v4 FINAL,” the earlier versions get moved to an archive subfolder so that anyone searching the active folder only finds the latest version. BSearch automates this curation at scale across 550,000 documents.

6.5 Citation Format

Every answer BSearch generates includes numbered citations so you can verify the source material. Each citation includes:

  • File name — the original document name.
  • Folder path — which deal folder or shared drive location the document lives in.
  • Document type (doc_type) — CIM, engagement letter, financial model, etc.
  • “Open in SharePoint →” — a deep link via the m365_url field that opens the document directly in SharePoint. One click from answer to source.

Citations appear in a Sources section at the end of each response, numbered to match inline references in the answer text. If Claude references a fact from source [2], you can scroll down and click through to verify it.

IB Analogy

Think of it like footnotes in an investor presentation. Every data point cites its source — “Source: Company management, FY2024 audited financials” — so the reader can trace any number back to its origin. BSearch does the same thing, but with clickable links that take you directly to the document.

6.6 Confidence Communication

All the quality machinery described above ultimately translates into a simple, clear experience for the user. Here is what you see in each scenario:

Scenario What You See
HIGH confidence Normal response, no banner. The system is confident in its answer.
LOW confidence Yellow banner: “Results may not fully answer your query.” Response includes hedging language.
NONE — no results No-results message with suggested alternative queries. Claude is not called at all.
Degraded reranker Orange banner: “Search quality may be reduced.” A system health issue, independent of query quality.
Multi-document synthesis “Searching across multiple documents...” indicator while the system works.

The design principle is simple: never let the user mistake a low-confidence answer for a definitive one. In IB, the cost of acting on a wrong answer — a misquoted EBITDA, a wrong deal status — can be significant. These banners are the system’s way of saying “double-check this one.”

Key Takeaway

BSearch treats search quality as a first-class concern, not an afterthought. Every result is assigned a confidence tier, the reranker is continuously monitored, embedding failures have automatic fallbacks, and near-duplicates are suppressed before they clutter your results. The user always knows how much to trust a given answer — just as you would caveat a preliminary valuation differently than a final fairness opinion.
