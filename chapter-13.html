Chapter 13

Glossary

Every technical term in this guide, defined in plain English.

API (Application Programming Interface)

A defined set of rules that lets one piece of software talk to another. When you hear "call an API," it means one system is sending a structured request to another and getting a structured response back.

In BSearch: the query pipeline is itself a FastAPI-based API. It also calls external APIs — Microsoft Graph for document access, Voyage AI for embeddings, Cohere for reranking, and Anthropic's Claude for synthesis.

Azure AD / Entra ID

Microsoft's cloud identity service that manages who can sign in and what they can access. Originally called Azure Active Directory, now rebranded to Microsoft Entra ID.

In BSearch: every user authenticates via Azure AD using the firm's existing M365 credentials. The web app and FastAPI backend both validate Azure AD tokens. Folder-level permissions are mapped to Azure AD identities to filter search results.

Azure AI Search

Microsoft's managed search service that stores and queries indexed content. It supports full-text keyword search, vector search, and combinations of both.

In BSearch: the primary search index. Stores all document chunks with their text, embeddings (1024-dimensional vectors from Voyage AI), metadata fields (doc_type, deal_id, source_folder), and entity fields. Supports hybrid queries combining BM25 keyword matching with vector similarity, plus OData filters for permissions and entity-aware narrowing.

Azure App Service

A managed platform for hosting web applications without managing servers. You deploy your code; Azure handles the infrastructure, scaling, and uptime.

In BSearch: hosts the FastAPI query API on a B1 tier (~$13/month). Chosen over Azure Functions for the API because App Service is always-on — no cold-start delays that would violate the <3-second TTFT target.

Azure Document Intelligence

A Microsoft AI service (formerly Form Recognizer) that extracts text, tables, and structure from documents — especially PDFs and scanned images — using machine learning models trained on document layouts.

In BSearch: the PDF parser in the ingestion pipeline. It preserves table structures and reading order that simple text extraction would lose, which is critical for financial documents where tables contain the most valuable data.

Azure Durable Functions

An extension of Azure Functions that lets you write long-running, stateful workflows in code. It handles checkpointing, retries, and coordination automatically — if a workflow fails partway through, it can resume from where it left off.

In BSearch: the ingestion pipeline runs as Durable Functions orchestrators. The bulk ingestion of ~550K documents uses the fan-out/fan-in pattern to process documents in parallel with automatic checkpointing. Webhook-triggered and delta sync ingestion use the same orchestrator codepath.

Azure Static Web Apps

A hosting service optimized for static frontends (HTML, CSS, JavaScript). It serves files from a global CDN with built-in Azure AD authentication and custom domain support, requiring zero server management.

In BSearch: hosts the Vite + React single-page application. Platform-managed Azure AD OIDC authentication means the frontend gets secure login without any custom auth code.

Bicep

A domain-specific language for deploying Azure resources as code. Instead of clicking through the Azure portal, you define your infrastructure in .bicep files that can be version-controlled, reviewed, and deployed repeatably.

In BSearch: all Azure infrastructure (AI Search, App Service, Durable Functions, Cosmos DB, Static Web Apps, Key Vault) is defined in Bicep templates under the infra/ directory, with separate parameter files for dev and prod environments.

BM25

A keyword-matching algorithm that ranks documents by how well their words match the query. It considers word frequency, document length, and how rare a word is across the entire index. It is the standard algorithm behind most traditional search engines.

In BSearch: one half of hybrid search. BM25 handles exact keyword matches (company names, deal terms, file names) where vector search might miss. Static IB synonym expansions and Haiku-generated reformulations are appended to the BM25 query to improve recall. Original query terms receive a higher BM25 boost weight (3.0) than expansion terms (1.5).

Chunk

A segment of a document that has been split into a manageable piece for indexing and retrieval. Documents are too long to process whole, so they are divided into chunks that each capture a coherent unit of information.

In BSearch: chunking is format-aware. Narrative text is split into 800–1,000 token chunks. Tables are kept intact (or split at structural boundaries if over 1,500 tokens). PowerPoint slides become one chunk per slide. Each chunk is embedded independently and stored in Azure AI Search with metadata linking it back to its source document.

CIM (Confidential Information Memorandum)

A detailed document prepared by a sell-side investment bank that describes the company being sold — its business, financials, market position, and growth prospects. It is the primary marketing document in an M&A process, shared with prospective buyers after they sign an NDA.

In BSearch: CIMs are among the most frequently searched document types. The system classifies them via doc_type extraction, and the IB synonym map expands "CIM" to include "confidential information memorandum," "offering memorandum," and "info memo" so queries find them regardless of phrasing.

Claude (Sonnet / Haiku)

A family of large language models built by Anthropic. Sonnet is the mid-tier model optimized for quality and speed. Haiku is the smallest and fastest model, designed for high-volume, low-latency tasks.

In BSearch: Claude Sonnet 4.6 is the synthesis model — it reads retrieved document chunks and generates the final answer streamed to the user. Claude Haiku 4.5 handles lightweight tasks: query classification, query expansion, doc_type classification during ingestion, and doc/deal summary generation. This split keeps latency low and costs manageable (~$11/month for Sonnet at expected query volume).

Cohere Rerank

A service from Cohere that takes a query and a list of candidate text passages, then re-scores them by how relevant each passage actually is to the query. It uses a cross-encoder model that reads the query and each passage together, producing much more accurate relevance scores than initial retrieval alone.

In BSearch: after hybrid search returns the top 20 candidates, Cohere Rerank re-scores them against the original query (not the expanded version) and selects the top 5. This step filters out noise introduced by synonym expansion and corrects ranking errors from the initial retrieval. A background health check pings Cohere every 60 seconds; on failure, reranking is skipped and confidence is capped at LOW.

Confidence Gating

A mechanism that evaluates how confident the system is in its search results before deciding what to do with them. Instead of always sending results to the LLM, the system checks whether results are good enough to warrant an answer.

In BSearch: a three-tier gate based on Cohere rerank scores. HIGH (≥0.50): proceed to Claude synthesis normally. LOW (0.15–0.50): proceed but add hedging language to the Claude prompt. NONE (<0.15): skip the Claude call entirely and tell the user no relevant results were found. Individual chunks below 0.10 are dropped. Thresholds are configurable via environment variables.

Contextual Embeddings

A technique where each chunk's embedding is generated with awareness of the full document it came from, not just the chunk's own text. This produces richer numerical representations because the model understands each chunk's role within the broader document.

In BSearch: Voyage AI's voyage-context-3 model supports contextual embeddings natively. During ingestion, each chunk is embedded with its document context (up to 32K tokens), so a table labeled "Revenue" is understood differently when it appears in a CIM versus a quarterly report. The embedding provider abstraction passes a document_context parameter that contextual models use and standard models ignore.

Cosine Similarity

A mathematical measure of how similar two vectors are, based on the angle between them. A score of 1.0 means identical direction (very similar meaning); 0.0 means completely unrelated. It ignores magnitude and focuses purely on direction.

In BSearch: used in two places. First, vector search in Azure AI Search ranks chunks by cosine similarity between the query embedding and chunk embeddings. Second, deduplication compares first-chunk embeddings of candidate duplicate files — pairs with cosine similarity ≥0.95 are confirmed as near-duplicates.

Cosmos DB

Microsoft's globally distributed, serverless NoSQL database. In serverless mode, you pay only for the storage and compute you actually use — at low volumes the cost approaches zero.

In BSearch: stores operational state — conversation history, delta sync tokens, webhook subscription records, retry queues, permission configuration, and duplicate cluster state. Chosen for its serverless pricing model (~$0/month at 18 users) and native integration with Azure Functions.

Cross-Encoder

A neural network architecture that reads a query and a document passage together as a single combined input, then outputs a relevance score. Unlike bi-encoders (which embed query and document separately), cross-encoders capture fine-grained interactions between query and document words, producing more accurate relevance judgments — but at higher computational cost.

In BSearch: Cohere Rerank is a cross-encoder. It is used after initial retrieval (not for first-pass search, which would be too slow at 550K documents) to re-score the top 20 candidates down to the top 5.

Data Extraction

The process of pulling a specific piece of information — a number, name, date, or metric — out of one or more documents. Distinguished from broad summarization by its precision: the user wants a particular fact, not an overview.

In BSearch: one of three query types. When the classifier detects a data extraction query (e.g., "What was the EBITDA for Acme?"), the pipeline biases retrieval toward table and model_summary chunk types, uses standard top-20 → rerank top-5, and prompts Claude for precise extraction with citations. This is the default classification when ambiguous.

Deal Summary

A pre-computed, concise summary (~400 tokens) that captures the key facts about a deal — parties involved, deal stage, financial metrics, and timeline — synthesized from all documents associated with that deal.

In BSearch: generated by Haiku during ingestion for deals with ≥3 associated documents. Stored as a special chunk type in the search index. Refreshed when deal-related files change and on a 6-hour timer. At query time, deal summaries reduce the cross-deal reasoning burden for multi-doc synthesis queries.

Deduplication (Near-Duplicate Detection)

The process of identifying files that are effectively the same document — different versions, copies with minor edits, or files saved under slightly different names. The goal is to surface only the most current version in search results.

In BSearch: a two-phase process at ingestion time. Phase 1: normalize filenames (strip "FINAL," "(2)," version markers) and group by normalized name within the same source folder. Phase 2: compare first-chunk embeddings via cosine similarity (≥0.95 = confirmed duplicate). The most recently modified file is marked is_canonical = true; all search queries filter on this field. Non-canonical chunks remain indexed for re-evaluation when files change.

Delta Sync

A synchronization strategy that only processes changes since the last sync — new, modified, or deleted items — rather than re-processing everything from scratch. "Delta" means "the difference."

In BSearch: the primary ongoing ingestion mechanism. Microsoft Graph delta queries return only files that changed since the last sync token. A timer-triggered delta sync runs every 12–24 hours as a fallback for any changes that webhooks might have missed. Delta tokens are stored in Cosmos DB.

Doc Summary

A brief summary (~200 tokens) of an entire document's content, generated by an LLM. It acts as a semantic bridge — connecting the themes of distant sections within the same document.

In BSearch: generated by Haiku for documents that produce ≥4 chunks. Indexed as chunk_type: "doc_summary" in Azure AI Search. Helps retrieval find the right document even when no single chunk matches the query well. Sibling chunk retrieval (injecting doc summaries alongside related chunks at query time) is deferred to v2.

Embedding

A list of numbers (a vector) that represents the meaning of a piece of text. Texts with similar meanings produce vectors that point in similar directions, enabling semantic search — finding content by meaning rather than exact keywords.

In BSearch: every document chunk and every search query is converted into a 1024-dimensional embedding vector using Voyage AI's voyage-context-3 model. These vectors are stored in Azure AI Search and compared via cosine similarity during vector search. The system supports scaling to 2048 dimensions via Matryoshka Representation Learning if needed.

Entity Extraction

The process of identifying and extracting structured information — company names, people, financial figures, deal stages — from unstructured document text. A subset of Named Entity Recognition, focused on domain-specific entities.

In BSearch: during ingestion, Haiku extracts deal-level entities (company names, roles, industries, financial metrics, deal stage, key contacts) from documents. These entities are written to Neo4j as nodes and relationships, and key fields (deal_id, doc_type, industries, company_roles) are stored as filterable fields on each chunk in Azure AI Search.

Extended Thinking

A Claude feature that allocates a separate "thinking" budget before generating the visible response. The model reasons through the problem internally — planning, cross-referencing, and organizing — before writing its answer, producing higher-quality output for complex tasks.

In BSearch: enabled for multi-doc synthesis queries with a budget of ~2,000 thinking tokens. This lets Claude organize cross-document reasoning (comparing deals, aggregating findings) before streaming the answer. Not used for file lookup or data extraction queries where speed matters more than reasoning depth.

Fan-out / Fan-in

A workflow pattern where a single task spawns many parallel sub-tasks (fan-out), waits for all of them to complete, then aggregates the results (fan-in). Common in distributed processing when you need to apply the same operation to many items concurrently.

In BSearch: the bulk ingestion orchestrator uses this Durable Functions pattern. A single orchestrator fans out to process thousands of documents in parallel (each getting parsed, chunked, embedded, and indexed), then fans in to report completion status and update sync tokens.

FastAPI

A modern Python web framework for building APIs. It uses Python type hints for automatic request validation and documentation, supports asynchronous request handling, and is one of the fastest Python frameworks available.

In BSearch: the query API runs on FastAPI, deployed to Azure App Service. It handles the full query pipeline: classification, expansion, hybrid search, reranking, confidence gating, and Claude synthesis streaming. FastAPI's native async support is essential for the parallel execution of classification and embedding tasks.

File Lookup

A query type where the user wants to locate a specific document — they want a link to open it, not information extracted from it.

In BSearch: one of three query types. Detected by regex fast path (filename with extension, or locator verb + document name with no content-question words) or by Haiku classification. Uses BM25-heavy search on the file_name field, returns top-1 with a direct link. Skips embedding entirely — BM25 on filenames is sufficient. Claude generates a terse response with the link and minimal context.

Graph API (Microsoft)

Microsoft's unified API for accessing data across Microsoft 365 services — SharePoint, OneDrive, Outlook, Teams, and more. It provides a single endpoint for reading files, user profiles, emails, calendar events, and organizational data.

In BSearch: the data source for all document content. The ingestion pipeline uses Graph API to enumerate SharePoint sites and OneDrive drives, download file content, and track changes via delta queries. Graph API webhooks provide near-real-time notifications when files are created, modified, or deleted.

HNSW (Hierarchical Navigable Small World)

An algorithm for efficiently searching through millions of high-dimensional vectors. It builds a layered graph structure where each layer provides a progressively coarser view, enabling fast approximate nearest-neighbor search without scanning every vector.

In BSearch: the vector index algorithm used by Azure AI Search to find chunks whose embeddings are most similar to the query embedding. HNSW enables sub-second vector search across millions of chunk embeddings (the ~550K documents produce roughly 2.2M chunks).

Hybrid Search

A search strategy that combines two complementary approaches: keyword search (BM25) for exact term matching and vector search for semantic meaning matching. The results from both are merged, capturing documents that match either by words or by concepts.

In BSearch: every non-file-lookup query uses hybrid search. BM25 catches exact company names, deal terms, and financial jargon that vectors might miss. Vector search catches conceptually related content even when wording differs. Azure AI Search runs both in a single query and merges results using Reciprocal Rank Fusion. If the embedding times out, the system falls back to BM25-only.

Index (Search Index)

A structured data store optimized for fast retrieval. Like a book's index, it lets you find information quickly without scanning every page. In search systems, an index stores pre-processed content (text, vectors, metadata) in a format designed for rapid querying.

In BSearch: the Azure AI Search index (named bsearch-v1) stores all document chunks with their text content, 1024-dimensional embedding vectors, and metadata fields (doc_type, deal_id, source_folder, is_canonical, duplicate_group_id, industries, company_roles, deal_status). The index schema is defined in Bicep and supports blue-green re-indexing for embedding model swaps.

Ingestion Pipeline

The end-to-end process that takes raw documents and transforms them into searchable, indexed content. It includes downloading files, parsing their content, splitting them into chunks, generating embeddings, extracting entities, and storing everything in the search index.

In BSearch: runs as Azure Durable Functions with format-aware parsers for Word (python-docx), Excel (openpyxl), PowerPoint (python-pptx), and PDF (Azure Document Intelligence). After parsing, content is chunked, embedded via Voyage AI, enriched with entity metadata via Haiku, and indexed into Azure AI Search. Entity data is written to Neo4j. The same pipeline handles bulk ingestion, webhook-triggered updates, and periodic delta syncs.

Knowledge Graph

A database that stores information as entities (things) and relationships (connections between things) in a graph structure. Unlike tables, graphs naturally represent complex webs of relationships — which company is the target of which deal, who advises whom, how entities relate across transactions.

In BSearch: Neo4j Aura stores the knowledge graph with nodes for Deals, Companies, People, and StageEntry records. Relationships capture roles (ADVISED_BY, TARGET_OF, ACQUIRER_OF), affiliations (WORKS_AT), and stage progression. At query time, entity context from the graph is injected into Claude's synthesis prompt, and graph traversals answer relationship questions ("Which companies has Berenson advised in healthcare?").

LRU Cache (Least Recently Used Cache)

A fixed-size in-memory store that keeps the most recently accessed items and automatically evicts the oldest when full. It speeds up repeated operations by returning cached results instead of recomputing them.

In BSearch: an in-memory LRU cache (~1,000 entries) stores query-to-embedding mappings. Before calling Voyage AI for a query embedding, the cache is checked first (~0ms on hit). This reduces latency and API costs for repeated or similar queries, and provides a fallback data source if the embedding service is temporarily unavailable.

Matryoshka Representation Learning (MRL)

A training technique that produces embedding vectors where the first N dimensions are themselves a valid (lower-resolution) embedding. Like nested Russian dolls, you can truncate the vector to a smaller size and still get useful similarity scores — just with less precision.

In BSearch: Voyage AI's voyage-context-3 supports MRL. The system uses 1024 dimensions by default but can scale to 2048 dimensions for higher precision without re-training the model or changing the embedding pipeline — only the index configuration needs updating.

Mermaid

A JavaScript library that renders diagrams from text descriptions. You write a simple text-based syntax (like a flowchart or sequence diagram), and Mermaid converts it into a visual diagram in the browser.

In BSearch: used throughout this technical guide to render architecture diagrams, flow charts, and sequence diagrams directly in the HTML pages without requiring image files. The Mermaid library is loaded from a CDN and initializes on page load.

Multi-Doc Synthesis

The process of reading across multiple documents and composing a single, coherent answer that draws on information from all of them. This requires the AI to reason about relationships, identify patterns, and reconcile potentially conflicting information from different sources.

In BSearch: one of three query types, used for broad questions like "What healthcare deals have we worked on?" or "Where do we stand on the Acme process?" Retrieval expands to top-10 chunks through reranking. Claude uses extended thinking (~2,000 token budget) with a structured extraction-then-synthesis prompt. Pre-computed deal summaries reduce the reasoning burden. TTFT target is relaxed to 5 seconds (vs. 3 seconds for other query types).

Neo4j

A graph database that stores data as nodes (entities) and relationships (connections), with properties on both. It uses the Cypher query language, which is designed for traversing and pattern-matching across graph structures.

In BSearch: Neo4j Aura hosts the knowledge graph. Entity extraction during ingestion writes Deal, Company, and People nodes with their relationships. At query time, Cypher queries retrieve entity context (deal status, financials, company roles, stage history) that is injected into Claude's prompt. For cross-deal queries, a composable Cypher query builder constructs traversals dynamically. Free tier for dev; ~$65/month Pro tier for production.

NER (Named Entity Recognition)

A natural language processing task that identifies and classifies named entities in text — such as people, companies, locations, dates, and monetary amounts — into predefined categories.

In BSearch: entity extraction during ingestion is a form of NER, performed by Haiku rather than a traditional NER model. Haiku identifies domain-specific entities (company names with roles, deal stages, financial metrics, industry classifications) that a generic NER system would miss or miscategorize in investment banking documents.

OData

A standardized query protocol that lets you filter, sort, and select data using URL parameters. It provides a SQL-like filtering syntax that can be appended to API requests.

In BSearch: Azure AI Search uses OData filter expressions to enforce permissions and entity-aware narrowing. Every search query includes source_folder and is_canonical eq true filters. Entity-aware queries add filters like industries/any(i: i eq 'healthcare') or deal_status eq 'active'.

OIDC (OpenID Connect)

An authentication protocol built on top of OAuth 2.0 that lets applications verify a user's identity through a trusted provider (like Microsoft) and obtain basic profile information. The user signs in once with the provider and the application receives a verified identity token.

In BSearch: Azure Static Web Apps uses platform-managed OIDC with Azure AD. Users authenticate with their existing Berenson M365 credentials. The OIDC token is validated by the FastAPI backend to determine user identity and folder permissions.

Query Expansion

The technique of adding related terms, synonyms, and alternative phrasings to a search query before executing it. This improves recall by matching documents that use different words to express the same concept.

In BSearch: two-layer expansion with zero added latency. Layer 1: a static IB synonym map (ib_synonyms.json) expands terms like "comp" to "comparable companies, precedent transactions, trading comps." Layer 2: Haiku generates 2–3 reformulations as part of the classification call (already running in parallel with embedding). Expansions enrich BM25 only; vector search uses the original query. Cohere reranker scores against the original query to filter expansion noise.

Query Pipeline

The sequence of processing steps that transforms a user's natural-language question into a final answer. Each stage refines, retrieves, filters, ranks, or synthesizes until a response is ready.

In BSearch: the pipeline has five stages, executed in the FastAPI backend. (1) Query classification + expansion (regex or Haiku, in parallel with embedding). (2) Entity context lookup from Neo4j if entity intent is detected. (3) Hybrid search in Azure AI Search with permission and entity filters. (4) Cohere reranking of top-20 to top-5, with confidence gating. (5) Claude streaming synthesis with retrieved chunks and optional entity context.

Reranking

A second-pass relevance scoring step applied to an initial set of search results. A more expensive but more accurate model re-evaluates each candidate against the original query, producing a refined ranking that better reflects true relevance.

In BSearch: Cohere Rerank re-scores the top 20 candidates from hybrid search down to the top 5 (or top 10 for multi-doc synthesis). Scoring is against the original query, not the expanded version, so expansion noise is filtered out. Rerank scores also drive confidence gating — the highest score determines whether the system proceeds with HIGH, LOW, or NONE confidence.

Scoring Profile

A set of rules in Azure AI Search that adjusts how results are ranked beyond basic text or vector matching. Scoring profiles can boost results based on field values, freshness, geographic proximity, or custom weights.

In BSearch: a recency scoring profile boosts more recently modified documents in search results. This helps surface the latest version of a document when multiple versions exist and addresses the firm's core pain point of working from outdated files.

SHA-256

A cryptographic hash function that takes any input and produces a fixed 256-bit (64-character hex) fingerprint. Even a tiny change in the input produces a completely different hash, making it useful for detecting whether content has changed.

In BSearch: used during ingestion to generate content hashes for document chunks. When a document is re-ingested (via webhook or delta sync), chunk hashes are compared against stored values to determine whether re-embedding is necessary, avoiding redundant Voyage AI API calls for unchanged content.

Source Folder

A top-level folder in SharePoint or OneDrive that serves as both an organizational unit and a permission boundary. Access is granted or denied at the folder level, not per individual file.

In BSearch: there are 5 top-level source folders. Each chunk in the search index carries a source_folder field. Every search query includes an OData filter that restricts results to folders the authenticated user has access to, enforcing the firm's existing permission model without any per-file ACL complexity.

SSE (Server-Sent Events)

A web standard for pushing real-time updates from a server to a browser over a single HTTP connection. Unlike WebSockets (which are bidirectional), SSE is one-way: the server streams data to the client. Simple, reliable, and works through most firewalls and proxies.

In BSearch: the FastAPI backend streams Claude's synthesis response to the React frontend via SSE. As Claude generates each token, it is pushed to the browser in real time, giving the user visible progress within seconds of submitting a query. SSE events also carry metadata like degraded_reranker: true to trigger UI indicators.

Streaming

Sending data incrementally as it becomes available rather than waiting for the entire result to be ready. In the context of LLMs, streaming means sending each generated token to the user as soon as it is produced, so the response appears word-by-word in real time.

In BSearch: Claude's synthesis response is streamed via SSE from FastAPI to the React frontend. This is critical for perceived latency — the user sees the first words of the answer within 3–5 seconds (TTFT), even though the full response may take 10–15 seconds to complete. Without streaming, users would stare at a blank screen until the entire response was finished.

Synonym Map

A lookup table that maps a word or phrase to a set of equivalent or related terms. When a search query contains a mapped term, the system automatically includes the synonyms, improving recall without requiring the user to guess the exact wording used in documents.

In BSearch: a curated JSON file (ib_synonyms.json) loaded at startup. Maps IB-specific terms — e.g., "comp" to ["comparable companies," "trading comps," "precedent transactions"], "lbo" to ["leveraged buyout," "LBO model," "buyout model"]. Applied to the BM25 portion of hybrid search only (not vector). Editable without redeployment.

TTFT (Time to First Token)

The elapsed time from when a user submits a query to when the first word of the AI-generated response appears on screen. A key measure of perceived responsiveness in streaming systems — even if the full answer takes longer, a fast TTFT makes the system feel instant.

In BSearch: the primary latency target. File lookup and data extraction queries target <3-second TTFT. Multi-doc synthesis queries target <5-second TTFT (with a "Searching across multiple documents..." indicator). Every stage of the query pipeline is optimized to minimize TTFT: parallel classification and embedding, LRU caching, reranker health monitoring, and streaming synthesis.

Vector Search

A search technique that finds content by comparing the mathematical meaning of a query against the mathematical meaning of indexed content. Instead of matching keywords, it compares embedding vectors using similarity measures like cosine similarity, finding content that is conceptually related even when different words are used.

In BSearch: one half of hybrid search. Vector search uses HNSW-indexed 1024-dimensional embeddings from Voyage AI to find chunks whose meaning is closest to the query. It catches conceptual matches that keyword search misses (e.g., a query about "valuation" matching a chunk discussing "enterprise value multiples"). Falls back gracefully to BM25-only if the embedding service times out.

Voyage AI

An AI company specializing in embedding models optimized for retrieval and search. Their models convert text into high-quality vector representations with support for contextual embeddings and Matryoshka Representation Learning.

In BSearch: the embedding provider. The voyage-context-3 model generates 1024-dimensional contextual embeddings for both document chunks (at ingestion time, with full document context) and search queries (at query time). Selected over OpenAI's text-embedding-3-large for +14.24% chunk retrieval performance and native contextual embedding support. voyage-4-large is the designated fallback model (drop-in, no pipeline changes required).

Webhook

A mechanism where one system automatically sends a notification to another when a specific event occurs. Instead of constantly polling ("did anything change?"), the source system pushes a message the moment something happens — a file is created, modified, or deleted.

In BSearch: Microsoft Graph API webhooks notify the ingestion pipeline within 30–120 seconds when files change in SharePoint or OneDrive. Webhook subscriptions are stored in Cosmos DB and renewed every 12 hours via a timer trigger. Webhooks trigger the same delta query + orchestrator pipeline used by periodic sync. A 12–24-hour delta sync timer runs as a safety net for any missed webhook events.
