Chapter 13

Phase 6 — Testing & Go-Live

Acceptance tests, bulk ingestion, bootstrap deployment, and the cost model. This is where you prove BSearch works.

13.1 What You're Building

This is the final phase. Everything built in Phases 1–5 — the infrastructure, parsers, ingestion pipeline, query API, and frontend — now gets wired together and validated against real acceptance criteria. You are doing three things: running a comprehensive test suite that proves BSearch answers real questions correctly, executing the full bulk ingestion of ~550,000 documents, and deploying the complete system to production in a specific sequence that avoids race conditions and wasted cost.

The acceptance tests are not abstract. They are five specific queries that Berenson users will actually ask, chosen to exercise every capability of the system: finding a file, extracting a data point, synthesizing across documents, navigating entity relationships, and handling version recency. If BSearch can answer all five well — with correct citations, within latency targets — the system is working as designed.

The bulk ingestion is the most operationally intensive step in the entire project. Processing 550K documents into ~2.2M searchable chunks takes 18–36 hours and costs ~$11–12K (primarily embedding and PDF parsing). It runs once, and after that, ongoing ingestion is incremental via webhooks and delta sync. The bootstrap deployment sequence ensures everything comes up in the right order: infrastructure first, then code, then webhooks (only after bulk ingestion completes, to avoid noise during the initial load), then the frontend.

13.2 Before You Start

Before You Start
  • All previous phases complete. Phases 1–5 are done and working individually. The config loads, parsers handle all four formats, the ingestion pipeline processes documents end-to-end, the query API returns streaming answers, and the frontend renders results with citations.
  • Full SharePoint access confirmed. The Azure AD app registration has Sites.Read.All, Files.Read.All, and User.Read.All permissions — all with admin consent granted by IT. Test this by running a Graph API call that lists all SharePoint sites in the tenant. If you get a 403, stop here and get IT to grant consent.
  • Confirm bulk ingestion timing. The full ingestion takes an estimated 18–36 hours. Plan to start it on a Friday evening or Saturday morning so it can run over a weekend. Durable Functions checkpointing means it survives restarts, but you want an uninterrupted window for the initial load.
  • Budget approved for one-time ingestion cost. The bulk ingestion costs approximately $11,000–$12,000. The largest component is Azure Document Intelligence for PDF parsing (~$9,200 after pre-parse PDF deduplication), followed by Claude Haiku entity extraction (~$1,200–$1,800) and Voyage AI bulk embedding (~$500–$1,000). This is a one-time cost — ongoing monthly costs are ~$343–$471. Get explicit sign-off before starting.

13.3 The Spec

This section covers the acceptance test queries, latency targets, bulk ingestion throughput model, bootstrap deployment sequence, monthly cost model, recovery paths, and blue-green re-indexing strategy.

5 Acceptance Test Queries

These five queries are the definition of “done.” Each one tests a different capability of the system. If BSearch answers all five correctly — with accurate citations and within latency targets — the system is production-ready.

# Query Expected Result What It Tests
1 "Where's the latest GovTech deck?" The correct file with a direct SharePoint link — not three old versions. Should route via regex fast path (file_lookup), skip embedding entirely, and use BM25 on the file_name field. Near-duplicate dedup ensures only the canonical version surfaces. File lookup — finding the right document, fast. Also validates dedup and version detection.
2 "Which firms did we reach out to for Govini?" A list of firm names extracted from the relevant documents, with source citations pointing to specific chunks. Should route to data_extraction via Haiku classification. Data extraction — pulling specific facts from documents with precise citations.
3 "Find me a good situation overview slide we've put together in the past for a banking deal" A specific slide from a specific deck, with a link to the full presentation. PowerPoint chunks are one-per-slide, so the result should point to the exact slide, not just the file. Slide-level retrieval — finding content inside documents. Also validates format-aware chunking for PowerPoint.
4 "What multiple did we use for [deal name]?" The EV/EBITDA or other relevant multiple, pulled from the model or deal summary. Results should include chunks with “EV/EBITDA” or “Enterprise Value” terminology, even though the query says “multiple.” Synonym expansion — understanding that “multiple” means “EV/EBITDA” in IB context. Validates both the static IB synonym map and Haiku query expansion.
5 "Where do we stand on our active deals?" A deal-by-deal overview organized by deal name (not by source chunk), with citations per claim and gaps flagged. Should trigger multi_doc_synthesis routing, extended thinking, and pull deal summary chunks. The deep_search: true flag should appear in the SSE stream. Multi-document synthesis — cross-deal reasoning using deal summaries + extended thinking. The hardest query — validates the entire pipeline.
Beyond the five

The five acceptance queries are necessary but not sufficient. Phase 6 also includes testing query classification routing (verify “find the Catalyst NDA” routes to file_lookup, not data_extraction), query expansion (verify “pull the comps for Acme” finds “comparable companies”), relevance gating (query for a nonexistent deal → no-results response, not a hallucination), permission filtering, document versioning, source deep links, and the full suite of entity-aware queries.

TTFT Measurement Targets

Time to first token (TTFT) is measured per query type. Each target is tested by running 20+ queries of that type and computing p50 (median) and p95 (worst case excluding extreme outliers). If p95 exceeds the target, there is a latency problem to diagnose.

Query Type TTFT Target (p95) Why This Target What Drives Latency
file_lookup <2 seconds File lookups skip embedding entirely (BM25 on filename is sufficient). Users expect near-instant results for “find me this file.” BM25 query + Claude terse response. No embedding, no reranking.
data_extraction <3 seconds The default query type and the most common. Classification + embedding + search + reranking + Claude all run, but context is limited to 5 chunks. Parallel classification/embedding (~500ms) + search + rerank (~200ms) + Claude start (~1–1.5s).
multi_doc_synthesis <5 seconds Cross-document reasoning requires more context (10 chunks + deal summaries) and extended thinking (~2,000 token budget). Users see a “Searching across multiple documents...” indicator while waiting. Same pipeline as data_extraction, plus extended thinking overhead and larger context window.

p50 vs. p95 — p50 is the median: half of queries are faster, half slower. p95 means 95% of queries meet the target. The acceptance criteria use p95 because it catches latency spikes that p50 would hide. If p50 is 1.5s but p95 is 4s, you have a problem even though the “average” looks fine.

Bulk Ingestion Throughput Model

The initial ingestion of ~550,000 documents produces approximately 2.2 million searchable chunks. The bottleneck is Graph API download speed and embedding generation. The entire process runs as a Durable Functions fan-out/fan-in orchestrator with automatic checkpointing — if it crashes at hour 14, it resumes from where it stopped.

Stage Rate Limit Estimated Time (550K docs)
Graph API download ~10,000 requests per 10-minute window ~9–12 hours
Format parsing (Word, Excel, PPT, PDF) CPU-bound, parallelized across activity functions ~2–4 hours
Haiku entity extraction (Tier 1 — doc-level) ~100 requests/second ~1.5 hours (550K calls)
Haiku entity extraction (Tier 2 — high-value chunks) ~100 requests/second ~0.5 hours (~150K high-value chunks)
Voyage AI embedding ~100–500 embeddings/second ~1–6 hours (~2.2M chunks)
PostgreSQL entity writes ~500 operations/second ~1–2 hours
Azure AI Search indexing ~1,000 documents/second ~0.5–1 hour
Total Wall Time

18–36 hours estimated. The bottleneck is Graph API download + embedding. Plan for a 2-day weekend window with checkpoint/resume via Durable Functions. Progress is monitored through Application Insights. Azure PostgreSQL Flexible Server (B2s, ~$26/month) handles the bulk load comfortably.

Run a pilot first

Before running the full 550K ingestion, run a pilot on a test SharePoint folder with ~100 representative documents (a mix of Word, Excel, PowerPoint, and PDF). This validates the entire pipeline end-to-end at low cost. Run the 5 acceptance test queries against the pilot data. Only proceed to full ingestion after the pilot passes.

Bootstrap Deployment Sequence (9 Steps)

This is the greenfield runbook — the exact order for deploying BSearch from scratch. Each step must succeed before proceeding to the next. The sequence is designed to avoid two specific problems: webhook noise during bulk ingestion (step 7 comes after step 6), and wasted compute from deploying the frontend before the index has data (step 8 comes last).

  1. Create Azure AD app registration. Register the application with Sites.Read.All, Files.Read.All, and User.Read.All permissions. Get admin consent from IT. Save the client_id, client_secret, and tenant_id.
  2. Provision Azure resources via Bicep. Deploy infra/main.bicep with the production parameter file. This creates Azure AI Search (S1), App Service (B2), Durable Functions (Consumption), Cosmos DB (serverless), Key Vault, Static Web Apps, and Application Insights. Verify all resources appear in the Azure portal.
  3. Store API keys in Key Vault. Populate Key Vault with all secrets: Voyage AI, Cohere, Anthropic API keys; Graph API credentials (client_id, client_secret, tenant_id); PostgreSQL connection string; Cosmos DB connection string; Azure AI Search admin key. Verify App Service and Functions can read secrets via managed identity.
  4. Deploy Azure Functions + App Service code. Deploy the ingestion pipeline (Durable Functions) and the query API (FastAPI on App Service). Verify the /health endpoint returns 200 on the App Service.
  5. Provision Azure PostgreSQL Flexible Server. Deploy via Bicep template, run the schema migration (creates deals, deal_aliases, companies, people, stage_entries, deal_companies, deal_people, company_people tables with indexes), and verify connectivity from the Azure Functions environment.
  6. Run bulk ingestion (18–36 hours, NO webhooks yet). Start the Durable Functions bulk ingestion orchestrator. It enumerates all documents across all SharePoint sites via Graph API delta queries, fans out to parse/chunk/embed/index activity functions, and writes progress to Cosmos DB. Webhooks are not registered yet — this prevents noise during the initial load. Monitor progress via Application Insights.
  7. Create Graph API webhook subscriptions. After bulk ingestion completes, register webhooks for all monitored drives. This enables near-real-time sync going forward. Verify by modifying a test file in SharePoint and confirming BSearch re-indexes it within a few minutes.
  8. Deploy Static Web App (React frontend). Deploy the web frontend with Azure AD authentication configured. Verify the login flow works with a Berenson user account.
  9. Smoke test. Verify the /health endpoint, run acceptance test query #1 (“Where's the latest GovTech deck?”), and check the Application Insights live stream for instrumentation events. If all three pass, the system is live.
Why webhooks come after bulk ingestion

If you register webhooks before bulk ingestion, every document in SharePoint triggers a webhook notification as it is being processed — creating a flood of duplicate work. By deferring webhook registration to step 7, the bulk ingestion runs uninterrupted, and webhooks only fire for genuinely new changes after the baseline is established.

Recovery Paths

Three failure scenarios require specific recovery procedures:

PostgreSQL Data Loss

If PostgreSQL data is lost, search continues without entity intelligence. Azure PostgreSQL Flexible Server provides continuous backups with point-in-time recovery (PITR) — restore to any point within the retention window takes minutes, not hours. As a last resort, the full entity store can be rebuilt from scratch by re-running the ingestion orchestrator on all indexed files.

Metric Value Meaning
RPO (Recovery Point Objective) Seconds to minutes Azure PostgreSQL Flexible Server provides continuous backups with point-in-time recovery. Worst case, you lose seconds to minutes of entity changes.
RTO (Recovery Time Objective) Minutes Point-in-time restore (minutes). Full re-ingestion only as last resort.
Degradation during rebuild Document-only search Entity-aware queries fall back to document-only search. No entity context in Claude prompts. postgresql_fallback logged. Users see normal results without entity enrichment.

Azure AI Search 503

If Azure AI Search returns 503 (service unavailable), this is hard downtime — there is no fallback for the search index itself. The query API returns a clear error, and the frontend shows “Search unavailable.” Auto-retry with exponential backoff covers transient issues. For sustained outages, check App Service health, then Azure AI Search availability, then Application Insights for error spikes.

Reranker Failure (Cohere Down)

When Cohere is unavailable, the system enters degraded mode: reranking is skipped, Azure Search's hybrid ranking (vector + BM25 + recency) is used on the top 12 results instead of the normal top 20, and confidence is capped at LOW. The background health checker detects the outage within 60 seconds (3 consecutive failures trigger an alert). The frontend renders a distinct orange degraded banner (different from the yellow LOW confidence banner). When Cohere recovers, the health checker detects it within 60 seconds, the next query returns degraded_reranker: false, and the orange banner disappears.

Blue-Green Re-Indexing

When you need to swap embedding models (e.g., upgrading from voyage-context-3 to a future model, or scaling from 1024 to 2048 dimensions), you cannot do a rolling update. Cosine similarity between vectors from different models is meaningless — the index must be fully consistent. The solution is blue-green re-indexing: create a parallel index, re-embed everything into it, validate, then flip.

The 5-Step Process

  1. Create new index. Create bsearch-v2 (or bsearch-{embedding_version}) with the new provider's vector dimensions. Identical schema to the current index except content_vector.dimensions matches the new model. This is a Bicep parameter change.
  2. Background re-indexing orchestrator. A Durable Functions orchestrator (reusing the existing fan-out/fan-in pattern) iterates all chunks in the old index: read chunk content + metadata in 1,000-chunk batches, embed the content field with the new provider, and write to the new index with updated embedding_model, embedding_dimensions, embedding_version, and content_vector. At ~500 embeddings/second, 2.2M chunks completes in ~1–2 hours. Durable Functions checkpointing means it survives restarts.
  3. Validation. Verify chunk count in the new index matches the old index (±tolerance for concurrent ingestion). Sample 100 random chunks and verify embedding_version and vector dimensions are correct. Run the acceptance test queries against the new index and compare result quality to the old index. Re-run dedup cluster evaluation with new-model embeddings and recalibrate dedup_cosine_threshold if the similarity distribution shifts.
  4. Flip active index. Update the ACTIVE_INDEX_NAME environment variable on App Service. The search API starts querying the new index. Restart App Service to flush the LRU embedding cache (cached vectors are from the old model).
  5. Cleanup / rollback. Keep the old index for 48 hours as a rollback safety net. If issues are found, flip ACTIVE_INDEX_NAME back. After the grace period, delete the old index.
Concurrent ingestion during re-index

New files arriving via webhooks or delta sync during the re-index window are written to both indexes. The ingestion pipeline checks config.reindex_in_progress — when true, it embeds with both the old and new providers and writes to both indexes. This prevents gaps in the new index. The config keys controlling re-indexing:

Config Key Default Description
ACTIVE_INDEX_NAME "bsearch-v1" Which index the search API queries
REINDEX_TARGET_INDEX null Target index for re-indexing job (null = no re-index in progress)
REINDEX_IN_PROGRESS false When true, ingestion dual-writes to active + target indexes
EMBEDDING_VERSION "v1" Incremented on any model/provider/dimension change

The key property of this approach: zero downtime, zero query impact. Queries always hit the active index, which has fully consistent embeddings. The re-indexing orchestrator runs in the background without affecting search performance. If re-indexing fails or produces worse results, the old index is still serving and the new index can be deleted.

Operational Runbook

Six common scenarios and what to do about each:

Scenario What to Check Resolution
Search returns 503 App Service health → Azure AI Search availability → Application Insights error spike Hard downtime. Wait for Azure AI Search recovery. Auto-retry with backoff handles transient issues.
Orange degraded banner Cohere status page Cohere is down. System auto-degrades. Wait for recovery — health checker restores automatically within 60 seconds.

PostgreSQL down Azure Portal → PostgreSQL Flexible Server metrics Entity enrichment degrades, search still works in document-only mode. Wait for recovery.

Threshold Calibration

Two sets of thresholds must be calibrated against real data before the system is production-ready. This calibration runs against the full 550K document index — not the pilot data.

Relevance Gate Thresholds

Run 30+ queries spanning all three query types (file_lookup, data_extraction, multi_doc_synthesis). Log all Cohere rerank scores. For each query, manually verify the top-10 results as correct or irrelevant. Analyze the score distributions to determine what percentage of correct results fall in each confidence tier (HIGH ≥0.50 / LOW 0.15–0.50 / NONE <0.15), and what percentage of irrelevant results sneak above the NONE threshold. Adjust no_results_threshold, low_confidence_threshold, and min_rerank_score based on the observed distributions. Document the calibrated values for quarterly re-evaluation as the corpus grows.

BM25 Boost Weights

Run acceptance test query #4 (“What multiple did we use for [deal name]?”) and 2–3 additional synonym-dependent queries (e.g., “pull the comps for Acme,” “what leverage did we use on the Catalyst deal”). Log which chunks surface and their BM25 component scores. Verify that expansion terms contribute to recall (synonym-matched chunks appear in top-20) without pushing original-term matches out. If expansion terms dominate or contribute nothing, adjust BM25_ORIGINAL_BOOST (default 3.0) and BM25_EXPANSION_BOOST (default 1.5).

13.5 How to Verify

How to Verify
  1. All 5 acceptance test queries return reasonable, cited answers. Each query produces a response with relevant content and source citations that link to real SharePoint documents. Query #5 (pipeline overview) organizes results by deal, not by source chunk. No hallucinated answers for any query.
  2. TTFT is within targets for each query type. Across 20+ runs per query type: file_lookup p95 <2s, data_extraction p95 <3s, multi_doc_synthesis p95 <5s. If p95 exceeds the target but p50 is well within it, investigate the tail — it may be cold-start related or a transient API spike.
  3. Permission filtering works. Test with a user who has limited folder access (e.g., only one of the five top-level folders). Query for content that exists in a restricted folder — verify it does not appear in results. Then query for content in the user's accessible folder — verify it does appear.
  4. Document versioning works. Update a file in SharePoint (add a sentence to a Word doc). Wait for the webhook to trigger re-ingestion (should happen within minutes). Query for the new content — verify old chunks are replaced and the new content appears in results.
  5. Reranker degradation and recovery. Block the Cohere API (e.g., via firewall rule or mock). Verify: health check detects failure within 60 seconds, subsequent queries return degraded_reranker: true in the SSE stream, frontend renders orange degraded banner, confidence is capped at LOW. Unblock Cohere. Verify: health check recovers within 60 seconds, next query returns degraded_reranker: false, orange banner disappears. Confirm reranker_degraded event appears in Application Insights.
  6. Full ingestion completes without errors (or with <1% error rate). After the 550K document ingestion, check Application Insights for failed files. A small number of failures is expected (corrupt files, unsupported formats), but the overall error rate should be below 1%. Verify chunk count in Azure AI Search is approximately 2.2M. Verify PostgreSQL contains deal, company, and person rows with relationship entries.

13.6 What Happens Next

With Phase 6 complete, BSearch is live. The system is ingesting documents in near-real-time, answering questions with cited sources, and running within its cost envelope. From here, the path forward is monitoring, calibration, and the v2 feature backlog: Outlook email search, Teams bot interface, document vision, and two-pass extraction. But those are future phases. Right now, all 18 users at Berenson have a system that turns 550,000 documents into answers.

Key Takeaway

Phase 6 is where BSearch goes from “it works in dev” to “it works for real.” The five acceptance test queries prove the system answers the questions Berenson users actually ask. The bulk ingestion of 550K documents is a one-time, 18–36 hour operation costing ~$11–12K. After that, ongoing monthly cost is ~$343–471 for 18 users. The bootstrap deployment sequence ensures everything comes up in the right order, and the blue-green re-indexing strategy means future embedding model swaps happen with zero downtime.
