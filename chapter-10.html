Chapter 10

Implementation Roadmap

The 6-phase build plan, acceptance tests, and what comes after v1.

Building BSearch is not a single monolithic effort. It’s a sequence of six phases, each producing a testable milestone — something you can run, poke at, and verify before moving on. This chapter lays out those phases, the acceptance tests that confirm the system works end-to-end, the features deliberately left out of v1, and the roadmap for what comes next.

IB Analogy

Think of each phase like a workstream in a sell-side process. You don’t try to do everything at once — you run the data room setup, then the marketing materials, then buyer outreach, then management presentations, then final bids. Each workstream has a deliverable that gates the next one. Same idea here.

10.3 The 6 Phases

Phase 1: Project Scaffold + Infrastructure

Human: ~2 days  |  Claude Code: ~20 min

What gets built:

  • Monorepo structure — the folder layout for the entire project, with separate directories for the API, web app, infrastructure templates, and shared code
  • Bicep templates — infrastructure-as-code files that define every Azure resource (search index, function app, Cosmos DB, etc.) so environments can be provisioned with a single command
  • Development environment — Python virtual environment, dependency management, linting rules
  • Config system — centralized settings using Pydantic BaseSettings (a Python library that validates configuration values and loads them from environment variables)
  • Data models — Python type definitions for documents, chunks, entities, and other core objects
  • Graph API client — the module for talking to Microsoft’s SharePoint API
  • Neo4j Aura provisioning — setting up the hosted graph database instance

Testable output: Run az deployment group create and all Azure resources provision successfully. The project structure exists, dependencies install, and the config system loads settings.

Key risks: Azure subscription permissions and quota limits. If your subscription doesn’t have the right resource providers registered, provisioning will fail with obscure error messages.

Phase 2: Format-Aware Parsers + Chunking

Human: ~1 week  |  Claude Code: ~30 min

What gets built:

  • All 4 parsers — Word, Excel, PowerPoint, and PDF (see Chapter 4 for details on each)
  • Table splitter — hierarchical row-group detection that finds smart split points in large financial tables
  • Document summary generation — Haiku-powered summaries for long documents
  • Deal summary generation — Haiku + Neo4j entity data to create deal-level overviews
  • Metadata extraction — document type classification, deal identification, entity extraction
  • Neo4j client + entity registry + graph cleanup — the code that writes entities to the knowledge graph and handles deduplication
  • Embedding provider abstraction — a clean interface around the Voyage AI embedding model, designed so the provider can be swapped later without changing the rest of the code

Testable output: Feed a real CIM through the pipeline. Verify that chunks look right — narrative sections are properly bounded, tables split at logical row groups, slide chunks carry their titles. Inspect the Neo4j graph and see entities with correct relationships.

Key risks: Edge cases in real documents. The parsers will work perfectly on clean test files and then encounter a 2018 CIM with nested tables inside text boxes inside grouped shapes. Budget time for iterating on real-world documents.

Phase 3: Search Index + Ingestion Orchestrator

Human: ~1 week  |  Claude Code: ~30 min

What gets built:

  • Azure AI Search index — the full schema with all fields, vector configuration, scoring profiles, and synonym maps
  • Durable Functions orchestrator — the fan-out/fan-in workflow engine that processes batches of documents in parallel (fan out to parse each file, fan in to collect results and push to the index)
  • Webhook receiver + subscription management — the endpoint that SharePoint calls when files change, plus the background job that keeps webhook subscriptions alive
  • Delta sync with Cosmos DB tokens — the periodic catch-up mechanism that queries SharePoint for changes since the last known state
  • Near-duplicate dedup — filename normalization + embedding similarity to catch “v3 FINAL FINAL” copies
  • Blue-green re-indexing orchestrator — the ability to build a new index alongside the live one and swap them atomically, so re-indexing never causes downtime
Fan-out/fan-in: A pattern where a single job spawns many parallel sub-jobs (fan out) and then waits for all of them to finish before continuing (fan in). Like assigning 10 analysts to each comp a different company, then collecting all the comp sheets to build the final page.

Testable output: Ingest 100 real documents. Query Azure AI Search directly and see results with correct metadata, embeddings, and entity fields.

Key risks: Webhook reliability and Graph API rate limits. Microsoft throttles API calls per tenant, and webhook notifications can arrive out of order or be delayed. The delta sync fallback exists precisely because webhooks alone aren’t 100% reliable.

Phase 4: Query Pipeline

Human: ~1 week  |  Claude Code: ~30 min

What gets built:

  • FastAPI app with SSE endpoint — the API server that receives search queries and streams answers back in real time (SSE = Server-Sent Events, the technology that lets the answer appear word by word)
  • Query classifier — regex patterns + Haiku to determine query type (file lookup, data extraction, precedent search, or multi-doc synthesis)
  • Query expander — static synonyms + Haiku + Neo4j aliases to broaden queries (so “multiple” also searches for “EV/EBITDA”)
  • Search client — hybrid search (keyword + vector) with entity-based filters
  • Reranker + health monitoring — the cross-encoder model that re-scores results for relevance, with automatic fallback if the reranker service degrades
  • Claude synthesis — three prompt strategies tailored to different query types
  • Auth — Azure AD token validation and SharePoint folder-level permission checks

Testable output: Ask a question via the API. Get a streaming answer with citations that link back to specific chunks and source documents.

Key risks: Latency. The query pipeline chains multiple API calls (classify, expand, search, rerank, synthesize), and each adds latency. If any step is slow, the user stares at a blank screen. Time-to-first-token (how quickly the first word of the answer appears) is the critical metric.

Phase 5: Web App

Human: ~1 week  |  Claude Code: ~30 min

What gets built:

  • Vite + React SPA — a single-page web application using modern JavaScript tooling
  • Search bar → SSE stream → progressive markdown rendering — the user types a question, the answer streams in word by word, and formatting (bold, bullet points, tables) renders as it arrives
  • Confidence banners — yellow banner for LOW confidence answers, orange banner when the reranker is degraded
  • Deep search indicator — visual feedback for multi-document queries that take longer to process
  • Source cards with SharePoint deep links — clickable citations that open the source document directly in SharePoint
  • Conversation history — past searches are saved and browsable
  • Azure AD auth via Static Web Apps — single sign-on using the same firm credentials everyone already has

Testable output: Open a browser, navigate to the app, search for something, and see a streaming answer with citations and source links. Click a source card and land on the right file in SharePoint.

Key risks: SSE streaming edge cases in browsers. Some corporate proxy configurations can buffer SSE streams, making the “word by word” effect appear as a delayed dump. Worth testing from the actual office network early.

Phase 6: Integration + Acceptance Testing

Human: ~3 days  |  Claude Code: ~20 min

What gets built:

  • Full Azure deployment — deploy everything to production infrastructure
  • Relevance gate calibration — run 30+ queries, analyze score distributions, and set the thresholds that determine when BSearch says “I don’t know” vs. returning low-confidence results
  • BM25 boost weight calibration — tune how much keyword match scores are weighted vs. vector similarity scores
  • 5 acceptance test queries — the final validation suite (detailed below)
  • Scale-up to full document set — index all ~550,000 documents and verify performance at scale

Testable output: All 5 acceptance queries return correct answers. The system handles the full 550K document corpus without performance degradation.

Key risks: Threshold calibration is empirical, not theoretical. You have to look at real score distributions across dozens of queries to find the right cutoffs. This is more art than science — expect a few iterations.

10.4 The 5 Acceptance Test Queries

These five queries are the final exam. Each one exercises a different capability, and together they validate that the entire system — from ingestion to synthesis — works correctly. If all five return good answers, v1 is ready to ship.

# Query Type Expected Result What It Validates
1 “Where’s the latest GovTech deck?” File lookup Latest version with SharePoint link Regex fast path, BM25 on file_name, recency boost
2 “Which firms did we reach out to for Govini?” Data extraction Firm names with source citations Entity extraction, precise synthesis
3 “Find me a good situation overview slide” Precedent search Slide-level result from historical deck Slide chunking, cross-deal search
4 “What multiple did we use for [deal]?” Data extraction EV/EBITDA value from model/summary Synonym expansion (“multiple” → “EV/EBITDA”)
5 “Where do we stand on active deals?” Multi-doc synthesis Deal-by-deal overview with citations Deal summaries, extended thinking, deep search UX
Why These 5?

Each query maps to a real pattern of how bankers search for information. Query 1 is the most common (“where’s that file?”). Query 5 is the most demanding (“synthesize across multiple deals”). Together they cover the full spectrum from simple file retrieval to complex multi-document analysis, and they exercise every major subsystem: the regex classifier, keyword search, vector search, entity extraction, synonym expansion, the reranker, and Claude synthesis.

10.5 What’s NOT in v1

Scope discipline matters. These features were considered and deliberately deferred. Each one has a clear rationale for why it’s not worth the complexity in the initial release.

Item Rationale
Email ingestion File search alone solves the core pain points. Emails add significant parsing complexity (threads, attachments, forwarding chains) for lower marginal value.
Teams bot (@BSearch) The web app is the primary interface for v1. A Teams bot is a nice-to-have, not a must-have.
Claude Vision (image/chart OCR) Text content covers the vast majority of IB document types. Charts and images are rarely the thing people search for.
Excel model summaries Basic sheet-level chunking first. Deep model comprehension is a v2 feature.
Full nightly reconciliation Webhooks + delta sync cover most cases. A full reconciliation job (comparing every file in SharePoint against every entry in the index) is an insurance policy, not a launch requirement.
Multi-tenant / external users BSearch is an internal firm tool. No need for tenant isolation or external access controls.
Mobile-optimized UI Desktop-first for IB workflows. Bankers search for documents at their desks, not on their phones.
CI/CD pipeline Set up after v1 ships. Manual deployment is fine for a single-developer project during initial development.
Rate limiting 18 trusted internal users. No risk of abuse or overload at this scale.
The Scope Rule

Every feature that’s “not in v1” could be added in v2 without rearchitecting the system. The design accounts for these future capabilities — but building them before the core search experience works would be premature optimization.

10.6 v2+ Roadmap

Once v1 is live and the team is using it daily, these are the highest-value additions — roughly in priority order:

  • Full nightly reconciliation job — a scheduled sweep that compares every file in SharePoint against the search index, catching anything webhooks and delta sync missed. The ultimate safety net.
  • Adaptive model routing — use Haiku for simple queries (file lookups, quick extractions) and Sonnet or Opus for complex ones (multi-doc synthesis, nuanced analysis). Saves cost on easy questions, improves quality on hard ones.
  • User feedback (thumbs up/down) — let users rate answers so you can identify weak spots and track improvement over time.
  • Multi-query vector embedding — embed the user’s query as multiple vectors (one per sub-question) instead of a single vector, improving recall for complex queries.
  • Sibling chunk retrieval — when a chunk matches a query, also retrieve the chunks immediately before and after it in the original document. This provides more context to the synthesis step without increasing the initial chunk size.
  • Additional embedding providers + migration tooling — the ability to swap in a better embedding model when one becomes available, plus describe-then-embed for charts (generate a text description of a chart, then embed that description).
  • Two-pass extraction for multi-doc synthesis — first pass extracts key facts from each source document, second pass synthesizes across all extracted facts. Better for questions that span many documents.
  • Deal summary active injection at query time — when a query mentions a specific deal, automatically include that deal’s summary in the context even if it didn’t rank highly in search results. Guarantees the synthesis has baseline deal context.

Why the Human Time Is So Much Longer

Claude Code generates code fast. But building a production system isn’t just code generation. You need to review every file it produces, test against real documents, debug edge cases, make design decisions, configure Azure resources, and iterate on search quality. The human time is where judgment lives — the Claude Code time is where keystrokes live.

Key Takeaway

BSearch is built in six phases over ~5–6 weeks, each producing a testable milestone. The phases are strictly sequential — each one depends on the previous one’s output. Five acceptance test queries validate the complete system from file lookup to multi-document synthesis. The v1 scope is deliberately narrow: solve the core search problem for 18 internal users first, then layer on email ingestion, model routing, and advanced features in v2.
