Chapter 10

Phase 3 — Search Index & Ingestion Pipeline

The Azure AI Search index, the Durable Functions orchestrator that fills it, webhooks, delta sync, and near-duplicate detection.

10.1 What You’re Building

This phase connects the parsers you built in Phase 2 to Azure AI Search and makes the index stay current as files change. You are building three things: the search index itself (the schema, vector configuration, and scoring profiles), the orchestrator that processes documents through the full pipeline (fetch, parse, chunk, embed, write), and the infrastructure that keeps the index in sync with SharePoint (webhooks, delta sync, and near-duplicate detection).

The orchestrator is the most complex piece. It uses Azure Durable Functions — a framework that lets you write multi-step workflows with built-in checkpointing and retry logic. A single orchestrator handles both the initial bulk load (all 550,000 documents) and ongoing incremental sync (individual file changes triggered by webhooks). Same code path, same retry behavior, same error handling.

By the end of this phase, you can ingest a batch of documents, query them in Azure AI Search, see their entities in PostgreSQL, and watch a file change in SharePoint automatically propagate through the pipeline into the index. Near-duplicate detection ensures that “Deal Summary v3 FINAL (2).docx” does not clutter search results alongside “Deal Summary v3 FINAL.docx.”

10.2 Before You Start

Prerequisites
  • Azure AI Search (S1) provisioned via Phase 1 Bicep templates
  • Cosmos DB (serverless) provisioned via Phase 1 Bicep
  • PostgreSQL (Azure Flexible Server) provisioned via Phase 1 Bicep templates
  • Phase 2 parsers working and tested — Word, Excel, PowerPoint, and PDF parsers produce correct chunks from real documents
  • Embedding provider abstraction implemented (Phase 2) — EmbeddingProvider interface, Voyage AI implementation, factory function
  • Graph API client working (Phase 1) — can authenticate and list files from SharePoint drives
Decisions to Confirm
  • Dedup cosine threshold: 0.95 — two documents must have first-chunk cosine similarity at or above this value to be considered duplicates. Too low and you merge genuinely different files; too high and you miss real duplicates. 0.95 is calibrated in Phase 6 against real data
  • Recency boost parameters: 1.5x for documents modified within the last 30 days — this controls how much newer documents are favored in search results. The boost decays linearly from 1.5x at day 0 to 1.0x at day 30

10.3 The Spec

Azure AI Search Index Schema

The index has 25+ fields organized into seven categories. Every chunk of every document gets all of these fields populated at ingestion time. The schema is designed for hybrid search (keyword + vector), entity-based filtering, and recency scoring — all at zero additional query-time cost.

Core Fields

Field Type Attributes Purpose
chunk_id Edm.String key Primary key. Deterministic: {file_id}_{chunk_index} for idempotent upserts
file_id Edm.String filterable Graph API item ID. Used for atomic delete-all-chunks-by-file on updates
file_name Edm.String searchable, filterable Original filename. BM25-searchable for file lookup queries
file_type Edm.String filterable Extension: docx, xlsx, pptx, pdf
source_folder Edm.String filterable Top-level SharePoint folder. Used for permission filtering at query time
m365_url Edm.String — SharePoint deep link. Shown in source cards so users can open the original file
last_modified Edm.DateTimeOffset filterable, sortable File last-modified timestamp. Drives the recency scoring profile

Content Fields

Field Type Attributes Purpose
content Edm.String searchable The chunk text. BM25-indexed for keyword search
chunk_index Edm.Int32 filterable, sortable Position within the document (0-indexed). Used for ordering and sibling retrieval
chunk_type Edm.String filterable One of: content, table, doc_summary, deal_summary. Controls which chunks are used for different query types
doc_section Edm.String — Section heading context (e.g., “Company Overview > Financial Highlights”)

Metadata & Classification Fields

Field Type Attributes Purpose
doc_type Edm.String filterable Document classification: cim, teaser, model, engagement_letter, nda, management_presentation, diligence_tracker, board_deck, process_letter, ioi, loi, purchase_agreement, memo, other

Deal Context Fields

Field Type Attributes Purpose
deal_id Edm.String filterable Deterministic hash of the deal subfolder path. Null for non-deal files
deal_name Edm.String filterable, searchable Canonical deal name from PostgreSQL. Same for all chunks in the deal
deal_aliases Edm.String searchable Space-separated name variants (code name + real name). BM25-searchable so “Project Atlas” finds “Acme Corp” chunks
deal_status Edm.String filterable One of: active, closed, on_hold, dead. Inferred from deal activity recency
deal_stage Edm.String filterable Inferred from doc_type: origination, marketing, bidding, negotiation, diligence, closing

Entity Fields

Field Type Attributes Scope
companies Collection(Edm.String) filterable, searchable Chunk-scoped — company names mentioned in this specific chunk (from NER)
company_roles Collection(Edm.String) filterable, searchable Document-scoped — “advisor:Goldman Sachs” format. Same for all chunks in the document
people Collection(Edm.String) filterable, searchable Chunk-scoped — person names mentioned in this specific chunk (from NER)
industries Collection(Edm.String) filterable, searchable Deal-scoped — from PostgreSQL deals row. Same for all chunks in the deal
Why Two Scopes for Entity Fields?

companies and people vary per chunk — they tell you what entities are mentioned in this specific passage. This powers queries like “which chunks mention Goldman?” The deal-scoped fields (company_roles, industries, deal_status, deal_stage) are the same for every chunk in a deal — they power filter queries like “chunks from deals where Goldman was advisor” or “chunks from healthcare deals.”

Dedup Fields

Field Type Attributes Purpose
duplicate_group_id Edm.String filterable, nullable Shared identifier for all files in a confirmed duplicate cluster. Null if no duplicates
is_canonical Edm.Boolean filterable (default: true) Only the most recent version in a duplicate cluster is true. All search queries filter on is_canonical eq true

Embedding Provenance Fields

Field Type Attributes Purpose
embedding_model Edm.String filterable Which model produced this vector, e.g., “voyage/voyage-context-3”
embedding_dimensions Edm.Int32 filterable Vector dimension count (1024). Required for blue-green re-indexing
embedding_version Edm.String filterable Config-driven tag (e.g., “v1”). Incremented on any model/provider/dimension change

Embedding provenance — Metadata stamped on every chunk recording which embedding model produced its vector, at what dimensions, and under which version tag. These fields are never used in search scoring — they exist so the blue-green re-indexing orchestrator can identify stale chunks and so operators can monitor for mixed-model drift.

Vector Search Configuration

Setting Value Notes
Algorithm HNSW Hierarchical Navigable Small World — the standard algorithm for approximate nearest-neighbor search. Fast at query time, builds the index incrementally
Dimensions 1024 Matches voyage-context-3 output. Parameterized in create_index() so blue-green re-indexing can create indexes with different dimensions
Distance metric Cosine Standard for text embeddings

Scoring Profile: recency_boost

A freshness function on the last_modified field that boosts recently modified documents in search results:

Parameter Value Effect
Boost factor 1.5 Maximum multiplier applied to BM25 score for the freshest documents
Boosting duration P30D (30 days) Boost decays linearly from 1.5x at day 0 to 1.0x at day 30. Documents older than 30 days get no boost
IB Analogy

The recency boost is like the instinct to check the most recently modified version of a model first. When someone asks “Where’s the latest GovTech deck?”, the word “latest” is implicit in almost every IB file search. The scoring profile bakes that instinct into the ranking algorithm.

Durable Functions Orchestrator

Azure Durable Functions — A framework for writing multi-step, long-running workflows in Azure Functions. It provides automatic checkpointing (if the process crashes at step 4 of 7, it resumes at step 4, not step 1), fan-out/fan-in parallelism (process 50 files simultaneously, then collect all results), and retry policies with exponential backoff.

The orchestrator is the central pipeline that takes a list of file changes (from webhooks, delta sync, or manual bulk trigger) and processes each file through the full ingestion chain. The same orchestrator handles both the initial bulk load of 550,000 documents and ongoing incremental updates from a single file change.

Activity Chain (per file)

Each file flows through this sequence of activity functions. Durable Functions checkpoints after each step — if any step fails, the orchestrator retries that step with exponential backoff rather than restarting the entire file.

  1. Fetch file — Download from SharePoint via Graph API
  2. Parse — Format-aware parser (Word, Excel, PPT, or PDF via Azure Document Intelligence)
  3. Extract metadata — Determine doc_type, compute deal_id from folder path, extract per-file deal_name, run Tier 1 entity extraction (companies, roles, industry, deal stage)
  4. Write to PostgreSQL — UPSERT deal row (aliases, canonical_name, industries), UPSERT company rows, UPSERT relationship rows with source_file_ids JSONB provenance, update deal status, insert stage_entries row if applicable
  5. Chunk — Split into content, table, and slide chunks using Phase 2 chunking logic. Generate doc summary if chunk count is above threshold
  6. Entity extraction (per chunk) — Tier 3 NER on every chunk (company/person mentions). Tier 2 Haiku extraction on high-value chunks only (financial tables, structured people sections, IOI/LOI body)
  7. Embed — All chunks (including summary) embedded via Voyage AI voyage-context-3 with full document text as context. Stamp embedding provenance metadata from provider instance
  8. Check dedup — Run near-duplicate detection, update canonical flags across the cluster
  9. Write to Azure AI Search — Upsert all chunks with complete field population
  10. Queue deal summary update — If file has a deal_id, queue a debounced deal summary regeneration

Fan-Out/Fan-In for Bulk Ingestion

Fan-out/fan-in — A pattern where a single job spawns many parallel sub-jobs (fan out) and then waits for all of them to finish before continuing (fan in). Like assigning 10 analysts to each comp a different company, then collecting all the comp sheets to build the final page.

The orchestrator processes files in batches. Configurable concurrency caps prevent overwhelming external APIs:

Mode Max Concurrent Files Use Case
Bulk ingestion 50 Initial load of all 550K documents. Conservative to avoid Graph API throttling
Delta sync 200 Processing a batch of changed files. Higher concurrency because batches are smaller

File Update and Deletion

When a file is updated or deleted, the orchestrator runs a three-step cleanup before (re-)ingesting:

  1. dedup.handle_file_deletion(file_id) — remove from duplicate cluster, promote next-most-recent if deleted file was canonical
  2. entity_cleanup.cleanup_file_from_entities(file_id, deal_id) — remove source_file_id from PostgreSQL relationship rows, delete rows with no remaining provenance, delete stage_entries rows from this file, recompute deal status
  3. indexer.delete_by_file_id(file_id) — atomically delete all chunks for this file from Azure AI Search

For updates, the normal ingestion pipeline then runs on the new file content.

Rate Limiting

Three layers prevent external API overload during bulk operations:

Layer Mechanism Limits
Orchestrator batch size Durable Functions fan-out batch 50 concurrent files (bulk), 200 (delta)
Per-provider semaphores asyncio.Semaphore per API Haiku: 80, Voyage AI: 50, Graph API: 20, Cohere: 20
Adaptive backpressure 429 rate monitoring If 429 rate exceeds 5% over 1 minute: halve that provider’s semaphore. Restore after 5 minutes of zero 429s

PostgreSQL Dual-Write

Every file ingestion writes to two data stores: PostgreSQL (entity relationships) and Azure AI Search (document chunks). The write order matters for consistency.

Write Order: PostgreSQL First, Then Search

PostgreSQL is the source of truth for entity data. Chunks in Azure AI Search carry denormalized copies of entity fields (deal_name, companies, industries, etc.) copied from PostgreSQL at ingestion time. Writing PostgreSQL first ensures entity fields on chunks are always current.

Failure Scenario Recovery
PostgreSQL write fails Write chunks to Azure AI Search without entity fields. Queue file in Cosmos DB retry_queue with exponential backoff (1min, 5min, 30min, 2hr). On retry: write to PostgreSQL, then patch chunk entity fields in Search. After 5 failures: alert + manual intervention
Azure AI Search write fails PostgreSQL already has entity data. Queue chunk for retry. On retry: write chunks with entity fields populated from PostgreSQL

Voyage AI Embedding

BSearch uses voyage-context-3 at 1024 dimensions for all embeddings. The defining feature of this model is contextual embedding: each chunk is embedded alongside the full document text, so the model understands what the chunk means in context rather than treating it as an isolated fragment.

Contextual embeddings — An embedding approach where each chunk is embedded with awareness of its surrounding document. A chunk saying “Revenue grew 15%” gets a different vector depending on whether the full document is an Acme Corp CIM or a general market report. This solves the fundamental chunking problem where isolated chunks lose meaning.

Setting Value Notes
Model voyage-context-3 +14.24% chunk retrieval improvement over OpenAI text-embedding-3-large
Dimensions 1024 Scalable to 2048 via MRL. 32K token context window
Ingestion Batch embed with document context Groups chunks by file, passes full document text. No cache, no timeout — retries handled by Durable Functions
Query Single query embed with LRU cache ~1000 entries, version-prefixed keys for automatic invalidation on model change. 500ms hard timeout, BM25-only fallback on failure

Provider Abstraction Layer

All embedding calls go through an abstract EmbeddingProvider interface. The interface accepts a document_context parameter — contextual models use it, standard models ignore it. A factory function reads the EMBEDDING_PROVIDER environment variable and returns the correct implementation. This means switching to a different embedding model is a config change + re-index, not a code rewrite.

Interface Method Signature Used By
embed_chunks (chunks, document_context?, input_type?) → list[list[float]] Ingestion pipeline
embed_query (query) → list[float] Query pipeline (with LRU cache)
model_name property → str Embedding provenance metadata, e.g., “voyage/voyage-context-3”
dimensions property → int Embedding provenance metadata, vector search config

Near-Duplicate Detection

IB folders routinely contain files like “Deal Summary v3 FINAL.docx”, “Deal Summary v3 FINAL (2).docx”, and “Deal Summary v3 FINAL revised.docx” — different file IDs with nearly identical content. Without dedup, search results are cluttered with near-identical chunks that push genuinely different content out of the top results.

Detection happens in two phases at ingestion time. Zero cost at query time — search queries simply add is_canonical eq true as a filter predicate.

Phase 1: Filename Normalization + Clustering

Normalize filenames by stripping IB version noise, then group by (normalized_name, source_folder). If a group has more than one file, it is a candidate duplicate cluster.

Normalization rules, applied in order:

  1. Lowercase
  2. Remove file extension
  3. Strip version patterns: v\d+, version\s*\d+
  4. Strip finality markers: final, revised, draft, clean, markup, redline
  5. Strip copy markers: \(\d+\), copy\s*\d*, - copy
  6. Strip trailing numbers after separator: [_\-\s]\d+$
  7. Collapse whitespace/underscores/hyphens to single space
  8. Trim
Original Filename Normalized
Deal Summary v3 FINAL.docx deal summary
Deal Summary v3 FINAL (2).docx deal summary
Deal Summary v3 FINAL revised.docx deal summary
Catalyst CIM Draft 2.pdf catalyst cim
LBO Model - Clean.xlsx lbo model
Same-Folder Constraint

Only files in the same source_folder are compared. This prevents false matches between intentionally different versions — a buyer model in one deal folder and a seller model in another, or a template in Templates and a filled-out version in Active Deals.

Phase 2: Cosine Similarity Verification

For each candidate cluster (files with the same normalized name and folder), compare first-chunk embeddings via cosine similarity. These embeddings are already computed during ingestion — no additional Voyage AI calls.

Cosine Similarity Result
≥ 0.95 (configurable) Confirmed duplicate — files are added to the cluster
< 0.95 Genuinely different files despite similar names — not grouped

This handles the case where “Deal Summary v3” has a completely rewritten executive summary compared to “Deal Summary v2” — similar name but different content.

Canonical Selection

  • Most recent last_modified is canonical (is_canonical: true)
  • All other files in the cluster are non-canonical (is_canonical: false)
  • On tie: largest file by byte count (more complete version)
  • All files in a cluster share the same duplicate_group_id — a deterministic SHA-256 hash of (normalized_name, source_folder)
  • Non-canonical chunks remain indexed for re-evaluation but are excluded from all search queries

When Dedup Runs

Trigger Action
New file ingested Check if (normalized_name, source_folder) matches existing files. If yes, run cosine similarity on first chunks. Update cluster membership and canonical flags
File updated Re-run dedup for the file’s cluster. Updated file may become new canonical if it has the newest last_modified
File deleted Remove from cluster. If deleted file was canonical, promote next-most-recent. If cluster drops to 1 file, dissolve cluster (duplicate_group_id = null, is_canonical = true)

Cluster State in Cosmos DB

Duplicate cluster state is stored in the duplicate_clusters container (partition key: source_folder). Each document stores the normalized name, source folder, and a list of member files with their file_id, last_modified, first-chunk embedding, and is_canonical flag.

Concurrent file ingestions racing on the same cluster are handled via optimistic concurrency — all Cosmos DB writes use ETags. On a 409 Conflict, the orchestrator does a read-merge-retry with a maximum of 3 attempts. Durable Functions retry covers persistent failures beyond 3 attempts.

Known Limitation

Dedup uses first-chunk embedding similarity only. Files with different first chunks but identical subsequent content will not be detected. This is an acceptable tradeoff — filename clustering catches the 90%+ case, and the threshold is calibrated against real data in Phase 6. If false-negative rates prove significant, multi-chunk comparison is a v2 extension.

Webhook Receiver

Webhooks are the primary sync mechanism for keeping the index current with SharePoint. When a file is created, updated, or deleted, Microsoft’s Graph API sends a notification to BSearch within 30–120 seconds. This reduces index staleness from hours (timer-based polling) to minutes.

How It Works

  1. Subscription creation. On deploy, BSearch creates one webhook subscription per monitored SharePoint drive via the Graph API. Each subscription specifies the notification URL (/api/webhooks/graph), a random clientState secret, and an expiration time (~2.95 days max).
  2. Validation handshake. When creating a subscription, Graph sends a GET request with a validationToken. The webhook receiver echoes it back as plain text with a 200 response. This proves BSearch controls the endpoint.
  3. Change notification. When a file changes, Graph sends a POST with the resource path and the clientState. The receiver validates clientState against the stored value in Cosmos DB, responds with 202 immediately, and starts the Durable Functions delta sync orchestrator for the affected drive.
The 3-Second Rule

The webhook receiver must respond to Graph API within 3 seconds. If it takes longer, Graph will retry (up to 4 times), and if it persistently times out, Graph disables the subscription. This is why the receiver does no inline processing — it validates, responds 202, and enqueues the work.

Subscription Lifecycle

Event Action
Deploy Create subscriptions for all monitored drives (HTTP trigger for bulk creation)
Every 12 hours Timer trigger checks expiration. Renew subscriptions with <24 hours remaining via Graph API PATCH
Renewal failure Log alert. Delta sync timer covers until resolved
Redeployment (new URL) Delete old subscriptions, create new ones

Subscription state is stored in Cosmos DB: { subscription_id, drive_id, expiration, notification_url, client_state }.

Delta Sync Timer

Delta sync is the fallback mechanism that catches anything webhooks miss — expired subscriptions, network blips, notifications that arrived while the Functions host was recycling. It runs on a timer trigger every 12–24 hours and uses the exact same delta query + orchestrator pipeline as webhook-triggered sync.

How Delta Queries Work

  1. Graph API provides a /drives/{id}/root/delta endpoint that returns all changes since a stored token
  2. BSearch stores the delta token in Cosmos DB (partition key: drive_id)
  3. On each run, BSearch sends the stored token to Graph, receives a list of created/updated/deleted files, and processes them through the orchestrator
  4. Graph returns a new delta token, which BSearch stores for next time

The delta token is shared between webhook-triggered sync and timer-triggered sync. Both paths read the same token from Cosmos DB, process changes, and store the updated token. This means a webhook-triggered sync at 2pm and a timer-triggered sync at 3pm will not duplicate work — the timer run picks up only changes that happened after the webhook run.

Cosmos DB Collections

Phase 3 uses five Cosmos DB containers for pipeline state management:

Container Partition Key Purpose
sync_tokens drive_id Delta query tokens for each monitored drive. Shared between webhook and timer paths
webhook_subscriptions drive_id Graph API subscription state: subscription ID, expiration, clientState, notification URL
retry_queue deal_id Failed items queued for retry with exponential backoff. Used when PostgreSQL or Search writes fail
duplicate_clusters source_folder Dedup cluster membership, canonical flags, first-chunk embeddings. ETag-based optimistic concurrency
deal_summary_pending deal_id Debounce tracking for deal summary regeneration. Upserts last_file_change_ts on each file change. Timer processes entries where last_file_change_ts < now - 5min
Why Cosmos DB for Debouncing?

Deal summary regeneration is debounced — if 10 files in the same deal change within 5 minutes, the summary is regenerated once, not 10 times. The debounce state lives in Cosmos DB rather than in-memory because Azure Functions workers can recycle at any time. In-memory timers would be lost on recycle; Cosmos DB survives it.

10.5 How to Verify

Verification Checklist
  1. Index creation. Run create_index() — verify the index appears in Azure portal with all 25+ fields, HNSW vector configuration at 1024 dimensions, and the recency_boost scoring profile.
  2. Bulk ingestion. Ingest ~100 test documents through the pipeline. Query Azure AI Search directly and verify all documents appear with correct metadata — file_name, doc_type, deal_id, chunk_type, source_folder, and entity fields all populated.
  3. PostgreSQL entity store. Check PostgreSQL for deal and company rows. Verify relationship rows (TARGET_OF, ADVISOR_ON, etc.) have correct source_file_ids JSONB provenance. Verify canonical_name on deal rows matches the most recently modified file’s deal name.
  4. Near-duplicate detection. Upload two similar files with different names (e.g., “Deal Summary v3 FINAL.docx” and “Deal Summary v3 FINAL (2).docx”) to the same folder. Verify dedup detects them as duplicates: both should have the same duplicate_group_id, and only the most recent should have is_canonical = true.
  5. Webhook sync. Upload a file to SharePoint. Verify it appears in the Azure AI Search index within 2 minutes. Check that the webhook receiver logged the notification and started the delta sync orchestrator.
  6. Cosmos DB state. Verify sync_tokens contains a delta token for each monitored drive. Verify webhook_subscriptions contains an active subscription for each drive with a valid expiration.
  7. File update. Modify an already-indexed file in SharePoint. Verify old chunks are deleted and new chunks appear with updated content. Verify PostgreSQL relationship rows are cleaned up and re-created.
  8. File deletion. Delete a file from SharePoint. Verify all its chunks are removed from Azure AI Search, its relationship rows are cleaned from PostgreSQL, and any duplicate cluster membership is updated.
  9. Embedding provenance. Query chunks in Azure AI Search and verify embedding_model, embedding_dimensions, and embedding_version are all populated correctly.

10.6 Key Risks

Risk Impact Mitigation
Graph API rate limiting Bulk ingestion slows or fails with 429 errors Three-layer rate limiting (concurrency cap, per-provider semaphores, adaptive backpressure). Conservative default: 20 concurrent Graph API calls
Webhook reliability Missed file changes leave the index stale Delta sync timer (every 12–24h) as fallback. Webhook subscription renewal every 12h. Same delta query pipeline handles both paths
Webhook notifications out of order File could be processed before a predecessor change Delta queries are inherently ordered by change sequence. The orchestrator processes the complete delta, not individual notifications
PostgreSQL + Search consistency Entity data in PostgreSQL could disagree with denormalized fields in Search Write PostgreSQL first. Queue retries in Cosmos DB with exponential backoff. Eventually consistent within retry window
Dedup false positives Genuinely different files marked as duplicates Two-phase detection (name clustering + embedding similarity). Same-folder constraint. Cosine threshold calibrated in Phase 6
Bulk ingestion duration Initial load of 550K documents takes 18–36 hours Durable Functions checkpointing allows resume after failure. Do not enable webhooks until bulk load completes
Key Takeaway

Phase 3 connects the parsers to the search index and makes it stay current. The Durable Functions orchestrator is the single code path for both bulk ingestion and incremental sync. Webhooks provide near-real-time updates; delta sync is the safety net. Near-duplicate detection keeps search results clean without any query-time cost. By the end of this phase, you can ingest documents, search for them, and watch file changes propagate automatically.
