Chapter 8

Phase 1 — Project Setup & Infrastructure

The foundation: project structure, configuration, and Azure resources. Nothing user-facing yet, but everything else builds on this.

8.1 What You're Building

This phase creates the foundation that every other phase depends on. You are setting up three things: the code structure (a monorepo where all Python code and infrastructure templates live together), the configuration system (how the application reads API keys, connection strings, and tuning parameters), and the Azure infrastructure (the cloud resources that will host everything).

Nothing in this phase is user-facing. There is no web app, no search, no document parsing. What you get at the end is a working project skeleton where import src.shared.config loads configuration without errors, and an Azure resource group containing all the services BSearch needs — search, compute, database, monitoring, and secrets management. Every subsequent phase plugs into this foundation.

Think of it like setting up the office before the first day of work: desks, phones, internet, security badges, and building access. Nobody is doing deals yet, but without this infrastructure in place, nothing else can start.

8.2 Before You Start

Before You Start
  • Create an Azure subscription (or confirm you have access to one). This is the billing account that all Azure resources will live under. If you already have one for other Berenson infrastructure, you can use the same subscription.
  • Create an Azure AD app registration with three permissions:
    • Sites.Read.All — lets BSearch see the list of SharePoint sites and their folder structure. Without this, BSearch cannot discover where documents live.
    • Files.Read.All — lets BSearch download and read the actual files (Word, Excel, PowerPoint, PDF) from SharePoint and OneDrive. This is read-only — BSearch cannot modify or delete anything.
    • User.Read.All — lets BSearch look up which security groups each user belongs to. This is how the permission system works: when you search, BSearch checks your group memberships to determine which folders you can see.
    All three are application permissions (not delegated), which means they require an Azure AD administrator to click "Grant admin consent" after you add them. These are read-only permissions — BSearch cannot change anything in your M365 tenant.
  • Create a Neo4j Aura account at neo4j.io. The Free tier is sufficient for development (~200K nodes, 400K relationships — enough for ~100 deals). You will upgrade to Pro (~$65/month) for production. After creating an instance, save the connection URI, username, and password.
  • Get API keys for three external services:
    • Anthropic — for Claude (the AI that reads documents and writes answers). You need an API key from console.anthropic.com.
    • Voyage AI — for embeddings (converting text into numbers that capture meaning, which is how semantic search works). API key from dash.voyageai.com.
    • Cohere — for reranking (re-scoring search results by relevance before Claude reads them). API key from dashboard.cohere.com.
  • Create a GitHub repository for the project. This is where all code and infrastructure templates will live.
Keep those credentials safe

You will collect roughly a dozen API keys and connection strings during this phase. Do not put any of them in code files. They will all go into Azure Key Vault (the cloud-based secrets manager), and the application will read them from there at runtime. For local development, they go in a .env file that is listed in .gitignore so it never gets committed.

Configuration System (src/shared/config.py)

Configuration follows a three-tier approach (Architecture Decision #18) to keep the Settings class focused and manageable (~25 keys instead of ~40):

Tier 1 — Module-Level Constants

Algorithm-design values that only change alongside code or prompt changes. Defined as UPPER_SNAKE_CASE constants at the top of the module that uses them. Not in the Settings class.

Constant Value Owning Module
BM25_ORIGINAL_BOOST 3.0 query_expander.py
BM25_EXPANSION_BOOST 1.5 query_expander.py
IB_SYNONYMS_PATH "src/shared/ib_synonyms.json" query_expander.py
FLAT_TABLE_SPLIT_ROWS 25 table_splitter.py
DOC_SUMMARY_MIN_CHUNKS 4 doc_summary.py
DOC_SUMMARY_MAX_TOKENS 200 doc_summary.py
DOC_SUMMARY_CONTENT_PREVIEW_TOKENS 2000 doc_summary.py
DEAL_SUMMARY_MIN_DOCS 3 deal_summary.py
DEAL_SUMMARY_MAX_TOKENS 400 deal_summary.py
MULTI_DOC_THINKING_BUDGET_TOKENS 2000 synthesizer.py

The principle: if changing a value requires also changing code or prompts to be meaningful, it belongs here — not in Settings.

Tier 2 — Removed Entirely

Values whose source of truth is external infrastructure. These do not appear in the Python codebase at all:

Value Where It Lives Instead
TTFT targets (3s / 5s) Documentation and SLA definitions only (DESIGN.md)
Deal summary refresh interval (6h) Azure Functions CRON expression in triggers.py
Embedding fallback alert rate (10%) Application Insights alert rule (defined in Bicep)

Tier 3 — Settings Class (~25 Keys)

The Pydantic BaseSettings class. Connection strings are required (no defaults — the app fails fast if they are missing). Everything else has sensible defaults that can be overridden via environment variables.

class Settings(BaseSettings):
    # --- Infrastructure (required, no defaults) ---
    azure_search_endpoint: str
    azure_search_admin_key: str
    cosmos_connection_string: str
    voyage_api_key: str
    cohere_api_key: str
    anthropic_api_key: str
    graph_client_id: str
    graph_client_secret: str
    graph_tenant_id: str
    neo4j_uri: str
    neo4j_username: str
    neo4j_password: str

    # --- Embedding Pipeline ---
    embedding_provider: str = "voyage"
    embedding_model: str = "voyage-context-3"
    embedding_dimensions: int = 1024
    embedding_version: str = "v1"
    active_index_name: str = "bsearch-v1"

    # --- Relevance Tuning ---
    min_rerank_score: float = 0.10
    low_confidence_threshold: float = 0.50
    no_results_threshold: float = 0.15
    dedup_cosine_threshold: float = 0.95

    # --- Timeouts ---
    classifier_timeout_ms: int = 500
    embedding_timeout_ms: int = 500

    # --- Resilience ---
    reranker_degraded_top_k: int = 12
    reranker_health_check_interval_s: int = 60
    reranker_health_check_timeout_ms: int = 2000
    reranker_consecutive_failures_alert: int = 3
    embedding_health_check_interval_s: int = 60
    embedding_health_check_timeout_ms: int = 500
    neo4j_health_check_interval_s: int = 60
    neo4j_health_check_timeout_ms: int = 2000
    neo4j_health_check_failure_threshold: int = 3
    embedding_cache_max_size: int = 1000
    max_table_chunk_tokens: int = 1500
    deal_summary_debounce_minutes: int = 5

    # --- Operational Flags ---
    reindex_in_progress: bool = False
    reindex_target_index: str | None = None
Why 12 required fields with no defaults?

The 12 infrastructure fields (connection strings, API keys, credentials) have no default values on purpose. If any one of them is missing, Pydantic raises a ValidationError at startup rather than letting the app start in a broken state. This is called "fail fast" — it is much better to get a clear error message on startup than to discover 30 minutes later that your API key is missing when someone tries to search.

Azure Infrastructure (Bicep Templates)

All Azure resources are defined as Bicep templates in infra/. A top-level main.bicep composes individual module files. Parameter files (dev.bicepparam and prod.bicepparam) define environment-specific values like SKU tiers and resource names.

Module Azure Service SKU / Tier Purpose
search.bicep Azure AI Search S1 Hybrid index (vector + BM25) for 550K docs, ~2.2M chunks
functions.bicep Azure Durable Functions Consumption Ingestion orchestrator, webhook receiver, delta sync timer
appservice.bicep Azure App Service B2 (~$26/mo) FastAPI search API — dual-core, 3.5GB RAM, always-on (no cold start)
cosmos.bicep Cosmos DB Serverless Conversations, sync tokens, webhooks, retry queue, duplicate clusters, feedback
staticwebapp.bicep Azure Static Web Apps Free / Standard React SPA hosting with built-in Azure AD authentication
keyvault.bicep Azure Key Vault Standard All API keys and secrets — App Service and Functions read secrets via Key Vault references
appinsights.bicep Application Insights + Log Analytics Workspace Pay-as-you-go Latency instrumentation (p50/p95), health check events, query flight recorder, alerts

Application Insights Setup

The appinsights.bicep module creates two resources: a Log Analytics workspace (where logs are stored and queried) and an Application Insights instance connected to it. This provides:

  • Latency instrumentation — p50 and p95 timing for every pipeline stage (query classification, embedding, search, reranking, Claude synthesis).
  • Health check events — degraded/recovered state transitions for Cohere, Voyage AI, and Neo4j.
  • Query flight recorder — structured logs for every query including classification type, chunk counts, confidence level, and timing breakdown.
  • Alert rules — automated alerts on health check failures (3+ consecutive), embedding fallback rate exceeding 10% over 1 hour, and latency SLA breaches.

Secrets Management

All API keys and credentials are stored in Key Vault. App Service and Azure Functions access them via Key Vault references (@Microsoft.KeyVault(SecretUri=...)) — no secrets appear in app settings or environment variables in Azure. The full set of secrets:

  • Voyage AI API key
  • Cohere API key
  • Anthropic API key
  • Graph API: client_id, client_secret, tenant_id
  • Neo4j: uri, username, password
  • Cosmos DB connection string
  • Azure AI Search admin key

8.5 How to Verify

How to Verify
  1. Config loads without error. Run:
    python -c "from src.shared.config import settings; print(settings)"
    This should either print all settings (if you have a .env file with the required values) or raise a clear ValidationError listing exactly which required fields are missing. Both outcomes confirm the config system is working.
  2. Bicep templates validate. Run:
    az deployment group validate --resource-group bsearch-rg --template-file infra/main.bicep --parameters infra/parameters/dev.bicepparam
    This validates the template syntax and parameter references without actually deploying anything. It should pass with no errors.
  3. Key Vault is accessible. After deploying the infrastructure, verify you can store and retrieve a test secret:
    az keyvault secret set --vault-name bsearch-kv-dev --name test-secret --value "hello"
    az keyvault secret show --vault-name bsearch-kv-dev --name test-secret
    Both commands should succeed. Then delete the test secret.
  4. Application Insights resource appears in the Azure portal. Navigate to the resource group in the Azure portal. You should see both an Application Insights resource and a Log Analytics workspace. Click into Application Insights and confirm it shows "Connected to workspace" with the correct workspace name.
  5. Project structure is correct. Verify the directory tree matches the monorepo layout: src/shared/config.py exists, infra/main.bicep exists, infra/modules/ contains all seven module files, pyproject.toml is at the root, and all __init__.py files are in place.
  6. Dependencies install. Run:
    pip install -e ".[dev]"
    All packages should install without conflicts. Verify with python -c "import fastapi; import anthropic; import neo4j; print('OK')".

8.6 What Happens Next

With the project scaffold and Azure resources in place, Phase 2 builds the document parsers and chunking pipeline. That is where BSearch starts reading actual Word documents, Excel models, PowerPoint decks, and PDFs — turning raw files into the searchable chunks that the rest of the system depends on.

Key Takeaway

Phase 1 produces no user-visible functionality, but it establishes the three foundations everything else depends on: a monorepo with shared configuration (~25 Settings keys organized by purpose), Azure infrastructure defined as code (7 Bicep modules composable via main.bicep), and a secrets management pattern (Key Vault references, no credentials in code). If this phase is done right, every subsequent phase can focus on its own domain without fighting infrastructure issues.
