Chapter 8

Authentication & Permissions

Who can log in, what they can see, and how the system enforces it.

BSearch holds deal documents across five SharePoint folders, some of which contain highly sensitive information. The system needs to answer two questions on every request: who are you? and what are you allowed to see? This chapter explains how both of those questions are answered — without requiring anyone at the firm to create a new username or remember another password.

8.1 Azure AD / Entra ID

Azure Active Directory (Azure AD), recently rebranded to Microsoft Entra ID — Microsoft’s identity system. It is the service that manages everyone’s Microsoft account at Berenson — the same account you use for Outlook, Teams, and SharePoint.

Everyone at the firm already has an Azure AD account. BSearch piggybacks on that existing identity instead of creating its own user database. This means:

  • No new passwords. You sign in with the same Microsoft credentials you already use every day.
  • Single sign-on (SSO). If you are already logged into Outlook or Teams, BSearch recognizes you automatically — no second login prompt.
  • Centralized management. When IT disables someone’s Microsoft account (e.g., they leave the firm), they lose BSearch access instantly. No separate offboarding step.
IB Analogy

Think of Azure AD as the firm’s building security badge. Your badge gets you into the lobby, your floor, and the conference rooms you are authorized for. BSearch is like a new room in the building — it does not issue its own badges; it reads the same badge you already carry.

8.2 How Login Works

OpenID Connect (OIDC) — An industry-standard protocol for authentication. It is the mechanism that lets one application (BSearch) delegate the “who are you?” question to another service (Microsoft). You have used OIDC every time you click “Sign in with Google” or “Sign in with Microsoft” on a website.

JWT (JSON Web Token) — A small, digitally signed data packet that proves who you are. After you log in, Microsoft hands BSearch a JWT — think of it as a tamper-proof digital ID card with your name, email, and group memberships stamped on it.

Here is the login flow, step by step:

  1. You open BSearch in your browser and click “Sign in with Microsoft.”
  2. Your browser redirects to Microsoft’s login page (the same one you see for Outlook). Azure Static Web Apps handles this automatically via its built-in .auth/login/aad endpoint.
  3. You enter your Microsoft credentials (or, if you are already signed in, this step is skipped entirely — that is SSO in action).
  4. Microsoft verifies your identity and redirects you back to BSearch with a JWT token attached.
  5. On every subsequent request (every search, every page load), the BSearch API receives that token and validates it — checking that it was genuinely issued by Microsoft, that it has not expired, and that it has not been tampered with.

If the token is expired or invalid, the API returns a 401 Unauthorized response, and BSearch redirects you back to the Microsoft login page to re-authenticate. In practice, tokens last long enough that you rarely see this — it happens silently in the background.

IB Analogy

The JWT is like a signed engagement letter. It states who the parties are (you and BSearch), it has an expiration date, and it is signed by a trusted authority (Microsoft). If someone forges one or it expires, the deal (your session) does not proceed until a new one is executed.

8.3 Folder Permission Model

BSearch indexes documents from five top-level SharePoint folders. Not everyone at the firm should see every folder. The permission model is simple and deliberate: access is all-or-nothing per folder. If you can see a folder, you can see every document in it. If you cannot, none of those documents will appear in your search results.

A configuration file maps each folder to one or more Azure AD security groups:

{
  "Active Deals": ["group-id-1", "group-id-2"],
  "Closed Deals": ["group-id-1"],
  "Pipeline": ["group-id-3"],
  "Templates": ["all"],
  "Admin": ["group-id-4"]
}

Reading this configuration: “Active Deals” is accessible to members of two AD groups. “Templates” is accessible to everyone ("all"). “Admin” is restricted to a single group.

There is no sub-file permissioning in v1. The system does not check whether you have access to a specific document within a folder — it checks whether you have access to the folder itself. With only 5 folders and 18 users, this level of granularity is sufficient.

Why not file-level permissions?

SharePoint supports fine-grained, file-level permissions, but syncing those permissions into the search index adds significant complexity (see Section 8.6). For a team of 18 people with 5 clearly separated folders, folder-level access is the right trade-off: simple to implement, simple to audit, and simple to explain.

8.4 Query-Time Enforcement

The permission model above would be meaningless if it were only enforced in the user interface. Someone could bypass the front-end and hit the API directly. That is why permissions are enforced at the database level, on every single search query.

Here is what happens every time you search:

  1. Resolve the user. The API reads your JWT to determine who you are.
  2. Look up group memberships. The system checks which Azure AD security groups you belong to.
  3. Determine permitted folders. Using the configuration from Section 8.3, the system identifies which of the 5 folders your groups grant access to.
  4. Add a filter to the search query. Before the query is sent to Azure AI Search, a filter is appended: source_folder IN ["Active Deals", "Templates", ...]. This filter runs inside the search engine itself.

The critical point: unauthorized documents never leave the search index. They are filtered out before the results are scored, before they reach the reranker, and before Claude ever sees them. There is no moment where a restricted document is “fetched and then hidden” — it is never fetched in the first place.

Group membership lookups are cached to avoid calling Microsoft’s API on every search. The cache is invalidated when group memberships change (e.g., someone joins or leaves a deal team).

IB Analogy

This is like an information wall (Chinese wall). When you are restricted from a deal, the compliance team does not just hide the folder from your desktop — the documents are walled off at the server level so you cannot access them through any channel. BSearch’s query-time filtering works the same way: the wall is enforced at the data layer, not the UI layer.

8.5 App Registration

App Registration — A configuration in Azure AD that defines what an application is allowed to do. It is how Microsoft knows that BSearch is a legitimate application and what data it can access.

BSearch needs to read files from SharePoint during the ingestion pipeline (when it pulls documents into the search index). To do this, it uses application-level permissions granted through the Graph API:

  • Files.Read.All — permission to read all files across SharePoint.
  • Sites.Read.All — permission to read site metadata and structure.

These are read-only permissions. BSearch cannot modify, delete, or create files on SharePoint. The permissions also require admin consent — a firm administrator must explicitly approve them before BSearch can access anything.

An important distinction: these application-level permissions are what BSearch uses to ingest documents (pull them into the search index). They are separate from the user-level permissions described in Section 8.3, which control what each person can search. The ingestion pipeline reads everything; the search query filters ensure each user only sees what they are authorized to see.

8.6 What Would Change with Finer Permissions

The current model (5 folders, group-based access) is deliberately simple. If the firm ever needed sub-folder or file-level permissions, here is what would change:

  1. Sync SharePoint permissions to the search index. The ingestion pipeline would need to read the access control list (ACL) for each file in SharePoint and store it alongside the document in the index.
  2. Tag every chunk with an ACL field. Each chunk in the index would carry a list of users and groups permitted to see it.
  3. Query filter checks chunk-level ACLs. Instead of filtering by source_folder, the system would filter by acl CONTAINS user_or_group on every chunk.
  4. Permission sync becomes ongoing. SharePoint permissions can change at any time (someone is added to or removed from a file), so the system would need to detect and propagate those changes to the index.

This is all achievable, but it adds real complexity to both the ingestion pipeline and the search query logic. For v1 — 5 folders, 18 users — it is not needed. It would be the right investment if the firm expanded BSearch to more teams with overlapping, fine-grained access requirements.

Key Takeaway

BSearch does not manage its own identities. It leverages the Microsoft accounts everyone already has, enforces folder-level access through Azure AD group memberships, and applies permission filters at the search-engine level so that unauthorized documents are never retrieved. The model is simple by design — 5 folders, group-based access, enforced at the data layer.
