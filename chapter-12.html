Chapter 12

Phase 5 — Web Frontend

The user-facing web application — what people at Berenson actually interact with.

12.1 What You're Building

This phase builds the thing people actually see: a React single-page application hosted on Azure Static Web Apps. Everything built in Phases 1 through 4 — the parsers, the index, the query pipeline — converges here into an interface that 18 people at Berenson will use every day. The web app connects to the FastAPI backend you built in Phase 4, consumes its SSE stream, and renders answers with inline citations, source cards, and confidence indicators.

The design goal is a clean, fast interface that feels like a natural extension of the search experience. No cluttered navigation, no sidebar, no dashboard widgets. A search bar, an answer, and the sources behind it. When an analyst types a question at 11pm, the interface should get out of the way and deliver the answer as quickly as the pipeline can produce it.

There is one deliberate design choice that runs through every visual decision in this chapter: AI slop mitigation. The answer text renders in Instrument Serif (a serif font that gives AI answers a "written" feel, not a "generated" feel), there are no "AI Answer" labels, and every claim is visibly linked to a source document. The goal is trust through transparency — not through branding the output as artificial.

12.2 Before You Start

Before You Start
  • FastAPI backend running and tested (Phase 4 complete). The /api/search SSE endpoint should stream answers with citations, confidence tiers, follow-up suggestions, and degraded-reranker flags. You will be consuming this stream in the frontend.
  • Azure Static Web Apps provisioned via the Phase 1 Bicep template (infra/modules/staticwebapp.bicep). The resource should exist in your Azure resource group with Azure AD authentication configured.
  • Font choices confirmed:
    • Instrument Serif — for answer text. A serif font that gives AI-generated content a "written" feel rather than a "generated" feel. Available on Google Fonts.
    • JetBrains Mono — for numbers, financial figures, and code-like content (file names, paths). A monospace font with clear number distinction. Available on Google Fonts.
    • System sans-serif stack for UI elements (buttons, labels, navigation).
  • Node.js 18+ installed on your development machine for running Vite and the React dev server.

Why Vite, not Create React App?

Create React App is effectively unmaintained. Vite starts the dev server in under 300ms, has native TypeScript support, and produces smaller production builds. For a project with 18 users, the build tooling choice barely matters — but Vite is what Claude Code will generate the cleanest output with.

Landing Page

The landing page is the first thing users see. It has three elements, centered vertically and horizontally:

  1. BSearch wordmark — the product name rendered in large text. No logo image, just typeset text. Styled in Instrument Serif at a large size.
  2. Pill-shaped search bar — a rounded-rectangle input field, roughly 600px wide, centered below the wordmark. Placeholder text: "Ask anything about Berenson's documents...". On submit (Enter or click), navigates to the results page.
  3. 5 example queries — clickable text chips below the search bar. Each one is a real BSearch query. Clicking any chip populates the search bar and immediately fires the search.

The five example queries match the acceptance test queries from Chapter 1:

# Example Query
1 "Where's the latest GovTech deck?"
2 "Which firms did we reach out to for Govini?"
3 "Find me a good situation overview slide for a banking deal"
4 "What multiple did we use for [deal name]?"
5 "Where do we stand on our active deals?"

Design constraints: no sidebar, no navigation bar, no footer. The landing page is deliberately minimal. A small history icon (clock) and dark mode toggle sit in the top-right corner — unobtrusive but accessible.

Results Page

When the user submits a query, the interface transitions to the results page. The key layout change: the search bar slides from the center of the page to a sticky topbar, freeing the viewport for the answer.

Topbar

The sticky topbar contains the search bar (now in its compact form), the history icon, and the dark mode toggle. The search bar retains its pill shape but is narrower in topbar mode. The user can edit and re-submit a query without returning to the landing page.

Answer Area

The main content area renders the AI-synthesized answer. Key rendering rules:

  • Font: Instrument Serif for answer body text. This is the single most important anti-slop decision — serif text reads as "written" rather than "generated."
  • Streaming: Text appears word-by-word as it arrives from the SSE stream. The cursor (a blinking bar) sits at the end of the currently rendered text.
  • Inline citations: Numbered references like [1], [2], [3] appear inline within the answer text. These are rendered as small, clickable badges.
  • No "AI Answer" label. No badge, no disclaimer, no "Generated by AI" banner. The user knows what BSearch is. Trust their intelligence.
  • No emoji, no markdown artifacts. If Claude's output contains raw markdown symbols (like **bold** that did not parse), strip them. No emoji in synthesized answers.

Citation Popovers

When the user hovers over a citation badge (e.g., [1]), a popover appears showing:

  • Source file name
  • A brief excerpt from the cited chunk (first ~100 characters)
  • "Open in SharePoint" link

The popover is positioned above or below the badge (whichever has more viewport space). It dismisses on mouse-out or Escape. For keyboard users, popovers open on focus and close on blur or Escape.

Confidence Banners

Banners appear between the topbar and the answer area. They communicate search quality to the user:

Confidence Level Banner Color Message
HIGH None (no banner) —
LOW Yellow / amber "Limited results — answer may be incomplete"
NONE Red "No relevant results found"
Reranker degraded Orange "Search quality temporarily reduced"
Banners can stack

A query can trigger both a LOW confidence banner (yellow) and a reranker degraded banner (orange) simultaneously. They are independent signals: LOW means "the documents may not answer your query well," while degraded means "the system that selects the best documents is temporarily unavailable." When both appear, the orange banner renders above the yellow one.

Intent Chip

A subtle chip below the topbar shows the detected query type: file lookup, data extraction, or multi-doc synthesis. This is informational — it helps the user understand why results are structured the way they are (e.g., a file lookup returns a single link rather than a full synthesized answer). Styled as a small, muted pill with lowercase text.

Deep Search Indicator

For multi_doc_synthesis queries, the SSE stream includes deep_search: true in its initial metadata event. The frontend renders a distinct progress state: instead of the standard blinking cursor, a message reads "Searching across multiple documents..." This indicator may be visible for 2–3 seconds before text starts streaming. This is expected — the 5-second TTFT target for multi-doc queries means users need feedback that the system is working, not broken.

Source Cards

Below the answer, source cards list every document that was cited. Each card contains:

Field Description
File name The document title (e.g., Catalyst CIM v3.pdf)
doc_type badge A small colored pill: CIM, model, teaser, engagement letter, etc.
Last modified date Rendered in JetBrains Mono (e.g., Mar 15, 2025)
Deep SharePoint link A clickable "Open in SharePoint" button that uses the m365_url from the search result to open the file directly in SharePoint — not a folder page, the actual file

Source cards are ordered by citation number (matching the inline references in the answer). Each card has a citation number badge in its top-left corner so the user can visually connect [1] in the text to source card #1 below.

Follow-Up Suggestions

Below the source cards, 2–3 follow-up question chips appear. These are parsed from the Claude synthesis response (the <follow_ups> tag). Each chip is clickable — clicking it populates the search bar and fires a new search. Styled as bordered pill buttons that match the example query chips on the landing page.

Feedback Buttons

A thumbs-up and thumbs-down button pair appears next to the answer. Clicking either sends a POST to the /api/feedback endpoint with the conversation_id and the rating. The button that was clicked fills in to confirm the selection. Feedback is stored in the Cosmos DB feedback container (partition key: conversation_id).

Export to .docx

An "Export" button below the feedback buttons generates a Word document containing the answer text and a formatted source list. Uses the docx npm package for client-side generation. The exported document uses Instrument Serif for the answer body (embedded as a font in the .docx) and includes citation references matched to a "Sources" section at the bottom. File name format: BSearch — {first 50 chars of query}.docx.

Mid-Stream Cancel

While the answer is streaming, a "Stop" button appears below the streaming text. Clicking it aborts the SSE connection (calls abort() on the AbortController). The answer rendered so far stays on screen with source cards for any citations already received. The stop button is replaced by the normal feedback buttons once streaming completes or is cancelled.

History Page

The history page shows a chronological list of past conversations. It is accessible from the clock icon in the topbar.

  • Day-grouped: Entries are grouped under date headers — "Today," "Yesterday," or the date (e.g., "March 23, 2026"). Each entry shows the query text and a truncated preview of the answer (first ~80 characters).
  • Infinite scroll: Loads more entries as the user scrolls down. Uses Cosmos DB cursor-based pagination on the conversations container (partition key: user_id), not offset-based pagination. Each page request includes the continuation token from the previous response.
  • Click to re-open: Clicking an entry navigates to the results page and displays the full saved conversation (query, answer, sources). Re-opened conversations show the cached answer — they do not re-run the search.

Cursor-based pagination — Instead of requesting "page 5" (which requires counting through all previous entries), each response includes a token that says "start from here next time." This is faster and more reliable for growing datasets — like using a bookmark in a book instead of counting pages from the beginning.

SSE Stream Consumption

The frontend connects to the FastAPI /api/search endpoint using the browser's native EventSource API (or a fetch + ReadableStream approach if custom headers are needed for auth tokens). The SSE stream sends these event types:

Event Type Payload Frontend Action
metadata { confidence, degraded_reranker, deep_search, intent, conversation_id } Render confidence banners, intent chip, deep search indicator
token { text } Append to answer text, parse inline citations
sources [ { file_name, doc_type, last_modified, m365_url, excerpt } ] Render source cards
follow_ups [ "question 1", "question 2", "question 3" ] Render follow-up suggestion chips
done {} Stop cursor animation, show feedback buttons
error { message } Show error message in the answer area

The useSSEStream hook manages the connection lifecycle: opening the stream on query submit, buffering tokens into a rendered string, parsing citation markers as they arrive, handling mid-stream cancellation, and cleaning up on unmount. Token text is appended to a React state variable using a ref-based approach to avoid re-render storms (update the ref on every token, batch-flush to state on requestAnimationFrame).

Search Bar Transition

The search bar has two modes:

Mode Location Width Context
Landing (centered) Vertically and horizontally centered ~600px Landing page
Topbar (compact) Sticky top bar, left-aligned or centered ~400px Results page, history page

The transition from centered to topbar is animated with a CSS transition (position, width, and top). The animation should take ~300ms with an ease-out curve. The SearchBar component accepts a mode prop ("landing" or "topbar") and renders accordingly.

Dark Mode

The app supports dark mode via two mechanisms:

  1. Auto-detect: On first load, the app checks the user's system preference via prefers-color-scheme media query and applies the matching theme.
  2. Manual toggle: A sun/moon icon button in the topbar lets the user override the system preference. The choice is persisted in localStorage.

Dark mode is implemented with CSS custom properties on the :root element. A data-theme="dark" attribute on <html> switches the property values. Key color mappings:

Property Light Dark
Background #FFFFFF #1A1A1A
Answer text #2B2B2B #E8E8E8
Source card bg #F7F7F7 #2A2A2A
Citation badge #E8EDF2 #3A4555
Confidence: LOW #FFF3CD (amber) #4A3F1F (dark amber)
Confidence: NONE #F8D7DA (red) #4A1F1F (dark red)
Degraded banner #FFE0B2 (orange) #4A351F (dark orange)

The useDarkMode hook encapsulates the detection, toggle, and persistence logic. It returns { isDark, toggle }.

Responsive Layout

The results page has three responsive breakpoints:

Breakpoint Width Layout
Wide >1200px 3-column: history sidebar (left), answer content (center), source panel (right)
Medium 1024–1200px 2-column: answer content (left), source panel (right). History accessible via icon.
Narrow <1024px Single column: everything stacked vertically. Sources below the answer. History via icon.

The wide layout's history sidebar shows the most recent 10–15 past queries (truncated to one line each) with a "View all" link that navigates to the full history page. This sidebar is hidden at medium and narrow breakpoints — history is accessible only via the clock icon.

Use CSS Grid for the multi-column layouts and media queries for breakpoint switching. The landing page is always single-column centered, regardless of viewport width.

Accessibility

The web app must meet WCAG 2.1 AA for the following:

Requirement Implementation
ARIA landmarks role="main" on the answer area, role="search" on the search bar, role="navigation" on the topbar, role="complementary" on the source panel
Keyboard navigation Tab through all interactive elements (search bar, citation badges, source cards, follow-up chips, feedback buttons, dark mode toggle). Enter activates. Escape closes popovers.
Touch targets All clickable elements have a minimum touch target of 44x44px on mobile breakpoints. This includes citation badges, follow-up chips, and feedback buttons.
Citation popovers Screen reader friendly: popovers use aria-describedby linking the badge to the popover content. Announced as "Source 1: [file name], [excerpt]" when focused.
Streaming announcements An aria-live="polite" region announces when the answer starts streaming and when it completes. Individual tokens are not announced (too noisy).

Authentication

Azure Static Web Apps provides built-in Azure AD authentication via OIDC. Configuration lives in staticwebapp.config.json:

{
  "auth": {
    "identityProviders": {
      "azureActiveDirectory": {
        "registration": {
          "openIdIssuer": "https://login.microsoftonline.com/{tenant_id}/v2.0",
          "clientIdSettingName": "AAD_CLIENT_ID",
          "clientSecretSettingName": "AAD_CLIENT_SECRET"
        }
      }
    }
  },
  "routes": [
    { "route": "/api/*", "allowedRoles": ["authenticated"] },
    { "route": "/*", "allowedRoles": ["authenticated"] }
  ],
  "navigationFallback": {
    "rewrite": "/index.html"
  }
}

All routes require authentication. The Static Web Apps platform handles the OIDC flow — no login page needs to be built. Unauthenticated users are automatically redirected to the Microsoft login page. The auth token is forwarded to the FastAPI backend via the x-ms-client-principal header that Static Web Apps injects automatically.

AI Slop Mitigation

Every design choice in this app is made to avoid the look and feel of generic AI-generated output. This is not cosmetic — it directly affects whether senior bankers trust and use the tool.

Anti-Slop Decision Implementation Why It Matters
Serif font for answers Instrument Serif for all answer body text Serif text reads as "written" not "generated." Sans-serif AI output (the ChatGPT look) triggers skepticism in professional settings.
No AI labels No "AI Answer," "Generated by AI," or robot icons anywhere The user typed a question into a search tool. They know the answer is AI-generated. Labeling it insults their intelligence and undermines confidence in the output.
Citation transparency Every claim linked to a numbered source. Sources shown as cards with one-click deep links. Trust comes from verifiability, not from disclaimers. If every fact can be traced to a document, the user can verify anything they want to verify.
Clean typography No emoji in answers. No raw markdown artifacts. Proper typographic quotes and dashes. Emoji and markdown cruft are AI tells. Professional output uses proper typography.
Numbers in monospace Financial figures and dates in JetBrains Mono Monospace numbers are easier to scan in financial context and signal precision. Standard in IB deliverables.

12.5 How to Verify

How to Verify
  1. Landing page renders correctly. Open the dev server (npm run dev). Verify the BSearch wordmark appears centered in Instrument Serif, the pill search bar is present, and all 5 example query chips are visible below it. Click an example chip — verify it populates the search bar and navigates to the results page.
  2. Streaming answer works. Submit a query against the running FastAPI backend. Verify the answer appears word-by-word in Instrument Serif (not all at once). Verify the blinking cursor tracks the end of the text. Verify the cursor disappears and feedback buttons appear when streaming completes.
  3. Citations are interactive. Hover over a citation badge (e.g., [1]) in the answer text. Verify a popover appears showing the source file name and a text excerpt. Click "Open in SharePoint" in the popover — verify it opens the correct file. Verify popover dismisses on mouse-out and Escape.
  4. Source cards render correctly. Below the answer, verify source cards appear with: file name, doc_type badge (colored pill), last modified date in JetBrains Mono, and "Open in SharePoint" button. Click the button — verify it opens the file directly in SharePoint (not a folder listing).
  5. Confidence banners display correctly. Test with the backend returning different confidence levels. Verify: HIGH shows no banner, LOW shows a yellow/amber banner, NONE shows a red banner. Simulate reranker degradation — verify the orange banner appears. Verify both LOW + degraded banners can appear simultaneously.
  6. Dark mode works. Toggle the dark mode switch. Verify background, text, source cards, citation badges, and confidence banners all switch colors. Verify the toggle state persists after page refresh (check localStorage). Set your OS to dark mode, clear localStorage, reload — verify the app auto-detects dark mode.
  7. History page works. Navigate to the history page. Verify past conversations appear grouped by day (Today, Yesterday, specific dates). Scroll to the bottom — verify more entries load (infinite scroll). Click an entry — verify it re-opens the saved conversation without re-running the search.
  8. Responsive layout works. Resize the browser window. Verify: above 1200px, the 3-column layout appears with history sidebar. Between 1024–1200px, the sidebar disappears and you see 2 columns. Below 1024px, everything stacks into a single column. Verify the landing page stays centered at all widths.
  9. Mid-stream cancel works. Start a query, then click the Stop button while streaming. Verify the answer stops where it was, source cards for already-received citations remain visible, and feedback buttons appear.
  10. Accessibility basics. Tab through the landing page — verify focus moves through the search bar, example chips, history icon, and dark mode toggle. Tab through the results page — verify focus reaches citation badges, source cards, follow-up chips, and feedback buttons. Press Escape while a citation popover is open — verify it closes.

12.6 What Happens Next

With the web frontend complete, every layer of the system has been built: infrastructure, parsers, search index, query pipeline, and user interface. Phase 6 brings them all together for integration testing, acceptance test validation, bulk ingestion of the full 550,000-document corpus, and go-live. The five acceptance test queries from Chapter 1 are the final exam.

Key Takeaway

The web frontend is where four phases of backend engineering become a product. The critical design decisions are not about frameworks or component libraries — they are about trust. Instrument Serif instead of sans-serif. Citations instead of disclaimers. Source cards with one-click deep links instead of "trust me" labels. The interface earns credibility by making every claim verifiable, not by telling users to trust it.
