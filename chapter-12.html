Chapter 12

Failure Modes & Resilience

What can go wrong, how the system recovers, and what you’ll see as a user.

No system runs perfectly all of the time. External APIs time out, files arrive corrupted, networks hiccup. The measure of a well-built system is not whether failures happen — it is how the system responds when they do. BSearch is designed around five resilience principles:

  1. Every failure has a handler. Retry, fallback, or explicit error — nothing is silently swallowed.
  2. Fail fast on configuration errors. If the embedding provider is misconfigured, the app refuses to start rather than producing garbage results.
  3. User transparency. When quality is degraded, BSearch tells you with colored banners — it never pretends everything is fine when it is not.
  4. Self-healing. Most transient failures resolve automatically through retries, delta sync, or periodic maintenance jobs.
  5. Bounded degradation. Even in the worst non-catastrophic failure, BSearch still returns results — they are just less refined.
IB Analogy

Think of BSearch’s resilience like a deal team’s backup plan. If the MD is stuck in another meeting, the VP runs the call. If the model is not ready, the team presents from the memo. The deal does not stop — it continues with a slightly leaner team. Similarly, BSearch always tries to give you an answer, even if some of its best tools are temporarily unavailable.

This chapter walks through every failure mode, organized by how much you — the user — would actually notice. Most failures are invisible. A few trigger a gentle banner. Only a handful result in an actual error screen.

12.1 Silent Failures — You Never Notice

These are failures the system handles entirely on its own. You will never see an error message, and in most cases the issue resolves before it could ever affect a search result.

Ingestion-Side Failures

These happen in the background when BSearch is pulling files from SharePoint and processing them into the search index.

What Fails What Happens User Experience Self-Healing?
Graph API download (404, auth error, timeout) Durable Functions retry the download automatically, then log and skip if retries are exhausted One file is not indexed Yes — next delta sync retries
Parser fails (corrupt or unsupported file) try/except catches the error, logs it, skips the file One file is not indexed No — file must be fixed at the source
Voyage AI embedding (during ingestion) Durable Functions retry with exponential backoff Ingestion slows down temporarily Yes — retries succeed once the service recovers
Neo4j deal record lookup (during ingestion) Chunk gets per-file deal_name as fallback, queued for retry Temporarily inconsistent deal name Yes — self-heals on next successful ingestion
Neo4j write failure (during ingestion) Chunks are written without entity fields, queued for retry Entity filtering unavailable for this file Yes — retry restores entity fields
Dedup Cosmos lookup timeout Skip dedup check, file indexed as canonical Temporary duplicate results possible Yes — periodic re-evaluation cleans up
Doc summary generation (Haiku 5s timeout) Skip summary, doc indexed normally Cross-section bridging unavailable for this doc Yes — regenerated on file update
Deal summary generation (Haiku 8s timeout) Skip summary, deal docs still individually searchable Cross-deal aggregation slightly degraded Yes — retried on next 6-hour cycle
Canonical name re-tagging Old deal_name persists on existing chunks Older chunks show previous name Yes — next ingestion triggers retag
Table splitter (no hierarchy detected) Falls back to fixed-size row splits (~25 rows) Slightly less precise table chunking N/A — fallback is adequate

Query-Side Failures

These happen in the few seconds between you pressing “Search” and seeing results.

What Fails What Happens User Experience Self-Healing?
Haiku classifier timeout (>500ms) Defaults to data_extraction intent Slightly less optimal retrieval strategy Yes — per-query, next query may succeed
Voyage AI query embedding timeout (>500ms) Falls back to BM25-only search (skip vector) Slightly lower recall Yes — per-query, cached next time
Cosmos DB write timeout (conversation log) Fire-and-forget with logging One conversation entry lost Yes — next write succeeds
Webhook missed Delta sync catches up within 12–24 hours Brief stale window Yes — delta sync is the safety net
Neo4j query failure (at query time) Skip alias expansion and entity context Search works without entity enrichment Yes — per-query transient
Extended thinking budget exhausted Claude truncates its reasoning, still generates answer May miss some cross-chunk connections N/A — inherent model limit
Why so many “silent” failures?

This is by design. The majority of things that can go wrong are transient (a brief API timeout, a momentary network blip). Showing an error message for something that fixes itself in seconds would just be noise. BSearch logs every failure internally for monitoring, but only surfaces issues to you when they actually affect your experience.

12.2 Graceful Visible Failures — You See a Banner

These failures are significant enough that you should know about them, but not severe enough to stop you from working. BSearch uses colored banners to communicate what is happening.

What Fails What Happens What You See
Cohere rerank API (error or timeout) Skip reranking, use Azure Search’s built-in ranking on the top 12 results, cap confidence at LOW Orange banner: “Search quality may be reduced”
LOW confidence (rerank scores 0.15–0.50) Claude adds hedging language to its synthesis Yellow banner: “Results may not fully answer your query”
Multi-doc synthesis (5s+ time-to-first-token) Streaming starts late but still completes “Searching across multiple documents...” visible for 2–3 seconds before text appears
Relevance gate miscalibrated Valid queries incorrectly get “not found” User sees no results (fixable by adjusting thresholds)
Orange vs. Yellow Banners

Orange means a system component failed and BSearch is running in a degraded mode — results are still useful but may not be as well-ranked. Yellow means the system is working normally but is not confident it found great results for your particular query. Both are informational — you can still trust and use the results.

12.3 Visible Errors — Something Is Clearly Wrong

These are the rare failures where you see an actual error message or are interrupted. They typically mean a core service is down and BSearch cannot work around it.

What Fails What Happens What You See
Azure AI Search down 503 error from the search service “Search unavailable” error message
Claude synthesis failure Return raw search results as source cards Degraded but functional — you see sources without AI synthesis
Azure AD token expired/invalid 401 Unauthorized Brief redirect to Microsoft login page
Embedding provider misconfigured App refuses to start, health check fails App down until configuration is fixed (fail-fast)

Notice that even within visible errors, BSearch tries to degrade gracefully. If Claude fails, you still get the source documents as clickable cards. The system only shows a full error screen when the search index itself is unreachable — because at that point, there is genuinely nothing to show.

12.4 Infrastructure-Level Concerns

These are behind-the-scenes issues that affect the system’s ability to maintain and update the search index. They are invisible to users in the short term but could affect data freshness if left unresolved.

Issue How BSearch Handles It
Index quota exceeded Alert fires via Application Insights. New files are not indexed until storage is freed, but existing searches continue normally.
Re-index orchestrator crash Durable Functions checkpointing resumes from the last completed batch. The old index continues serving searches in the meantime.
Re-index validation failure Blocks the flip to the new index. The old index keeps serving — users never see incomplete data.
Concurrent ingestion during re-index (target write fails) The old index already has the file. The gap is caught by post-flip validation.
Dedup false positive (different files grouped together) Cosine similarity threshold of 0.95 plus same-folder constraint minimizes this risk. Non-canonical chunks remain indexed for periodic re-evaluation.
IB Analogy

The re-index process works like a CIM revision. The team prepares the updated version in a separate workspace. The old version stays live and available to buyers. Only when the new version passes QC does it replace the old one. If the new version has issues, the old version stays in circulation — nobody ever sees a half-finished document.

12.6 The Five Resilience Principles

1. No Silent Data Loss

Every failure is either retried, handled by a fallback, or explicitly logged with an error. Nothing disappears into a void. If a file fails to index, that fact is recorded in Application Insights and the next delta sync will try again.

2. Fail Fast on Configuration Errors

If the embedding provider, API keys, or index connection strings are wrong, BSearch refuses to start. This is intentional. A misconfigured system that appears to work is far more dangerous than one that clearly says “I cannot start — fix this first.” The health check endpoint catches these issues before any user traffic arrives.

3. Transparent Degradation

When BSearch is running in a reduced mode — no reranker, BM25-only, no entity enrichment — it tells you. The orange and yellow banners exist specifically so you can calibrate your trust in the results. Hiding degradation would be worse than showing it.

4. Self-Healing by Default

Look at the “Self-Healing?” column in the tables above: the vast majority say Yes. Durable Functions retry failed downloads. Delta sync catches missed webhooks. The 6-hour summary cycle regenerates failed deal summaries. Most problems fix themselves within minutes to hours without any intervention.

5. Bounded Degradation

Even in the worst realistic scenario — Voyage AI is down, Cohere is down, Neo4j is unreachable — BSearch still returns results. They come from BM25 keyword matching with Azure Search’s default ranking and no entity enrichment. That is a meaningfully worse experience than the full pipeline, but it still works. You still get relevant documents. The floor is never zero.

Key Takeaway

BSearch is designed to bend, not break. The majority of failures are invisible and self-healing. The few that are visible are clearly communicated. And even in the worst case, the system returns results — just less refined ones. Every component has a fallback, and the fallback always produces something useful.
