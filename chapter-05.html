Chapter 5

The Query Pipeline

What happens when you ask BSearch a question — from classification to streaming AI synthesis.

When you type a question into BSearch and press Enter, seven discrete steps execute in roughly one second. Each step narrows the search space, filters out noise, and adds context — so that by the time Claude begins writing its answer, it has exactly the right material to work with, and nothing else.

IB Analogy

Think of a well-run deal process. You start with a universe of hundreds of potential buyers, screen down to a long list, narrow to a short list, then focus your time on the two or three who will actually close. The query pipeline works the same way: start broad, filter aggressively, and spend the expensive resources (Claude) only on the highest-quality matches.

This chapter walks through all seven steps in detail, then follows a concrete example query through the entire pipeline so you can see exactly what happens at each stage.

Notice that Steps 1 and 3 run in parallel — the system does not wait for classification to finish before starting the embedding. This parallelization is what keeps total latency under two seconds even in the worst case.

5.2 Step 1 — Classification

The first thing BSearch does is figure out what kind of question you asked. This matters because a file lookup ("find Report_v3.docx") requires a completely different workflow than a cross-document analysis ("What healthcare deals have we worked on?").

Query classification — The process of categorizing a user's question into one of several predefined types, so the system can choose the right search and synthesis strategy for each type.

Two classification paths

Regex fast path (<5ms). A set of pattern-matching rules catches obvious file lookups before any AI model is involved. If your query contains a filename with an extension ("find Report_v3.docx") or phrases like "where's the GovTech deck", the regex catches it immediately and classifies it as a FILE_LOOKUP. This is the fastest possible path — no network call, no model inference.

Regex — Short for "regular expression." A pattern-matching technique that scans text for specific structures (like filenames ending in .docx or .xlsx). Extremely fast but rigid — it can only catch patterns you explicitly define.

Haiku LLM fallback (~150–250ms). If the regex does not match, BSearch sends the query to Claude Haiku — a small, fast AI model — which classifies it into one of three categories:

  • FILE_LOOKUP — The user wants a specific file. Example: "Where's the GovTech pitch deck?"
  • DATA_EXTRACTION — The user wants a specific data point from one or a few documents. Example: "What was the EBITDA margin for Catalyst?"
  • MULTI_DOC_SYNTHESIS — The user wants an answer that requires reading across multiple documents. Example: "Summarize all healthcare deals from the last two years."

When the query is ambiguous, the system defaults to DATA_EXTRACTION — the middle ground that works well enough for most questions. If Haiku does not respond within 500ms, the system also defaults to DATA_EXTRACTION and moves on.

Entity intent extraction

In the same Haiku call, BSearch also extracts entity intent — structured information embedded in the query. From "What multiple did we use for Catalyst?", Haiku identifies:

  • Company name: Catalyst
  • Financial concept: valuation multiple
  • Role filter: deals where Berenson was the advisor ("we")

These extracted entities are used later (in Step 4) to filter search results precisely.

5.3 Step 2 — Query Expansion

Investment banking terminology is wildly inconsistent. The same concept might be called "multiple," "EV/EBITDA," "trading comp multiple," or "valuation metric" depending on who wrote the document and when. If BSearch only searched for the exact words you typed, it would miss relevant results constantly.

IB Analogy

This is like when you are searching for precedent transactions and you need to check for "acquisition," "merger," "buyout," "take-private," and "business combination" — all of which might describe the same deal in different documents. Query expansion automates that kind of synonym awareness.

Query expansion happens in three layers:

Layer 1: Static IB synonym map (0ms)

A JSON configuration file maps common IB shorthand to its formal equivalents. When BSearch sees "multiple" in your query, it automatically adds "EV/EBITDA" as a search term. "Comp" expands to "comparable companies." "LOI" expands to "letter of intent." This is instantaneous because it is a simple dictionary lookup — no network call required.

Layer 2: Haiku contextual expansion (0ms added)

This runs inside the same Haiku call as classification (Step 1), so it adds zero additional latency. Haiku understands the intent behind your query and generates contextual expansions. For example, "what did we pay for Catalyst" expands to include "purchase price," "enterprise value," and "transaction value" — terms that would not appear in a static synonym map but are clearly relevant.

Layer 3: Entity alias expansion from Neo4j (~5–10ms)

BSearch's knowledge graph stores aliases for companies and people. If you search for "GS," the system expands that to "Goldman Sachs" and "Goldman Sachs & Co." so that documents using the full name are not missed. This requires a quick graph lookup, adding roughly 5–10ms.

Neo4j — A graph database that stores relationships between entities (companies, deals, people) as a web of connections. BSearch uses it to understand that "GS" and "Goldman Sachs" refer to the same entity, or that a company is connected to a specific deal.

How expansion terms are weighted

Expansion is applied to the keyword search (BM25) only — not to the vector search, which already understands semantic similarity on its own. To prevent expansion terms from overwhelming your original query, BSearch uses a weighting system: your original terms are boosted 3x, while expansion terms get a 1.5x boost. This ensures the original query always dominates.

5.4 Step 3 — Embedding

While Steps 1 and 2 handle the keyword side, BSearch simultaneously converts your query into a vector embedding — a list of numbers that captures the meaning of what you asked, not just the words.

Vector embedding — A mathematical representation of text as a list of numbers (typically 1,024 of them). Texts with similar meanings end up with similar numbers, even if they use completely different words. This is what allows BSearch to find a document about "enterprise value" when you searched for "what did we pay."

Voyage AI's voyage-context-3 model handles this conversion, typically in 100–300ms. Two optimizations keep this fast:

  • LRU cache (~1,000 entries). If you (or anyone) recently asked the same or a very similar question, BSearch reuses the cached embedding instantly (0ms). LRU stands for "least recently used" — the cache keeps the 1,000 most recent embeddings and evicts the oldest when full.
  • 500ms hard timeout. If Voyage AI is slow or unreachable, BSearch gives up on vector search entirely and falls back to keyword search (BM25) only. A slightly less precise search is better than a query that hangs.
  • File lookup skip. Queries classified as FILE_LOOKUP skip the embedding step entirely — finding a file by name does not require semantic understanding.

The key insight: Haiku classification and Voyage AI embedding run concurrently using Python's asyncio.gather. The critical path through Steps 1–3 is determined by whichever finishes last, not the sum of both. Entity alias expansion from Neo4j is sequential — it must wait for Haiku to identify the entities before looking up their aliases. Separately, Neo4j entity context retrieval (deal status, financials) runs in parallel with Azure AI Search in Step 4.

5.5 Step 4 — Search

With the expanded query terms and the vector embedding in hand, BSearch sends a hybrid query to Azure AI Search. "Hybrid" means it combines two fundamentally different search strategies:

BM25 (Keyword Matching)

Finds documents containing your exact terms and their expansions. Great for specific names, ticker symbols, and deal codes that vector search might miss.

Vector Similarity

Finds documents with similar meaning, even if they use different words. Great for natural-language questions where you do not know the exact terminology used in the documents.

Azure AI Search merges the scores from both approaches, and also applies a recency scoring profile that gives a slight boost to more recently modified documents — because in IB, newer versions are almost always more relevant.

Filters narrow the search space

Before the search runs, BSearch applies several filters drawn from the entity extraction in Step 1:

  • Source folder (permissions). Users only see results from folders they have access to. This is enforced as a hard filter — no exceptions.
  • Canonical flag (dedup). Only the canonical version of near-duplicate documents is returned, preventing the same CIM draft from appearing five times in results.
  • Entity filters. If Haiku detected industry, company role, deal status, deal stage, or financial constraints in your query, those are applied as filters too. For example, "active healthcare deals" would filter for industry=healthcare and deal_status=active.
  • Neo4j pre-query. For entity-rich queries, BSearch queries the knowledge graph first to resolve entity relationships, then passes those as filters to Azure AI Search. For example, if you search for a company name, Neo4j can identify which deal it belongs to, enabling a precise deal-level filter.

The search returns the top 20 candidates (or top 12 if the reranker is degraded — more on that in Step 5).

5.6 Step 5 — Reranking

Azure AI Search is fast, but its ranking is approximate. To get truly precise relevance ordering, BSearch sends the top 20 results to Cohere's reranking model, which re-scores each result against the original query — not the expanded version.

IB Analogy

This is like the difference between an initial buyer screen (broad criteria, fast) and management's deep assessment of each potential acquirer (slower but much more precise). The search step casts a wide net; the reranker decides which catches are actually worth keeping.

Using the original query is deliberate. Query expansion (Step 2) may have added marginal terms like "comparable companies" that surface loosely related documents. The reranker evaluates each result against what you actually asked, so those marginal matches get dropped.

After reranking, BSearch keeps:

  • Top 5 results for FILE_LOOKUP and DATA_EXTRACTION queries
  • Top 10 results for MULTI_DOC_SYNTHESIS queries (because cross-document answers need more source material)
Fallback

If Cohere's reranking service is unavailable, BSearch skips reranking entirely and uses Azure AI Search's native ranking instead. When this happens, the system caps confidence at LOW (see Step 6) because the results have not been precision-verified. The search result count is also reduced to top 12 to compensate for lower ranking quality.

5.7 Step 6 — Relevance Gating

Before sending anything to Claude, BSearch checks whether the search results are actually relevant. This prevents the system from confidently synthesizing an answer from documents that have nothing to do with your question.

The system uses three confidence tiers based on the top reranker score:

Tier Score Range What Happens
HIGH ≥ 0.50 Strong match. Normal synthesis — Claude generates the answer with full confidence.
LOW 0.15 – 0.50 Marginal match. Claude still generates an answer, but adds hedging language: "Results may not directly answer your query."
NONE < 0.15 No good match. BSearch skips the Claude call entirely and shows a no-results message. This saves cost and avoids hallucination.

Additionally, there is a per-chunk floor of 0.10. Even within an otherwise high-confidence result set, individual chunks scoring below 0.10 are dropped before being sent to Claude. This prevents irrelevant noise from contaminating the synthesis.

IB Analogy

This is like a deal screen with hard cutoffs. If a potential buyer does not meet your minimum revenue threshold, they do not make the list — no matter how interesting they are on other dimensions. Relevance gating prevents BSearch from wasting expensive AI resources (and your time) on low-quality matches.

5.8 Step 7 — Synthesis

This is where the search results become an actual answer. Claude Sonnet 4.6 receives the top-ranked document chunks, the original query, and entity context from Neo4j, then generates a written response that streams to the user in real time.

The synthesis strategy varies by query type:

FILE_LOOKUP

Terse response. Claude returns the file link and minimal context (e.g., "Here's the GovTech pitch deck, last modified March 15."). No extended thinking — speed is everything for file lookups.

DATA_EXTRACTION

Precise response. Claude extracts the specific data point requested and cites the source document. For example: "The EV/EBITDA multiple used for Catalyst was 8.5x, per the valuation analysis in the CIM dated January 2025." No extended thinking needed — the answer is typically in a single chunk.

MULTI_DOC_SYNTHESIS

Structured chain-of-thought. Claude uses extended thinking — a feature where the model explicitly reasons through its extraction and synthesis steps before writing the final answer. The thinking budget is roughly 2,000 tokens. This is the only query type where extended thinking is enabled, because cross-document analysis genuinely benefits from the model "showing its work" internally. While Claude thinks, the user sees a "Searching across multiple documents..." indicator.

Extended thinking — A mode where the AI model writes out its reasoning process (extraction, comparison, synthesis) before producing the final answer. The reasoning is internal — the user only sees the polished result, but the quality is significantly higher for complex, multi-source questions.

Entity context injection

Before Claude generates its response, BSearch injects entity context from Neo4j into the prompt. This includes structured information about the entities mentioned in the query: deal status, financial metrics, associated companies, key people, and stage progression. This context helps Claude produce more accurate and grounded answers without needing to infer facts from document fragments alone.

Citations

Every claim in Claude's response is linked to its source via clickable SharePoint deep links. Users can verify any statement by clicking the citation to open the original document in SharePoint. This is not just a nice feature — it is essential for trust. In investment banking, no one will rely on a system that does not show its sources.

5.9 Latency Budget

Every millisecond in the pipeline is accounted for. Here is where time is spent:

Stage Typical (p50) Worst Case (p99)
Regex check <5ms <5ms
Haiku classify + expand ~150ms ~250ms
Voyage AI query embedding ~150ms ~400ms
Critical path (parallel) ~150ms ~400ms
Azure AI Search ~100ms ~200ms
Cohere rerank ~200ms ~350ms
Claude TTFT ~500ms ~800ms
Total TTFT ~950ms ~1.75s

p50 / p99 — Percentile measurements. p50 (the median) means half of all queries are faster than this. p99 means 99% of queries are faster — only 1 in 100 takes longer. The gap between p50 and p99 shows how much variance the system has.

The "Critical path (parallel)" row is the key to understanding the total. Because Haiku classification and Voyage embedding run simultaneously, the pipeline only waits for whichever is slower. In the typical case, both take ~150ms, so the parallel step adds only ~150ms — not 300ms.

Why This Matters

A total time-to-first-token under one second (typical) means BSearch starts streaming its answer almost immediately. The user never stares at a blank screen — words start appearing within a second of pressing Enter.

5.10 Concrete Example: "What multiple did we use for Catalyst?"

Let's walk through this exact query, step by step, and see what happens inside BSearch.

Step 1 — Classification (0–150ms)

The regex engine scans the query. No file extension, no "find" or "where's" pattern — regex does not match. The query goes to Haiku.

Haiku classifies this as DATA_EXTRACTION — you want a specific data point, not a file and not a cross-document summary. Haiku also extracts entity intent:

  • Company: "Catalyst"
  • Financial concept: "multiple" (likely EV/EBITDA)
  • Role: "we" implies Berenson was the advisor

Step 2 — Query Expansion (150–160ms)

Three expansion layers fire:

  1. Static synonym map: "multiple" → adds "EV/EBITDA" as an expansion term
  2. Haiku contextual expansion (already completed in the same call as classification): adds "purchase price," "enterprise value," "valuation multiple"
  3. Neo4j alias lookup: "Catalyst" is queried in the knowledge graph. It resolves to "Catalyst Technologies, Inc." and "Catalyst Tech" (both are aliases used in different documents)

The expanded BM25 query looks like: "multiple"(3x) "Catalyst"(3x) "EV/EBITDA"(1.5x) "purchase price"(1.5x) "enterprise value"(1.5x) "valuation multiple"(1.5x) "Catalyst Technologies"(1.5x) "Catalyst Tech"(1.5x)

Step 3 — Embedding (0–150ms, parallel with Step 1)

Simultaneously, Voyage AI converts the original query into a 1,024-dimensional vector. The LRU cache misses (nobody has asked this exact question recently), so a fresh embedding is generated in ~150ms. This vector captures the meaning: "what valuation multiple was applied in a deal involving a company called Catalyst."

Step 4 — Search (160–280ms)

Azure AI Search receives a hybrid query: the expanded BM25 terms plus the vector embedding. Filters are applied:

  • Source folder filter (based on the user's permissions)
  • Canonical flag = true (exclude duplicate versions)
  • Entity filter: company matches "Catalyst Technologies, Inc."

In parallel, Neo4j retrieves entity context about Catalyst: deal status (closed), deal stage (completed Q3 2024), Berenson's role (sell-side advisor), and basic financial metrics.

Azure returns 20 candidate chunks, including excerpts from the Catalyst CIM, the valuation analysis, the board presentation, and a few tangentially related documents.

Step 5 — Reranking (280–480ms)

Cohere rescores all 20 candidates against the original query: "What multiple did we use for Catalyst?" — not the expanded version. The valuation analysis chunk containing "implied EV/EBITDA of 8.5x" scores highest (0.82). The CIM overview chunk scores 0.61. A loosely related comp table from a different deal, surfaced by the "EV/EBITDA" expansion, scores only 0.12 and is dropped. The top 5 are kept.

Step 6 — Relevance Gating (480ms)

Top reranker score is 0.82, well above the 0.50 threshold. Confidence tier: HIGH. All five chunks score above the 0.10 per-chunk floor. No chunks are dropped. Proceed to synthesis with full confidence.

Step 7 — Synthesis (480–980ms TTFT)

Claude Sonnet 4.6 receives:

  • The original query
  • Five ranked document chunks (valuation analysis, CIM, board presentation, etc.)
  • Neo4j entity context (Catalyst Technologies, closed deal, sell-side, Q3 2024)
  • Query type: DATA_EXTRACTION (so: be precise, cite sources, no extended thinking)

Claude begins streaming its response at ~980ms:

Example Response

"The EV/EBITDA multiple used in the Catalyst Technologies transaction was 8.5x, based on LTM EBITDA of $47.2M, implying an enterprise value of approximately $401M. This is documented in the valuation analysis section of the CIM dated January 2025."

Source: Catalyst Technologies — Valuation Analysis.docx [clickable link]

Total time from pressing Enter to the first word appearing: under one second. The full response finishes streaming 2–3 seconds later.

Key Takeaway

The query pipeline is a seven-step funnel: classify the question, expand the terminology, embed the meaning, search the index, rerank for precision, gate on relevance, and synthesize the answer. Each step eliminates noise and adds precision, so Claude only works with the best material. Parallelization across steps keeps total latency under one second for the typical query.
