Chapter 5

How Questions Get Answered

From the moment you press Enter to the first word of the answer — every step of the query pipeline, and why each one matters.

When you type a question into BSearch and press Enter, ten things happen in roughly one to five seconds. Some of them happen simultaneously. By the time words start streaming onto your screen, the system has already figured out what kind of question you asked, expanded your terminology so it does not miss relevant documents, searched half a million files, re-scored the top results for precision, decided whether the results are actually good enough to answer your question, and begun composing a written response with citations.

This chapter walks through all ten steps in order. Many of them overlap in time — the system does not wait for one to finish before starting the next. That parallelism is what keeps the whole process fast.

IB Analogy

Think of a well-run deal process. You start with a universe of hundreds of potential buyers, screen down to a long list, narrow to a short list, then focus your time on the two or three who will actually close. The query pipeline works the same way: start broad, filter aggressively, and spend the expensive resources only on the highest-quality matches.

5.2 Step 1 — Query Classification

The very first thing BSearch does is figure out what kind of question you asked. This matters because a file lookup ("find Report_v3.docx") requires a completely different workflow than a cross-document analysis ("Summarize our healthcare deal pipeline"). Knowing the question type up front lets the system tailor every subsequent step — how many documents to retrieve, how much reasoning to apply, and how fast to aim for.

Query classification — The process of categorizing a user's question into one of several predefined types so the system can choose the right search strategy, retrieval depth, and synthesis approach for each type.

Two classification paths

Regex fast path (<5ms). A set of pattern-matching rules catches obvious file lookups before any AI model is involved. If your query contains a filename with an extension ("find Report_v3.docx") or uses a locator phrase with a document name and no content-question words ("pull up the Catalyst NDA"), the regex catches it immediately and classifies it as a file lookup. This is the fastest possible path — no network call, no model inference, under five milliseconds.

Regex — Short for "regular expression." A pattern-matching technique that scans text for specific structures (like filenames ending in .docx or .xlsx). Extremely fast but rigid — it can only catch patterns you explicitly define.

Haiku LLM fallback (~150ms). If the regex does not match, BSearch sends the query to Claude Haiku — a small, fast AI model — which classifies it into one of three categories:

  • file_lookup — The user wants to find a specific file. Example: "Where's the GovTech pitch deck?"
  • data_extraction — The user wants a specific fact or number from one or a few documents. Example: "What was the EBITDA margin for Catalyst?" This is the default when the query is ambiguous.
  • multi_doc_synthesis — The user wants an answer that requires reading and reasoning across multiple documents. Example: "Summarize all healthcare deals from the last two years."

When the query could go either way, the system defaults to data_extraction — the middle ground that works well for most questions. If Haiku does not respond within 500 milliseconds, the system also defaults to data_extraction and moves on. No question gets stuck waiting.

IB Analogy

This is like a senior analyst who instantly knows whether you are asking "where's the model?" (find me a file), "what was the multiple?" (pull a specific number), or "summarize the deal pipeline" (cross-document analysis). They do not need to think about it — years of experience mean the right approach is obvious within seconds. The regex path is like pure muscle memory; the Haiku path is like a quick judgment call.

Entity intent extraction

In the same Haiku call, BSearch also extracts entity intent — structured information embedded in the query. From "What multiple did we use for Catalyst?", Haiku identifies:

  • Company name: Catalyst
  • Financial concept: valuation multiple
  • Role filter: deals where Berenson was the advisor ("we")

It can also detect industry filters ("healthcare deals"), deal status ("active deals"), deal stage ("anything in diligence"), financial constraints ("EBITDA over $50M"), and whether the question requires traversing relationships across multiple deals. All of this is extracted in a single call — no extra latency.

Entity intent — The structured information BSearch extracts from a natural-language query: which company, which person, what financial metric, what deal stage, and so on. This is used downstream to filter search results and enrich the AI's context.

5.3 Step 2 — Query Expansion

Investment banking terminology is wildly inconsistent. The same concept might be called "multiple," "EV/EBITDA," "trading comp multiple," or "valuation metric" depending on who wrote the document and when. If BSearch only searched for the exact words you typed, it would miss relevant results constantly.

Query expansion solves this by automatically adding related terms to your search. It happens in two layers, and neither adds any latency to the overall pipeline.

IB Analogy

This is like knowing that when someone says "comps" they might mean trading comps, precedent transactions, or comparable companies. Or when searching for precedent transactions, you need to check for "acquisition," "merger," "buyout," "take-private," and "business combination" — all of which might describe the same deal in different documents. Query expansion automates that kind of synonym awareness.

Layer 1: Static IB synonym map (0ms added)

A curated JSON dictionary maps common IB shorthand to its formal equivalents. When BSearch sees "multiple" in your query, it automatically adds "EV/EBITDA," "EV/Revenue," and "valuation multiple" as search terms. "Comp" expands to "comparable companies" and "precedent transactions." "LOI" expands to "letter of intent." This is instantaneous because it is a simple dictionary lookup — no network call, no AI model, just an in-memory table loaded when the system starts.

The synonym map is a configuration file that can be updated without redeploying the application. As the team encounters new terminology gaps, new mappings can be added within minutes.

Layer 2: Haiku contextual expansion (0ms added)

This runs inside the same Haiku call as classification (Step 1), so it adds zero additional latency. Haiku generates two to three alternative phrasings that capture the intent behind your query — context-dependent expansions the static map cannot handle. For example:

  • "what did we pay for Catalyst" → adds "purchase price," "enterprise value," "transaction value"
  • "who ran the Acme process" → adds "lead advisor," "sell-side banker," "engagement team"
  • "what's our healthcare pipeline look like" → adds "healthcare deal pipeline," "healthcare active deals," "healthcare mandates"

How expansion terms are weighted

Expansion is applied to the keyword search (BM25) only — not to the vector search, which already understands semantic similarity on its own. To prevent expansion terms from overwhelming your original words, BSearch uses a weighting system: your original terms are boosted higher than expansion terms. This ensures the original query always dominates, with expansion terms acting as a safety net for terminology mismatches.

File lookup queries skip expansion entirely — when you ask for a specific file, adding synonyms just creates noise.

BM25 — A keyword-matching algorithm used in search engines. It finds documents containing your exact words (and their expansions) and scores them based on how frequently and prominently those words appear. Fast and precise for exact terms, but blind to meaning — it would not know that "what did we pay" and "enterprise value" are related.

5.4 Step 3 — Entity Context Retrieval from Neo4j

When your query mentions a company name, BSearch does not just search for that name — it consults the knowledge graph to understand everything it knows about that entity.

Neo4j — A graph database that stores relationships between entities (companies, deals, people) as a web of connections. BSearch uses it to understand that "GS" and "Goldman Sachs" refer to the same entity, that "Project Atlas" is the code name for a specific acquisition target, and that a particular company is connected to a specific deal.

Alias expansion

IB deals always have a code name ("Project Atlas") used in early documents and a real company name ("Acme Corp") in later ones. If you search for "Acme," BSearch looks up that company in the knowledge graph, finds all its aliases — "Project Atlas," "Acme Corporation," "Acme Corp" — and adds them to the BM25 search terms. This way, documents that use the code name are not missed.

Financial constraint filtering

For queries with financial constraints ("deals with EBITDA over $50M"), BSearch queries Neo4j for all deals matching that criterion. Neo4j returns a list of qualifying deal identifiers, which are then passed as a hard filter to the search engine. Only documents from those deals appear in the results.

If this filter is too restrictive — fewer than three results come back — BSearch automatically retries without the financial filter and adds a note to the AI: "Financial filter relaxed — not all results may match the stated financial criteria."

Entity context for synthesis

Neo4j also retrieves structured context about the entities in your query: deal status (active, closed, on hold), financial metrics, company roles (target, acquirer, advisor), stage progression, and key contacts. This context is injected into the AI's prompt later (in Step 8) so it can produce more accurate answers without needing to infer facts from document fragments alone.

This entire step adds roughly 10 milliseconds. The alias expansion must complete before search begins (because the aliases need to be in the search query), but the entity context retrieval for synthesis runs in parallel with search — it does not block anything.

5.5 Step 4 — Embedding

While Steps 1 through 3 handle the keyword side of the search, BSearch simultaneously converts your query into a vector embedding — a list of numbers that captures the meaning of what you asked, not just the words.

Vector embedding — A mathematical representation of text as a list of 1,024 numbers. Texts with similar meanings end up with similar numbers, even if they use completely different words. This is what allows BSearch to find a document about "enterprise value" when you searched for "what did we pay."

This step runs in parallel with classification (Step 1). The system fires off both requests at the same time, so the total wait is determined by whichever finishes last — not the sum of both. Three mechanisms keep embedding fast and resilient:

  • LRU cache (1,000 entries). If anyone recently asked the same or a very similar question, BSearch reuses the cached embedding instantly — zero milliseconds, no network call. "LRU" stands for "least recently used": the cache keeps the 1,000 most recent embeddings and evicts the oldest when full.
  • 500ms hard timeout. If the embedding service is slow or unreachable, BSearch abandons vector search entirely and falls back to keyword search only. A slightly less precise search is better than a query that hangs.
  • File lookup skip. Queries classified as file lookups via the regex fast path skip the embedding step entirely — finding a file by name does not require semantic understanding, so there is no point in computing an embedding.

But entity context retrieval (deal status, financials) for synthesis runs in parallel with Azure AI Search, adding no time to the critical path.

5.6 Step 5 — Hybrid Search

With the expanded query terms and the vector embedding ready, BSearch sends a hybrid query to the search index. "Hybrid" means it combines two fundamentally different search strategies in a single request:

BM25 (Keyword Matching)

Finds documents containing your exact terms and their expansions. Great for specific names, ticker symbols, deal codes, and precise terminology that vector search might miss. Expansion terms are boosted lower than your original words.

Vector Similarity

Finds documents with similar meaning, even if they use completely different words. Great for natural-language questions where you do not know the exact terminology used in the documents. Uses the original query embedding — not expansion terms.

The search engine merges scores from both approaches and applies a recency scoring profile — a slight boost for recently modified documents. In IB, newer versions are almost always more relevant, so a valuation model updated last week should rank above one from six months ago.

Filters narrow the search space

Before results are scored, several hard filters remove documents that should not appear:

  • Permission filter. Users only see results from folders they have access to. This is enforced as a hard filter at the search level — no exceptions, no leakage.
  • Canonical filter. Only the latest version of near-duplicate documents is returned, preventing the same CIM draft from appearing five times in results.
  • Entity filters. If Haiku detected industry, company role, deal status, deal stage, or financial constraints in your query, those are applied as search filters. "Active healthcare deals" automatically filters for healthcare industry and active status.
  • Financial filter. If Neo4j returned a list of deal identifiers matching a financial constraint ("EBITDA over $50M"), only documents from those deals are included.

The search returns the top 20 candidates for reranking.

5.7 Step 6 — Reranking

The search engine is fast but its ranking is approximate. To get truly precise relevance ordering, BSearch sends the top 20 results to a dedicated reranking model (Cohere), which re-scores each result against the original query — not the expanded version.

Reranking — A second-pass scoring step where a specialized AI model reads each search result alongside the original query and assigns a precise relevance score. Much slower than the initial search (~200–350ms for 20 documents), but dramatically more accurate at identifying which results are truly relevant.

Using the original query for reranking is deliberate. Query expansion (Step 2) may have added marginal terms like "comparable companies" that surface loosely related documents. The reranker evaluates each result against what you actually asked, so those marginal matches get dropped. This is why aggressive expansion is safe — the reranker acts as a noise filter.

IB Analogy

This is like the difference between an initial buyer screen (broad criteria, fast) and management's deep assessment of each potential acquirer (slower but much more precise). The search step casts a wide net; the reranker decides which catches are actually worth keeping.

After reranking, BSearch keeps the top 5 results for file lookups and data extraction queries, or the top 10 for multi-document synthesis queries (because cross-document answers need more source material).

Fallback

If the reranking service is unavailable, BSearch skips reranking entirely and uses the search engine's native ranking instead. When this happens, the system caps confidence at LOW (see Step 7) because the results have not been precision-verified, and reduces the candidate pool to 12 to compensate. The user sees an orange banner: "Search quality may be reduced — results are ranked without AI reranking."

5.8 Step 7 — Relevance Gating

Before sending anything to the AI for synthesis, BSearch checks whether the search results are actually relevant to what you asked. This prevents the system from confidently composing an answer from documents that have nothing to do with your question.

Relevance gating — A quality checkpoint that evaluates whether the retrieved documents are good enough to generate a trustworthy answer. Based on the reranker's scores, the system decides whether to proceed with full confidence, proceed with caution, or stop and tell the user it found nothing relevant.

The system uses three confidence tiers based on the top reranker score:

Tier Score Range What Happens Banner Behavior
HIGH ≥ 0.50 Strong match. The AI generates a full-confidence answer. No banner
LOW 0.15 – 0.50 Marginal match. The AI still generates an answer but adds hedging language. Yellow banner: "Results may not directly answer your query."
NONE < 0.15 No good match. BSearch skips the AI call entirely. "No relevant results found" message

Additionally, there is a per-chunk floor of 0.10. Even within an otherwise high-confidence result set, individual chunks scoring below 0.10 are dropped before being sent to the AI. This prevents irrelevant noise from contaminating the answer.

The NONE tier is particularly important. When the system genuinely has nothing relevant to show you, skipping the AI call entirely is better than letting the model improvise an answer from irrelevant scraps. It saves cost and — more importantly — avoids hallucination.

IB Analogy

This is like a deal screen with hard cutoffs. If a potential buyer does not meet your minimum revenue threshold, they do not make the list — no matter how interesting they are on other dimensions. Relevance gating prevents BSearch from wasting expensive AI resources (and your time) on low-quality matches.

5.9 Step 8 — Tiered Synthesis

This is where the search results become an actual answer. Claude Sonnet receives the top-ranked document chunks, the original query, entity context from Neo4j, and a prompt strategy tailored to the query type. The response streams to the user in real time.

The synthesis approach varies significantly by query type because different questions have fundamentally different reasoning demands:

file_lookup — Terse and fast

The AI returns the file link and minimal context — "Here's the GovTech pitch deck, last modified March 15." No extended reasoning, just one or two chunks in context. Speed is everything for file lookups. Target: first word on screen in under 2 seconds.

data_extraction — Precise facts with citations

The AI extracts the specific data point requested and cites the source document. For example: "The EV/EBITDA multiple used for Catalyst was 8.5x, per the valuation analysis in the CIM dated January 2025." Up to five chunks in context, no extended thinking needed. Target: first word on screen in under 3 seconds.

multi_doc_synthesis — Structured reasoning across documents

This is the most demanding query type. The AI uses extended thinking — a mode where it explicitly reasons through extraction and synthesis steps before writing the final answer. The thinking budget is roughly 2,000 tokens. Up to ten chunks in context, plus any deal summary chunks that surfaced during search.

Extended thinking — A mode where the AI writes out its reasoning process (extracting facts from each document, identifying conflicts, noting gaps) before producing the final answer. The reasoning is internal — the user only sees the polished result — but the quality is significantly higher for complex, multi-source questions.

The AI follows a structured approach: first extract facts from each source chunk, then identify which deals are represented and any conflicts between sources, then synthesize a response organized by deal (not by source document), with citations for every claim and explicit flags for stale or incomplete information.

While the AI thinks, the frontend shows "Searching across multiple documents..." — a distinct indicator that tells the user something more complex is happening. Target: first word on screen in under 5 seconds.

Deal summary injection

For multi-document synthesis queries, if any pre-computed deal summary chunks appear in the top 10 results, they are placed first in the AI's context with a label like "Deal Overview: Catalyst Technologies." These summaries give the AI a structured foundation for cross-deal reasoning, reducing the burden of piecing together deal-level facts from scattered document fragments.

Entity context injection

Before the AI generates its response, BSearch prepends entity context from Neo4j: deal status, financial metrics, company roles, stage progression, and key contacts. This structured information helps the AI produce more accurate and grounded answers without needing to infer facts from document excerpts alone.

Citations

Every claim in the AI's response is linked to its source via clickable SharePoint deep links. Users can verify any statement by clicking the citation to open the original document in SharePoint. This is not optional — in investment banking, no one will rely on a system that does not show its sources.

5.10 Step 9 — Streaming Answers

BSearch does not wait for the entire answer to be composed before showing it to you. The response streams word by word via SSE (Server-Sent Events) — a protocol that pushes data from the server to the browser as it becomes available.

Server-Sent Events (SSE) — A lightweight protocol where the server pushes updates to the browser over a single persistent connection. Unlike a normal web request where you send a request and get back a complete response, SSE lets the server send data incrementally — perfect for streaming an AI-generated answer word by word.

The moment the AI produces its first token, that token appears on your screen. You see words materializing in real time as the model writes. This means you can start reading the answer — and in many cases, get what you need — before the full response is even finished generating.

The SSE stream also carries metadata: confidence level, whether the reranker is degraded, the query type, and eventually the follow-up suggestions. The frontend uses these signals to display the appropriate banners and UI indicators.

5.11 Step 10 — Follow-up Suggestions

After the answer, BSearch suggests two to three related questions you might want to ask next. These are generated by the AI as part of the same synthesis call — no extra latency, no additional API call.

For example, after answering "What multiple did we use for Catalyst?", the follow-ups might be:

  • "What comparable transactions supported the 8.5x multiple?"
  • "What was Catalyst's LTM revenue at the time of the deal?"
  • "Which other active deals are in a similar valuation range?"

Follow-up suggestions help users explore related questions without having to formulate them from scratch. They are parsed from the AI's output stream and displayed as clickable buttons below the answer.

5.12 Latency Budget

Every millisecond in the pipeline is accounted for. Here is where time is spent for a typical data extraction query:

Stage Typical (p50) Worst Case (p99) Parallel?
Regex check <5ms <5ms First check
Haiku classify + expand + entity intent ~150ms ~250ms Parallel with embedding
Voyage AI query embedding ~150ms ~400ms Parallel with Haiku
Critical path (parallel) ~150ms ~400ms max(Haiku, embedding)
Entity alias expansion (Neo4j) ~10ms ~10ms Sequential (after Haiku)
Azure AI Search (hybrid) ~100ms ~200ms Sequential
Cohere rerank ~200ms ~350ms Sequential
Claude synthesis (TTFT) ~500ms ~800ms Sequential, streaming
Total TTFT (data_extraction) ~960ms ~1.76s

TTFT (Time to First Token) — The time between pressing Enter and the first word of the answer appearing on screen. This is the metric that determines how "fast" the system feels. Even if the full answer takes five seconds to stream, a sub-one-second TTFT means you never stare at a blank screen.

p50 / p99 — Percentile measurements. p50 (the median) means half of all queries are faster than this. p99 means 99% of queries are faster — only 1 in 100 takes longer. The gap between p50 and p99 shows how much variance the system has.

The "Critical path (parallel)" row is the key to understanding the total. Because Haiku classification and the embedding run simultaneously, the pipeline only waits for whichever is slower. In the typical case, both take about 150ms, so the parallel stage adds only 150ms — not 300ms.

TTFT targets by query type

Query Type TTFT Target Chunks in Context Extended Thinking
file_lookup <2 seconds 1–2 No
data_extraction <3 seconds 5 No
multi_doc_synthesis <5 seconds 10 + deal summaries Yes (~2,000 tokens)

File lookups are the fastest because they can skip both the Haiku call (caught by regex) and the embedding step, going straight to keyword search. Multi-document synthesis is the slowest because it sends more context to the AI and enables extended thinking, but the relaxed five-second target accommodates this — and the "Searching across multiple documents..." indicator keeps the user informed.

Why This Matters

A total time-to-first-token under one second (typical for data extraction) means BSearch starts streaming its answer almost immediately. The user never stares at a blank screen — words start appearing within a second of pressing Enter. Even for the most complex multi-document queries, the five-second target ensures the system feels responsive, not sluggish.

5.13 Concrete Example: "What multiple did we use for Catalyst?"

Let us walk through this exact query, step by step, and see what happens inside BSearch.

Step 1 — Classification (0–150ms)

The regex engine scans the query. No file extension, no "find" or "where's" pattern — regex does not match. The query goes to Haiku, which runs in parallel with the embedding (Step 4).

Haiku classifies this as data_extraction — you want a specific data point, not a file and not a cross-document summary. It also extracts entity intent:

  • Company: "Catalyst"
  • Financial concept: "multiple" (likely EV/EBITDA)
  • Role: "we" implies Berenson was the advisor

And generates contextual expansions: "purchase price," "enterprise value," "valuation multiple."

Step 2 — Query Expansion (0ms added)

Two expansion layers activate:

  1. Static synonym map: "multiple" adds "EV/EBITDA," "EV/Revenue," and "valuation multiple"
  2. Haiku contextual expansion (already completed in the same call): adds "purchase price," "enterprise value," "transaction value"

Original terms are boosted higher than expansion terms, so the search prioritizes exact matches.

Step 3 — Entity Context (150–160ms)

"Catalyst" is queried in the knowledge graph. It resolves to "Catalyst Technologies, Inc." and "Catalyst Tech" — aliases used in different documents. These are added to BM25 search terms. Neo4j also begins retrieving entity context (deal status: closed, Q3 2024, sell-side advisor), which will be injected into Claude's prompt later.

Step 4 — Embedding (0–150ms, parallel with Step 1)

Simultaneously, the original query is converted into a 1,024-number vector that captures its meaning: "what valuation multiple was applied in a deal involving a company called Catalyst." The cache misses (nobody asked this exact question recently), so a fresh embedding is generated in about 150ms.

Step 5 — Hybrid Search (160–280ms)

The search engine receives both the expanded keyword query and the vector embedding. Filters are applied: permission filter (user's folders), canonical filter (latest versions only), and entity filter (company matches "Catalyst Technologies, Inc."). It returns 20 candidate chunks from the Catalyst CIM, the valuation analysis, the board presentation, and a few tangentially related documents.

Step 6 — Reranking (280–480ms)

Cohere rescores all 20 candidates against the original query: "What multiple did we use for Catalyst?" The valuation analysis chunk containing "implied EV/EBITDA of 8.5x" scores highest (0.82). The CIM overview scores 0.61. A loosely related comp table from a different deal, surfaced by the "EV/EBITDA" expansion, scores only 0.12 and is dropped below the per-chunk floor. The top 5 are kept.

Step 7 — Relevance Gating (480ms)

Top reranker score is 0.82 — well above the 0.50 threshold. Confidence tier: HIGH. All five chunks score above the 0.10 per-chunk floor. No chunks dropped, no banner, full confidence. Proceed to synthesis.

Step 8 — Synthesis (480–980ms TTFT)

Claude receives the original query, five ranked document chunks, Neo4j entity context (Catalyst Technologies, closed deal, sell-side, Q3 2024), and the data_extraction prompt strategy (be precise, cite sources, no extended thinking). It begins streaming its response at roughly 980ms:

Example Response

"The EV/EBITDA multiple used in the Catalyst Technologies transaction was 8.5x, based on LTM EBITDA of $47.2M, implying an enterprise value of approximately $401M. This is documented in the valuation analysis section of the CIM dated January 2025."

Source: Catalyst Technologies — Valuation Analysis.docx [clickable link]

Steps 9 & 10 — Streaming + Follow-ups

The answer streams word by word via SSE. Total time from pressing Enter to the first word appearing: under one second. The full response finishes streaming two to three seconds later. After the answer, three follow-up suggestions appear as clickable buttons.

Key Takeaway

The query pipeline is a ten-step process: classify the question, expand the terminology, retrieve entity context, embed the meaning, search the index, rerank for precision, gate on relevance, synthesize the answer, stream it to the user, and suggest follow-ups. Each step eliminates noise and adds precision, so the AI only works with the best material. Parallelization across steps keeps total latency under one second for typical queries and under five seconds even for complex cross-document analysis.
